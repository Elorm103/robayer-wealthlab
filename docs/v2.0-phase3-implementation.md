## Stage 3 — Orders (complete)

**Backend**: `services/admin/orderService.ts` (read-only over `purchase_sessions`/`payment_transactions`/`deliveries`/`email_log` — `listOrders`, `getOrderByReference`, plus `resendReceipt`/`resendDownload`, the only two functions in this file that cause a side effect, and even those write nothing directly: they call the existing `emailService.sendEmail()`, which owns its own `email_log` insert), `routes/admin/orders.ts` (thin HTTP layer — `GET /meta`, `GET /`, `GET /:reference`, `POST /:reference/resend-receipt`, `POST /:reference/resend-download`, wired into `worker/index.ts` with `/meta` ordered before `/:reference`).

**Role-gating divergence, the mirror image of Stage 1/2's**: list/detail are open to all three roles (support included — read visibility is a legitimate part of the support workflow). The two resend actions are `super_admin`/`editor` only, via `requireRole()` — the first Phase 3 write endpoints gated this way, because this is the first Phase 3 module where a mistake has a real, external, customer-facing consequence (an unwanted email), matching Products' `EDITOR_ROLES` convention instead of Consultation/Contact Manager's all-roles-can-write one. Enforced server-side; the frontend hiding the buttons for `support` is UX only.

**Reference validation**: `/^RWL-\d{4}-\d{6,}$/`, checked in the route file before any D1 query, identical convention to the public `routes/purchases.ts`.

**Synthetic test-order flag**: `isSyntheticTestEmail()` matches the known QA pattern `checkout+rwl-...@robayerwealthlab.com` (see `docs/v1.1-ceo-dashboard.md`/`docs/analytics-implementation-plan.md`'s prior finding) and surfaces a "Test" badge in both the list and the drawer, so QA activity is never mistaken for a real sale.

**No new migration** — pure reads plus reuse of the existing `sendEmail()`; zero new tables or columns.

**Frontend**: `admin/orders/index.html` (replaces the empty shell), `js/components/admin/admin-orders.js` — data table (Reference, Product, Customer email, Amount, Status, Date), search (reference/email substring), status chips, date-range filters, and a drawer (reusing Stage 1's `.drawer*` CSS) showing order details, payment transactions, deliveries, and email history, with the two resend buttons hidden entirely (not just disabled) for `support`.

**Local dev infrastructure fix, found while trying to verify this stage in a real browser**: the existing `.claude/launch.json` served the static frontend (`npx serve`, port 5500) and the API (`wrangler dev`, port 8787) as two unrelated servers with no proxy between them — a relative `fetch('/api/...')` from a page on :5500 resolved to :5500 itself (404), never reaching the Worker at all. Added `scripts/dev-proxy.mjs` (static file server + `/api/*` reverse proxy to :8787, Node built-ins only) and pointed `launch.json`'s `robayer-static-site` config at it. Fixes real-browser local verification for this and every future admin module, not just this stage.

**Real defect found and fixed**: the drawer's resend-button footer used `element.hidden = true` to hide it for the `support` role, but `css/admin.css`'s `.drawer__footer { display: flex; ... }` (added in Stage 1 for the always-visible mailto-link footer, which was never previously toggled) unconditionally overrides the browser's default `[hidden] { display: none }` — same specificity, author stylesheet always wins over the UA stylesheet regardless of order. Result: a `support`-role admin would have *seen* the resend buttons even though the server correctly rejected the action (403) if clicked. The server-side rejection meant this was never an actual security hole, but it was a real, confusing UI bug — a support agent seeing an action they can't use. Fixed with `.drawer__footer:not([hidden])`. Found and fixed via real browser testing (a curl-only pass would never have caught this — the JSON response and the DOM's `hidden` attribute were both already correct; only the rendered CSS was wrong).

**Local verification** (`wrangler dev` behind the new proxy + real browser): fresh super_admin and support logins (JS-dispatch workaround for the same Browser-tool click-simulation quirk documented in Stage 1/2), list render with a real verified test order + the pre-existing failed order + a synthetic-flagged order, drawer open/detail-load showing payment transaction + delivery + (correctly empty) email history, search by reference and by email substring, status-chip and date-range filtering, "Test" badge rendering. Resend buttons: visible+enabled for `super_admin` on the verified order, visible+disabled on the non-verified order, hidden entirely for `support` (after the CSS fix above) regardless of order status.

**Adversarial pass** (curl against live `wrangler dev` + real local D1): unauthenticated list/detail/resend → 401; `support` read → 200; `support` resend-receipt and resend-download on a verified order, real CSRF token → both 403 `FORBIDDEN`, confirming the role check is server-side and not just UI-hidden; mutation without CSRF token → 403; mutation with a garbage CSRF token → 403; resend on a non-`verified` order (the pre-existing `RWL-2026-000002`, status `failed`) → 400 `VALIDATION_ERROR`, no email attempted; a SQL-injection-shaped reference (`RWL-2026-900001'; DROP TABLE purchase_sessions; --`) → 404, rejected by the regex before ever reaching D1, table confirmed intact; a well-formed but non-existent reference → 404; invalid `status` filter value → silently ignored (falls back to unfiltered) rather than erroring, matching this project's established tolerant-filter convention.

Real send verification deferred to production, where Stage 1/2's own live acknowledgement emails already prove the production Resend secret is valid — see the production verification section below.

**Fresh code re-read before deploy**: `orderService.ts` and `routes/admin/orders.ts` re-read in full immediately before deploying — no defects found beyond the CSS bug above (already fixed and re-verified).

## Stage 4 — Analytics + Dashboard fix (complete)

**Backend**: `services/admin/analyticsService.ts` (`getSummary`, `getTimeseries`, `getTopProducts` — every number a live D1 aggregate, no caching layer, per `docs/v2-analytics-spec.md`'s explicit "this platform's real row counts don't justify one yet"), `routes/admin/analytics.ts` (`GET /summary`, `GET /timeseries`, `GET /top-products`, all read-only, open to all three roles — nothing to mutate on this page for any role). New shared utility `utils/dateRange.ts` (`exclusiveEndDate`, `previousPeriod`, `deltaPercent`, `everyDateInRange`) — see the real defect below for why this exists as a shared file rather than inline math in the service.

**Real defect found and fixed, in *previously-shipped* Stage 3 code**: while designing Stage 4's own date-range filtering, re-derived the exact semantics of `orderService.ts`'s `dateTo` filter (`created_at <= ?` bound to a bare `YYYY-MM-DD` string) and found it silently broken — `'2026-07-18 08:36:45' <= '2026-07-18'` is `false` under SQLite's lexicographic TEXT comparison (the longer, time-bearing string sorts *after* the bare-date prefix), so filtering Orders by an end date excluded essentially every row from that day except an impossible exact-midnight match. Confirmed with a standalone comparison test before touching any code. Fixed with an exclusive-upper-bound pattern (`created_at < exclusiveEndDate(dateTo)`), extracted into `utils/dateRange.ts` specifically so Stage 4's own date-range math (which has the identical correctness requirement) reuses the fixed, tested logic rather than risking the same mistake a second time. Re-typechecked clean; Orders' own adversarial battery was not re-run in full since the fix is additive and narrowly scoped to the one comparison operator, but the corrected logic is exercised indirectly by every Analytics query below.

**KPI summary** (`revenuePesewas`, `orders`, `newSubscribers`, `downloadsServed`, `consultations`, `contacts` — matching the architecture plan's Section 5 list, with "consultation/contact volume" split into two separate metrics, mirroring the Dashboard's own established convention of two separate cards rather than one merged figure): each returns `{ current, previous, deltaPercent }` for the selected date range vs. the immediately-preceding period of equal length. `deltaPercent` is `null` (rendered "New") rather than `0` or `Infinity` when `previous` is `0` — a percentage change from zero is genuinely undefined, not a fake number.

**Data sources, each a real, existing table** — revenue/orders: `purchase_sessions.verified_at` (not `created_at`, so a session that never completed doesn't count); new subscribers: `newsletter_subscribers.subscribed_at` (counts signup *events* in the window regardless of later unsubscribe, an honest historical-volume answer, distinct from the Dashboard's own "currently active" figure); downloads served: `download_tokens.used_at` (the real per-download redemption event, not the `deliveries.downloads_used` running total); consultations/contacts: `created_at`, `deleted_at IS NULL`.

**Timeseries**: `ordersPerDay`/`subscribersPerDay`, zero-filled for every date in range (a day with zero real activity renders as a real zero, not a silently-skipped gap indistinguishable from missing data) — rendered by a new hand-rolled, dependency-free inline SVG chart helper (`js/components/admin/timeseries-chart.js`, `window.AdminCharts.renderTimeseries`), per `docs/v2-analytics-spec.md`'s explicit "no charting library" recommendation at this platform's real data volume.

**Top Products**: `GROUP BY product_slug` over verified orders in range, `LEFT JOIN`ed to the *live* `products.title` (confirmed working correctly during verification — see below) rather than each order's own historical snapshot, matching the architecture plan's explicit instruction; falls back to the order's snapshotted title if no live product row matches (e.g. a since-deleted product).

**Visitors/Sessions/Traffic Sources**: never faked. The page's final card states plainly that Cloudflare Web Analytics has no API for this data and links out to the real Cloudflare dashboard, per `docs/v2-analytics-spec.md`'s "show what's real, link to what isn't" rule — the one deliberate, documented divergence from the older, rougher `docs/v2-dashboard-ux-spec.md` draft (which listed "Visitors" as an embeddable KPI before the later analytics spec established the real data-source boundary).

**Dashboard fix**: `services/admin/dashboardService.ts`'s `productsCount` was hard-coded `null` since Phase 0.2, when the product catalog still lived in `content/products/*.json`. Version 2.0 Phase 2 migrated it into a real `products` D1 table, but this field was never wired to it — a real, stale figure sitting on the Dashboard's own homepage. Fixed with the exact same fault-tolerant `try/catch → null` pattern every other field here already uses: `SELECT COUNT(*) FROM products WHERE deleted_at IS NULL`. `admin/index.html`'s Products card, previously static hard-coded markup (`<p class="stat-card__value stat-card__value--muted">No data yet</p>`, never even wired to a `data-stat-*` attribute), rebuilt to match the other five cards' skeleton-loading pattern; `admin-dashboard.js` gained the one missing `renderProducts()` function.

**Frontend**: `admin/analytics/index.html` (replaces the empty shell) — date-range picker (7/30/90-day preset chips plus manual date inputs, reusing Orders' own date-input pattern), six `.stat-card` KPIs with a badge-based delta indicator (`badge--success`/`badge--error`/`badge--info` for positive/negative/new, zero new CSS), two inline SVG charts, a Top Products `.table`, and the honest Cloudflare Web Analytics link-out card. `js/components/admin/admin-analytics.js` drives it.

**Local verification** (`wrangler dev` behind Stage 3's proxy fix + real browser, both roles): fresh super_admin and support logins; Dashboard's Products card confirmed showing the real live count (`2`) instead of the old static "No data yet"; Analytics page confirmed rendering all six KPIs, both charts (visually inspected via screenshot — line + point markers render correctly, tooltip title elements present), and the Top Products table with real data; 7/30/90-day preset chips confirmed refetching with a correctly recalculated range and correctly changed numbers; dark mode confirmed rendering correctly with zero additional CSS (screenshot-verified); `support` role confirmed able to read Analytics (no role gate exists for this module, matching the plan); console confirmed clean (no errors) throughout.

Independently computed every expected KPI value via direct SQL (`COUNT`/`SUM` with the identical date-boundary logic, run separately from the service code being tested) *before* calling the API, then compared: revenue (8000 current / 2000 previous / **+300%**), orders (2 / 1 / **+100%**), new subscribers (8 / 1 / **+700%**), downloads served (1 / 0 / **New**), consultations (1 / 0 / **New**), contacts (1 / 0 / **New**) — every figure matched exactly, no rounding surprises. Timeseries independently confirmed zero-filled to exactly 30 days with non-zero counts landing on precisely the two expected dates. Top Products confirmed joining to the *live* product title (`Starting to Invest with GH₵100`, with the ₵ symbol) rather than the order's own snapshotted title (`Starting to Invest with GH100`, without it) — direct proof the `LEFT JOIN products` clause works as designed, not just typechecks.

**Adversarial pass** (curl against live `wrangler dev` + real local D1): unauthenticated → 401 on all three endpoints; invalid/garbage date params → silently falls back to the default 30-day range rather than erroring; a reversed range (`from` after `to`) → swapped, not rejected; a 10-year range → clamped to 366 days; a SQL-injection-shaped date parameter → rejected by the regex validator before ever reaching D1 (confirmed via the same date-pattern check `routes/admin/orders.ts` already uses), table confirmed intact afterward.

## Stage 5 — Acceptance Audit + Release Checkpoint (complete)

A fresh, from-scratch audit — not a continuation of Stage 4's own work. Every Stage 1–4 file was re-read in full against the checklist in the architecture plan (hidden regressions, dead code, stale documentation, unused CSS/JS, security issues, permission-boundary mistakes, caching problems, performance issues), all 7 admin modules were exercised together in a real browser against real data, and a full adversarial battery (authentication, authorization, CSRF, XSS, SQL injection, IDOR, cache behavior) was run against a live `wrangler dev` instance. Per the standing "stop, fix, verify, then continue" rule, four genuine issues were found and fixed before the audit continued.

**Finding 1 — missing `Cache-Control` on every admin API response except Dashboard's.** A direct `curl` header comparison across four endpoints showed only `GET /api/admin/dashboard/summary` sent `Cache-Control: no-store` (via its own local `withNoStore()` wrapper, added in Phase 0.2) — Orders, Analytics, Consultations, Contacts, Products, and Media Library sent none at all, despite every one of them returning admin PII, financial data, or internal operational data. Fixed by moving the rule into the shared `securityHeaders()` function in `middleware/securityHeaders.ts`, applied globally to every non-HTML response (HTML pages, i.e. `routes/books.ts`, are deliberately excluded — a public product page has no reason to forbid caching). `routes/admin/dashboard.ts`'s now-redundant local wrapper was removed as dead code once the global rule made it a duplicate.

**Finding 2 — `handleConsultationGet`/`handleContactGet` missing rate limiting.** Every other Phase 3 read endpoint (Orders, all three Analytics endpoints, both list endpoints for Consultations/Contacts) called `isRateLimited()`; the single-record `GET /:id` handlers for Consultations and Contacts did not — a real inconsistency against the architecture plan's stated "reads share a generous rate limit" intent, not a deliberate exception. Fixed by adding the identical `isRateLimited(request, env, READ_RATE_LIMIT)` call already used by the list endpoints.

**Finding 3 — mailto-injection-adjacent gap in the reply-by-email links.** This codebase's email validation regex (`/^[^\s@]+@[^\s@]+\.[^\s@]+$/`, unchanged by this stage) permits `?`/`&` in the local part before `@`, so an unescaped `mailto:${email}?subject=...` href construction (both `admin-consultations.js` and `admin-contacts.js`) could let a crafted email address inject extra `mailto:` parameters (e.g. a hidden `bcc`). Fixed by wrapping the email in `encodeURIComponent()` (RFC 6068-compliant — `%40` for `@` is correctly decoded by every mail client).

**Finding 4 — stale "Getting started" copy on the Dashboard.** `admin/index.html`'s own homepage card still read "Product management, the media library, and the rest of the sidebar's modules arrive in later phases" — true when Phase 0.2 shipped it, false since Phase 1/2 (Media Library, Products) and now Stages 1–4 of this phase all went live. Rewritten to accurately state what's live versus what's still a shell (Resources, Blog, Newsletter, Settings, Users).

**Cross-module live verification** (real browser, `wrangler dev` behind the Stage 3 proxy): Dashboard, Orders, Analytics, Consultations, Contacts, Media Library, and Products all loaded, rendered real data matching their respective APIs (revenue GH₵0.00, 0 orders, 6 subscribers, 2 products — cross-checked against direct `curl` responses), and produced zero console errors across the full pass.

**Adversarial battery** (curl against live `wrangler dev` + real local D1, this stage's own independent pass — not a rerun of Stage 1–4's): mutation without `X-CSRF-Token` → 403; mutation with a garbage token on a valid session → 403; SQL-injection-shaped search strings (`' OR '1'='1`, `'; DROP TABLE consultation_requests; --`) on Orders/Consultations → safely return empty results, tables confirmed intact (parameterized `.bind()` throughout, no string-concatenated SQL exists in this codebase); IDOR probes (nonexistent id, negative id, non-numeric id) on Consultations → clean 404s, no information leak; a live XSS probe — a real consultation submitted via the public `POST /api/consultation` endpoint with `<script>alert(1)</script>` in `name` and `<img src=x onerror=alert(2)>` in `description` — rendered as literal escaped text in both the admin table and the drawer (confirmed via `element.innerHTML` showing `&lt;script&gt;...`, proving `textContent` assignment, not `innerHTML` injection; no alert fired, no console errors); role-boundary probes — a `support`-role session attempting Orders' resend-receipt, Products create, and Media Library upload-init all returned 403 `FORBIDDEN` (role checked server-side, not just hidden in the UI); session cookie confirmed `HttpOnly; Secure; SameSite=Lax`; login with a wrong password and with a nonexistent email both returned the identical generic `INVALID_CREDENTIALS` message (no user-enumeration signal).
