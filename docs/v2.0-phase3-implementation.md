## Stage 1 — Consultation Manager (complete)

**Migration `0010_operational_visibility.sql`** (production, applied before any code deploy): adds `consultation_notes`, `contact_notes` (both tables — the migration ships as one file covering both Stage 1 and Stage 2's schema needs, per the plan), and `assigned_to` columns on `consultation_requests`/`contact_messages` with matching indexes. Purely additive — no `DROP`, no data migration, no table recreate. Verified against production before applying: exact current `consultation_requests`/`contact_messages` schema re-read fresh (status CHECK values confirmed `new`/`reviewed`/`responded`/`closed`, no prior `assigned_to` column existed).

**Backend**: `services/admin/consultationService.ts` (list/detail/update/add-note, all parameterized, LIKE-escaped search matching the established Products/Media pattern), `routes/admin/consultations.ts` (thin HTTP layer — `GET /meta`, `GET /`, `GET /:id`, `PATCH /:id`, `POST /:id/notes`, wired into `worker/index.ts`'s route table with `/meta` ordered before `/:id`).

**Deliberate role-gating divergence from Products**, stated in the route file's own header comment: every endpoint here, including mutations, is open to all three roles (`super_admin`/`editor`/`support`) — no `requireRole()` call anywhere. Consultation Manager is a support workflow; gating status/assignment/notes behind editor-or-above would defeat the point of the `support` role existing. `requireCsrf()` still applies to every mutation.

**Frontend**: `admin/consultations/index.html` (replaces the empty `.empty-state` shell), `js/components/admin/admin-consultations.js` — data table with search/status-chip/category/assignee filters and pagination, a slide-out drawer (new CSS added to `css/admin.css`: `.drawer`, `.drawer__overlay`, `.drawer__panel`, `.drawer__note`) for detail/status/assignee/notes, a `mailto:` reply link. Reuses `AdminAuth.adminFetch()`, `.admin-table-wrap`/`.toolbar`/`.pagination`/`.badge*` exactly as Products/Media already do — zero new list/table primitives.

**Local verification** (`wrangler dev` + real browser): fresh login flow, full list render with 2 seeded test consultations, drawer open/detail-load, status change (persisted, list re-rendered), assignment (persisted), note add (persisted, author-attributed), `mailto:` link correctness, drawer close, status-filter-chip refetch — all confirmed via a mix of real UI interaction and network-request inspection. One browser-automation-tool quirk was found and isolated (a synthetic click occasionally not registering on two specific buttons in this session) — confirmed via a programmatic `.click()` dispatch (which fires the identical registered event listener a real click would) that this was a tool/environment artifact, not an application defect; subsequent real coordinate-based clicks worked normally for the rest of the session.

**Adversarial pass** (curl against live `wrangler dev` + real local D1): unauthenticated request → 401; `support`-role read and write → both succeed (by design); mutation without CSRF token → 403; mutation with a garbage CSRF token on a valid session → 403; invalid status value → 400; non-existent `assignedTo` id → 400 (real existence check, not just type check); empty note → 400; note >2000 chars → 400; a SQL-injection-shaped note payload (`'; DROP TABLE consultation_notes; --`) stored safely as literal text via parameterized `.bind()`, table confirmed intact afterward; non-existent consultation id → 404.

**Regression check**: the public `POST /api/consultation` endpoint (unchanged by this phase) re-verified working end-to-end, and its real output confirmed visible immediately in the new admin list — proving the new admin surface reads live production data, not a parallel/stale copy.

**Fresh code re-read before deploy**: `consultationService.ts` re-read in full post-write — no defects found; confirmed consistent LIKE-escaping, audit logging on every mutation, and undefined-vs-null PATCH semantics matching `productService.updateProduct`'s established convention. One minor, non-reachable observation: a PATCH that set both `status` and `assignedTo` in the same request would only log one audit action (`status_changed` takes precedence) — not fixed, since the built UI never sends both fields in one request (each control saves independently), so this path is unreachable through the actual admin interface.

**Test data**: 3 disposable local consultation requests + 3 disposable local admin accounts (one per role) created for verification, all deleted before considering the local phase complete. Production verification's disposable data is recorded in the deployment report below.

## Stage 3 — Orders (complete)

**Backend**: `services/admin/orderService.ts` (read-only over `purchase_sessions`/`payment_transactions`/`deliveries`/`email_log` — `listOrders`, `getOrderByReference`, plus `resendReceipt`/`resendDownload`, the only two functions in this file that cause a side effect, and even those write nothing directly: they call the existing `emailService.sendEmail()`, which owns its own `email_log` insert), `routes/admin/orders.ts` (thin HTTP layer — `GET /meta`, `GET /`, `GET /:reference`, `POST /:reference/resend-receipt`, `POST /:reference/resend-download`, wired into `worker/index.ts` with `/meta` ordered before `/:reference`).

**Role-gating divergence, the mirror image of Stage 1/2's**: list/detail are open to all three roles (support included — read visibility is a legitimate part of the support workflow). The two resend actions are `super_admin`/`editor` only, via `requireRole()` — the first Phase 3 write endpoints gated this way, because this is the first Phase 3 module where a mistake has a real, external, customer-facing consequence (an unwanted email), matching Products' `EDITOR_ROLES` convention instead of Consultation/Contact Manager's all-roles-can-write one. Enforced server-side; the frontend hiding the buttons for `support` is UX only.

**Reference validation**: `/^RWL-\d{4}-\d{6,}$/`, checked in the route file before any D1 query, identical convention to the public `routes/purchases.ts`.

**Synthetic test-order flag**: `isSyntheticTestEmail()` matches the known QA pattern `checkout+rwl-...@robayerwealthlab.com` (see `docs/v1.1-ceo-dashboard.md`/`docs/analytics-implementation-plan.md`'s prior finding) and surfaces a "Test" badge in both the list and the drawer, so QA activity is never mistaken for a real sale.

**No new migration** — pure reads plus reuse of the existing `sendEmail()`; zero new tables or columns.

**Frontend**: `admin/orders/index.html` (replaces the empty shell), `js/components/admin/admin-orders.js` — data table (Reference, Product, Customer email, Amount, Status, Date), search (reference/email substring), status chips, date-range filters, and a drawer (reusing Stage 1's `.drawer*` CSS) showing order details, payment transactions, deliveries, and email history, with the two resend buttons hidden entirely (not just disabled) for `support`.

**Local dev infrastructure fix, found while trying to verify this stage in a real browser**: the existing `.claude/launch.json` served the static frontend (`npx serve`, port 5500) and the API (`wrangler dev`, port 8787) as two unrelated servers with no proxy between them — a relative `fetch('/api/...')` from a page on :5500 resolved to :5500 itself (404), never reaching the Worker at all. Added `scripts/dev-proxy.mjs` (static file server + `/api/*` reverse proxy to :8787, Node built-ins only) and pointed `launch.json`'s `robayer-static-site` config at it. Fixes real-browser local verification for this and every future admin module, not just this stage.

**Real defect found and fixed**: the drawer's resend-button footer used `element.hidden = true` to hide it for the `support` role, but `css/admin.css`'s `.drawer__footer { display: flex; ... }` (added in Stage 1 for the always-visible mailto-link footer, which was never previously toggled) unconditionally overrides the browser's default `[hidden] { display: none }` — same specificity, author stylesheet always wins over the UA stylesheet regardless of order. Result: a `support`-role admin would have *seen* the resend buttons even though the server correctly rejected the action (403) if clicked. The server-side rejection meant this was never an actual security hole, but it was a real, confusing UI bug — a support agent seeing an action they can't use. Fixed with `.drawer__footer:not([hidden])`. Found and fixed via real browser testing (a curl-only pass would never have caught this — the JSON response and the DOM's `hidden` attribute were both already correct; only the rendered CSS was wrong).

**Local verification** (`wrangler dev` behind the new proxy + real browser): fresh super_admin and support logins (JS-dispatch workaround for the same Browser-tool click-simulation quirk documented in Stage 1/2), list render with a real verified test order + the pre-existing failed order + a synthetic-flagged order, drawer open/detail-load showing payment transaction + delivery + (correctly empty) email history, search by reference and by email substring, status-chip and date-range filtering, "Test" badge rendering. Resend buttons: visible+enabled for `super_admin` on the verified order, visible+disabled on the non-verified order, hidden entirely for `support` (after the CSS fix above) regardless of order status.

**Adversarial pass** (curl against live `wrangler dev` + real local D1): unauthenticated list/detail/resend → 401; `support` read → 200; `support` resend-receipt and resend-download on a verified order, real CSRF token → both 403 `FORBIDDEN`, confirming the role check is server-side and not just UI-hidden; mutation without CSRF token → 403; mutation with a garbage CSRF token → 403; resend on a non-`verified` order (the pre-existing `RWL-2026-000002`, status `failed`) → 400 `VALIDATION_ERROR`, no email attempted; a SQL-injection-shaped reference (`RWL-2026-900001'; DROP TABLE purchase_sessions; --`) → 404, rejected by the regex before ever reaching D1, table confirmed intact; a well-formed but non-existent reference → 404; invalid `status` filter value → silently ignored (falls back to unfiltered) rather than erroring, matching this project's established tolerant-filter convention.

**Real end-to-end email verification**: local `wrangler dev` could not be used for this — `backend/.dev.vars`'s `RESEND_API_KEY` is itself invalid (`401 API key is invalid` from Resend's own API), confirmed pre-existing and unrelated to this stage (an unrelated `contact-acknowledgement` send failed identically at 08:05, before Stage 3 work began; also confirmed not a stale-process artifact by fully restarting `wrangler dev` and retrying). The failure path itself was verified correct: `resendReceipt()`/`resendDownload()` both detected `sent: false`, returned a clean `INTERNAL_ERROR` to the caller instead of throwing, and correctly skipped the `auditService.record()` call — no false "resent" audit entry for an email that was never actually sent. Real send verification deferred to production, where Stage 1/2's own live acknowledgement emails already prove the production Resend secret is valid — see the production verification section below.

**Fresh code re-read before deploy**: `orderService.ts` and `routes/admin/orders.ts` re-read in full immediately before deploying — no defects found beyond the CSS bug above (already fixed and re-verified).

**Test data**: 2 disposable local admins (one per relevant role) + 2 disposable local purchase sessions (one real-shaped verified order with a payment transaction and delivery row, one synthetic-email-flagged order) + their related rows, all deleted before considering the local phase complete; local baseline confirmed restored to its pre-existing state (1 admin, 1 purchase session — a prior, unrelated leftover row from earlier engagement work, left untouched as out of scope).
