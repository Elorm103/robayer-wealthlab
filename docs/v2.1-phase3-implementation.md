# Version 2.1 Phase 3 — Identity & Security: Implementation Report

**Status:** Complete, deployed to production, verified.
**Scope:** Change/forgot/reset password, password strength policy, account lockout with dedicated login history, active-session viewing and revocation, forced password-change gate.

## Scope delivered vs. the original architecture plan

`docs/v2.1-architecture-plan.md`'s Phase 4 draft additionally proposed Profile features (update name, change email with confirmation, avatar, notification preferences). None of those were in the user's own Phase 3 requirement list (Authentication / Sessions / Security only) and none are built here — a deliberate scope match to what was actually requested, not an oversight. If profile self-service is wanted later, it's a small, additive follow-up (no schema conflicts with anything built in this phase).

## Database — migration `0013_identity_security.sql`

```sql
ALTER TABLE admin_users ADD COLUMN must_change_password INTEGER NOT NULL DEFAULT 0;
ALTER TABLE admin_users ADD COLUMN failed_login_attempts INTEGER NOT NULL DEFAULT 0;
ALTER TABLE admin_users ADD COLUMN locked_until TEXT;
ALTER TABLE admin_users ADD COLUMN password_updated_at TEXT;

CREATE TABLE password_reset_tokens ( ... );  -- 256-bit token, 30-minute TTL, single-use
CREATE TABLE login_history ( ... );          -- outcome IN ('success','failed_password','failed_locked','failed_inactive')
```

`login_history` is written alongside (not instead of) the existing `audit_logs` `admin.login`/`admin.login_failed` events — a dedicated table so a "Login History" UI doesn't filter a large, unrelated event stream on every page load, matching the `consultation_notes`/`contact_notes` precedent.

**Note on `failed_inactive`:** the CHECK constraint includes this outcome for schema completeness, but no code path in this phase produces it. `login()`'s existing lookup (`WHERE email = ? AND is_active = 1 AND deleted_at IS NULL`) deliberately can't distinguish "nonexistent email" from "inactive account" without a second query — and building that second query would reopen exactly the account-enumeration side channel the login endpoint has always avoided. Both cases collapse into the same generic outcome, matching the response the requester already sees.

## Backend

- **`backend/utils/passwordPolicy.ts`** (new) — `validatePasswordStrength()`: 12-character minimum (length over composition rules, matching current NIST 800-63B guidance and this project's own stated preference for real, tested measures over box-ticking ones), a small weak-pattern deny-list, and a check against the account's own email local-part. One shared function, used identically by change-password and reset-password.
- **`backend/utils/adminSessionToken.ts`** (extended) — `generatePasswordResetToken()`, same 256-bit `crypto.getRandomValues()` pattern as the session/CSRF tokens already there.
- **`backend/services/admin/loginHistoryService.ts`** (new) — the only code that writes to `login_history`.
- **`backend/services/admin/sessionService.ts`** (extended) — `listSessions()`, `revokeSessionById()` (IDOR-checked — verifies `admin_id` matches the caller before revoking), `revokeAllSessionsExcept()`, `revokeAllSessions()`. `validateSession()`'s SELECT now also returns `must_change_password`, since `requireAuth()` needs it on every request.
- **`backend/services/admin/authService.ts`** (extended):
  - `login()` gained account lockout: 5 consecutive failures locks the account for 15 minutes (time-boxed, not permanent — a permanent lockout needing another admin's manual intervention is a real support burden for a 1–3-person team). A locked account still runs the full PBKDF2 comparison against its real hash (not the dummy hash) so a locked-vs-wrong-password attempt stays timing-indistinguishable; the lockout decision is made independently of whether the password was actually correct.
  - `changePassword()` — requires current-password re-entry, validates strength, revokes every **other** session (keeps the one that just made the change).
  - `forgotPassword()` / `resetPassword()` — same no-enumeration discipline as `login()` (identical generic response whether or not the email exists); reset-password revokes **every** session (forced logout, per the user's explicit requirement) and atomically marks the token used, closing the same two-requests-race window `used_at IS NULL` already closes elsewhere in this codebase (unsubscribe tokens, download tokens).
- **`backend/middleware/requireAuth.ts`** — the one file genuinely touching the core auth path: rejects every `/api/admin/*` request with `MUST_CHANGE_PASSWORD` when the flag is set, except `session`/`change-password`/`logout`. Centralized here so no other route (existing or future) needs to remember this check itself.
- **`backend/routes/admin/auth.ts`** — new endpoints: `POST change-password`, `POST forgot-password`, `POST reset-password`, `GET sessions`, `POST sessions/:id/revoke`, `GET login-history`. Rate limits: forgot-password 3/15min (an unlimited version is itself a spam/harassment vector against a real inbox), reset-password 10/15min, change-password 10/15min.
- **`backend/emails/templates/password-reset.html`** (new) + `emailService.ts` registration.
- New `ApiErrorCode`s: `MUST_CHANGE_PASSWORD` (403), `INCORRECT_PASSWORD` (401), `INVALID_TOKEN` (400).

## A real bug found and fixed during verification

`admin_sessions.last_seen_at` is written in two different formats depending on the code path: `datetime('now')` at session creation (SQL, space-separated, no timezone suffix) vs. `new Date().toISOString()` on every subsequent validated request (`sessionService.ts`'s `validateSession()`, already `T`-separated with a `Z` suffix). The Account page's `formatDate()` assumed the first format only, producing "Invalid Date" for the current session's own row — the one row guaranteed to have gone through the second path. Fixed by normalizing both formats before parsing. Found via real browser verification, not by inspection.

## Verification performed

**Local (`wrangler dev`):**
- Lockout: 5 wrong passwords locked the account; the 6th attempt was rejected even with the correct password (`failed_locked` recorded distinctly from `failed_password` in `login_history`).
- Change-password: revoked all other sessions (5 → 1), old password rejected afterward, new password worked. Strength/policy rejections confirmed individually (too short, weak pattern, contains email, same-as-current, wrong current password).
- Forgot/reset-password: identical response for a real vs. nonexistent email; real token consumed the password, all sessions revoked, and reusing the same token failed (`INVALID_TOKEN`).
- `must_change_password` gate: session-check reported the flag; `sessions`, `login-history`, and an unrelated module route (`resources`) were all rejected with `MUST_CHANGE_PASSWORD`; the browser redirected to `/admin/account/`; the locked-down view rendered correctly; change-password cleared the flag and unlocked everything.
- IDOR probe: an authenticated admin could not revoke a different admin's session by id (`NOT_FOUND`, and the target session's `revoked_at` was confirmed still `NULL` afterward).
- CSRF: a mutating request with no `X-CSRF-Token` header was rejected (`FORBIDDEN`).
- SQL injection probe on the login form: rejected as ordinary invalid credentials (parameterized queries throughout, as everywhere else in this codebase).

**Production:** disposable admin created via direct D1 SQL (never the real `robayerwealthlab.admin@gmail.com` account) — login, change-password, forgot-password (confirmed `email_log` shows `status: sent` via the real Resend API), reset-password (real token retrieved from D1, consumed, new password verified), then full cleanup (`admin_sessions`, `login_history`, `password_reset_tokens`, `email_log`, `audit_logs`, `admin_users` rows deleted). Real admin account's `failed_login_attempts`/`locked_until`/`must_change_password` confirmed unchanged (`0`/`NULL`/`0`) after the pass.
