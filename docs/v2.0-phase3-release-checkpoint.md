## Known limitations and future work

1. **Static-asset browser caching is a pre-existing, site-wide characteristic, not something this phase introduced or fixed.** GitHub Pages/Cloudflare's default `Cache-Control: max-age=14400` applies to every JS/CSS file on the site; a returning admin's browser may serve a stale cached script for up to 4 hours after a deploy. Confirmed as the root cause of one apparent (but not real) Stage 4 production regression. Site-wide cache-busting/versioned filenames would fix this but is a materially larger, unrequested scope change â€” recommended as a standalone future item, not bundled into this phase.
2. **Production's own leftover-test-data cleanliness for Stages 1â€“4 rests on those stages' own already-documented, independently-verified cleanup** (each stage's deployment report in `docs/v2.0-phase3-implementation.md` records an exact before/after row-count match). Stage 5's acceptance audit did not re-create test data in production to re-verify this from scratch, since doing so would itself require creating new production rows for no net benefit â€” the existing per-stage evidence was judged sufficient.
3. **`consultation_requests`/`contact_messages` currently have 0 rows in production** â€” both features are fully built and verified (Stages 1/2's own deployment reports in `docs/v2.0-phase3-implementation.md` record real end-to-end submission tests against production, since cleaned up) but have not received further real customer submissions since. Not a defect â€” genuinely awaiting real traffic.
4. **No automated test suite** â€” every verification in this phase (as in every prior phase) was manual: real browser interaction, direct `curl` adversarial probes, and hand-computed SQL comparisons. Consistent with this codebase's existing testing posture; not a regression introduced by this phase.

**Recommended next phase**: per `docs/v2-development-roadmap.md`, the remaining shell modules (Resources, Blog, Newsletter, Settings, Users) are the next logical scope, now that this phase has proven the "support-workflow modules can safely diverge from Products' stricter role-gating" and "every admin JSON response gets `Cache-Control: no-store` by default" patterns they'll reuse.

## Rollback procedure

1. **Worker code:** `cd backend && npx wrangler rollback` (reverts to the previously deployed Version ID), or `git checkout <prior-commit> && npx wrangler deploy` (`fc14e0d` is the last commit before Phase 3 began entirely).
2. **Database:** migrations in this project are forward-only by convention (no `.down.sql` files exist). Rolling back `0010`'s schema would require a hand-written reverse migration; since it only adds nullable-by-default columns and new tables with no other table's foreign key pointing at them, a rollback carries low risk of orphaning data, but should still not be attempted without first confirming no post-migration writes (new notes, new assignments) would be lost.
3. **Cache-Control regression specifically:** if the Stage 5 global `Cache-Control` rule in `middleware/securityHeaders.ts` needs to be reverted in isolation, remove the `if (!isHtmlResponse(response))` block added in that function and redeploy â€” a fast, code-only, single-file rollback.
