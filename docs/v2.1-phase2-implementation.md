# Version 2.1 Phase 2 — Implementation Log (Blog CMS)

Tracks what was actually built against `docs/v2.1-architecture-plan.md` Section 3, refined by the user's Phase 2 kickoff message (draft/published states, rich text, featured image, categories/tags, SEO fields, author attribution, preview before publishing, `publishedAt`, search/filtering, soft delete/restore, audit logging). Written after implementation, matching the established `docs/v2.1-phase1-implementation.md` format.

---

## Scope delivered

Full D1-backed CMS for the blog, mirroring the Resources Module's proven pattern (D1 is the sole source of truth, server-rendered public page via a dedicated Workers Route, no GitHub-publish pipeline). The one deliberate simplification versus the original architecture plan: **two states only (draft/published)**, not the 5-state draft/in_review/scheduled/published/archived lifecycle an earlier planning pass considered — the user's own Phase 2 brief asked for exactly "Draft and Published states," and soft delete (`deleted_at`) already covers "remove from public view without deleting," making a third `archived` status redundant.

**Migration `0012_blog_cms.sql`** (production, applied before any code deploy): one new table, `blog_posts` — `post_id`/`slug` (matching `products`/`resources`' identifier convention), `category` (CHECK-constrained to `saving`/`investing`/`budgeting` — the real vocabulary confirmed against the live static pages' filter pills before choosing the constraint), `tags`, `status` (`draft`/`published`), `featured`, `cover_media_id` (FK to `media_assets`), `author_id` (FK to `admin_users` — real attribution, not a free-text field), SEO fields, `published_at`. Four indexes (status, category, deleted_at, created_at). Purely additive.

**Backend**: `services/blogService.ts` (CRUD, publish/unpublish lifecycle, soft delete/restore, duplicate — a trimmed, field-renamed copy of `resourceService.ts`'s shape, plus two genuinely new pieces: real author resolution via `LEFT JOIN admin_users` and `estimateReadingTimeMinutes()`, a derived value computed from the stored body's word count at read time, not an admin-editable field), `routes/admin/blog.ts` (thin HTTP layer, `editor`/`super_admin`-gated writes, imports `listAssignableAdmins()` from `services/admin/consultationService.ts` for the editor's author dropdown rather than duplicating that query), `routes/blog.ts` (public `/blog/*` Workers Route, server-rendered, mirroring `routes/resources.ts`'s "own the whole path" pattern).

**Preview before publishing**: `?preview=1` on a detail page bypasses the published-only filter only if the request carries a valid admin session — reusing `requireAuth()` directly rather than building a separate preview-token table. A logged-out visitor requesting the same URL, with or without the query param, gets an honest 404; a draft's existence is never revealed.

**Rich-text table support**: `richTextSanitizer.ts`'s server-side allowlist was extended with `table`/`thead`/`tbody`/`tr`/`th`/`td` (low-risk — the same blanket attribute-strip already applies to them) so the one migrated legacy article's comparison table could survive sanitization. The admin editor's client-side sanitizer (`admin-blog-editor.js`) was built with the identical extended allowlist from the start, specifically to avoid a silent-data-loss trap: without it, opening the migrated article in the visual editor and clicking Save would have stripped its table client-side before the request ever reached the server.

**Admin frontend**: `/admin/blog/` (list: search/filter by category+status+featured/sort/pagination/bulk actions, mirroring `admin/resources/`'s structure, author column instead of downloads), `/admin/blog/new/` + `/admin/blog/edit/` (editor: title/slug/excerpt, rich-text body via the established contenteditable+`execCommand`+client-and-server-sanitized pattern, cover-image Media Library picker, author select populated from `/api/admin/blog/meta`, SEO fields, tags/featured, status select, a "Preview" link to the public `?preview=1` view, live preview, danger zone). Zero new CSS primitives.

## Content migration: "What Are Treasury Bills in Ghana?"

The one real, SEO-indexed legacy article (`blog/what-are-treasury-bills-in-ghana/`) used a pull-quote, styled alert boxes, a "Key takeaways" card with checkmark icons, a sticky table-of-contents, and a per-article FAQ block — none reproducible in the new schema's plain rich-text body. Per the user's explicit choice (via `AskUserQuestion`, "Migrate with reduced fidelity"): the pull-quote became a plain `<blockquote>`, the two alert boxes became plain paragraphs, the "Key takeaways" card became a plain `<ul>`, and the sticky TOC and per-article FAQ were dropped entirely (the disclaimer paragraph is emitted by the route template itself, not stored in the post body, so it was not duplicated). The one piece of real structured content that *is* preserved — the tenor/rate comparison table — is exactly why the sanitizer allowlist was extended.

Real metadata preserved: category `investing` (from the article's own `article:section`), `publishedAt` backdated to `2026-06-27` (the article's real original publish date — the create/update API has no way to set a custom `publishedAt`, since every other post's is set automatically on first publish, so the migration inserted this one row directly via D1 SQL rather than through the API, after building and verifying the exact same content locally through a real create-then-publish API call first). `dateModified` (`2026-07-01` on the old static page) has no equivalent field distinct from `updated_at` in the new schema and was not preserved — a deliberate, minor scope acceptance, not an oversight. No cover image was set (the legacy page never had a dedicated one; the public route already falls back to the site's default OG image).

## Real defect found during local verification, fixed before deploy

**`.admin-bulk-bar[hidden]` had no `display: none` override.** `.admin-bulk-bar` sets `display: flex` unconditionally; the bare `[hidden]` attribute alone loses to an author class selector at equal specificity, so the bar rendered visible even when marked `hidden`. Found while visually checking the new Blog list page in a real browser — the Publish/Unpublish/Delete bulk-action buttons were showing with zero rows selected. This bug predates Phase 2 (`.admin-bulk-bar` is shared by Resources and Products, confirmed by checking `/admin/resources/` and finding the identical symptom), but this codebase has an established, documented fix for exactly this class of bug (`.empty-state[hidden]`, `.btn[hidden]`, `.alert[hidden]`, etc., each with a matching comment). Fixed by adding `.admin-bulk-bar[hidden] { display: none; }` following the same pattern — a one-line, low-risk fix that also corrects the pre-existing bug on Resources and Products' list pages.

## Local verification (`wrangler dev` + real browser + adversarial battery)

**Browser**: logged in as a disposable local `super_admin`; list page rendered correctly (bulk bar fix confirmed live); editor pages rendered with populated category/author/status selects; created the real migrated article through a direct authenticated `adminFetch()` call (not the visual editor, since the contenteditable toolbar has no table-authoring UI and pasting strips HTML to plain text — table-bearing content can only enter the system pre-formed, exactly as the sanitizer's own header comment states), then published it via the status-transition endpoint; confirmed the table, blockquote, and full flattened content rendered correctly on the real public detail page (hit directly on `localhost:8787`, since this project's local dev proxy only forwards `/api/*`); re-opened the post in the visual editor, confirmed the table loaded into the contenteditable field intact, and confirmed a real Save round-trip through the client sanitizer preserved the table server-side — the exact scenario the client allowlist extension exists to prevent.

**Adversarial pass** (via authenticated `fetch`/`adminFetch` calls in the real browser session): a `support`-role admin was rejected creating (`403`) and deleting (`403`) posts but could still list them (`200`) — role gating confirmed server-side; mutation with no `X-CSRF-Token` header → `403`; mutation with a wrong/garbage token → `403`; a SQL-injection-shaped slug (`x'; DROP TABLE blog_posts; --`) was rejected by slug-format validation before ever reaching a query, table confirmed still present afterward; a stored-XSS attempt (`<script>`, an `<img onerror>`, a `javascript:` link) was fully neutralized by the sanitizer — the script tag was unwrapped to inert text, the image was removed outright (unsafe `src`), the link survived with its text but no `href`; publishing a post with an empty body was rejected both at creation-time status-transition and at update-time (`"A post needs body content before it can be published."`); the preview URL for a draft post returned a clean `404` to a genuinely anonymous request (`credentials: 'omit'`, bypassing this browser's host-scoped — not port-scoped — shared cookie jar) both with and without `?preview=1`, returned `200` with a visible preview banner for an authenticated admin, and the same published post returned `200` anonymously as expected.

**Test data**: two disposable draft posts (one XSS probe, one empty-body probe) were soft-deleted after verification; the disposable `support`-role admin was deactivated (a hard delete was blocked by a foreign-key constraint from its own audit log rows — deactivation is the correct disposal here, matching how a real disabled account would be handled). The one real post (the migrated article) was left in place, as intended.

## Known limitations

1. **No revision history for posts** — matches Resources and Products' own posture; not a regression specific to this phase.
2. **Local dev proxy doesn't forward `/blog/*`** — the same pre-existing `scripts/dev-proxy.mjs` gap already noted in the Phase 1 log for `/resources/*` and `/books/*`. Verifying the public Blog route locally requires hitting the Worker directly on `localhost:8787`.
3. **No table-authoring UI in the rich-text editor** — the contenteditable toolbar has no `insertTable` command, and pasted HTML is stripped to plain text by design (this field renders raw to every site visitor, so paste-as-plain-text is a deliberate security posture, not an oversight). The extended sanitizer allowlist lets an already-present table survive editing and re-saving; it does not let an editor author a new one from the UI. A future phase could add a table-insert toolbar button if this becomes a recurring authoring need.
