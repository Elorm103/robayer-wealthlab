# Version 2.1 — Technical Debt Register

Companion to `docs/v2.1-release-checkpoint.md` (the Final Acceptance Audit). This is **not** a list of bugs — every item below is a deliberate, already-approved architectural decision made somewhere across Version 2.0/2.1 that trades some future capability for simplicity today. Each entry names the current approach, why it was chosen, the concrete signal that should trigger revisiting it, and the recommended direction when that signal appears.

Two items (8 and 9) were added after the register's initial draft, once the Phase 7 audit's operational-documentation review surfaced them while correcting `docs/backup-and-recovery.md` — both are genuine current gaps rather than mitigated trade-offs, listed here (not silently left implicit in the corrected runbook) so they have an explicit owner and trigger like every other item.

## 1. Newsletter sending: `ctx.waitUntil()` + a durable roster table, not Cloudflare Queues

**Current approach:** `campaignService.ts` sends a campaign by processing its `newsletter_campaign_recipients` roster inside a single `ctx.waitUntil()` background task, one recipient at a time, with a configurable safety cap (`site_settings.campaign_recipient_cap`, default 300).

**Why it was chosen:** Cloudflare Queues, Cron Triggers, and Durable Objects do not exist anywhere in this project. Introducing any of them for a subscriber list that has never exceeded a few dozen real people would have been infrastructure ahead of need. The durable per-recipient roster with atomic conditional `UPDATE` claims already provides the properties that would otherwise motivate a queue — duplicate-send prevention and safe mid-send resume — without the operational surface area of a new primitive.

**Future trigger:** The subscriber list approaching or exceeding the configured cap on a regular basis, or a real need for sends larger than a single Worker invocation's practical execution budget can comfortably process.

**Recommended long-term direction:** Migrate to Cloudflare Queues once volume justifies it — the roster table's shape (one durable row per intended recipient, already idempotent on `(campaign_id, subscriber_id)`) requires no redesign to become a queue's producer/consumer model; only the dispatch mechanism changes.

## 2. GitHub Pages (static site) + Cloudflare Worker (API) as two separately deployed origins made same-origin via a Workers Route

**Current approach:** The public site is static files served by GitHub Pages; `/api/*` is routed to the Worker via a Cloudflare Workers Route on the same domain, established during the "same-origin migration" work.

**Why it was chosen:** GitHub Pages is free, simple, and already the site's home; moving the entire static site into the Worker would mean rebuilding a working, git-native publishing flow for no functional gain. The Workers Route achieves same-origin cookie/CSRF semantics without that migration.

**Future trigger:** A genuine need for server-side rendering, request-time personalization, or edge logic on the public-facing pages themselves (not just the admin/API surface) that a static host cannot provide.

**Recommended long-term direction:** No change needed until that trigger appears. If it does, Cloudflare Pages (not Workers) is the more natural next step, since it preserves the git-native static publishing model while adding Workers-adjacent capabilities.

## 3. Deployment metadata (`DEPLOYED_COMMIT`/`DEPLOYED_AT`) supplied via `wrangler deploy --var`, not runtime introspection

**Current approach:** Every production deploy passes the current commit hash and timestamp as build-time variables; the Worker has no way to determine these at runtime on its own.

**Why it was chosen:** Cloudflare Workers have no built-in equivalent of reading `git rev-parse HEAD` at runtime — the only two options were passing it in at deploy time or building a CI step to inject it into source before build. The former is simpler and has no failure mode (a forgotten `--var` just leaves the field blank, which is self-evidently a deployment-process gap rather than a silent lie).

**Future trigger:** Moving deployment off manual `wrangler deploy` calls and onto a CI/CD pipeline (GitHub Actions or similar) — at that point, injecting these vars automatically becomes nearly free and removes the human step that could forget them.

**Recommended long-term direction:** Automate the `--var` injection from CI once one exists; no urgency before then, since a missing value is immediately visible on the Settings diagnostics page, not a silent correctness bug.

## 4. Fully synchronous email sending with no automated retry scheduler

**Current approach:** `emailService.ts` calls Resend inline, at the moment the triggering action happens, and records the outcome to `email_log`. A failed send is recorded as `status = 'failed'` and nothing automatically retries it — the only path to redelivery today is a human noticing and manually re-triggering the underlying action (or, for campaigns specifically, the explicit Resume flow).

**Why it was chosen:** No Cron Trigger infrastructure exists in this project (see item 1). Building a scheduled retry-drain loop for email specifically would be the first Cron Trigger introduced anywhere in the codebase, for a failure mode (Resend transient errors) that has been rare in practice.

**Note found during this audit:** `docs/backup-and-recovery.md` and `docs/monitoring-and-alerting.md` both describe a "Cron-Trigger-driven `email_log` retry loop" as if it exists — it does not. This is flagged separately as a documentation-accuracy issue in the release readiness report; it's called out here too because it's the clearest illustration of this debt item's actual current state: failures are captured durably but not automatically retried.

**Future trigger:** A real, observed incident where a Resend outage caused a meaningful number of unrecovered failed sends that required manual intervention to fix.

**Recommended long-term direction:** A lightweight Cron Trigger that queries `email_log WHERE status = 'failed'` periodically and retries with backoff — a small, self-contained addition once the underlying Cron primitive is introduced for any reason (see item 1's migration trigger, which would likely arrive first and make this one cheap to add alongside it).

## 5. PBKDF2 at 100,000 iterations, not OWASP's higher current guidance

**Current approach:** `backend/utils/passwordHash.ts` uses 100,000 PBKDF2-SHA256 iterations.

**Why it was chosen:** This is **not a design preference** — it is the hard ceiling of Cloudflare Workers' `SubtleCrypto` implementation, discovered in production (`NotSupportedError` above 100,000). It remains a real, credible iteration count, well above legacy defaults, just below OWASP's current upper recommendation for dedicated server hardware.

**Future trigger:** Cloudflare raising or removing this platform ceiling, or this project adopting a different hashing primitive (e.g., via a WASM-based Argon2 implementation) that isn't subject to it.

**Recommended long-term direction:** No action needed unless the platform changes — this is a platform constraint, not a project decision, and re-litigating it without a platform change would accomplish nothing.

## 6. Last-Super-Admin guard is correct but has never been exercised against real production data

**Current approach:** `adminUserService.ts`'s `guardLastSuperAdmin()` blocks disabling, deleting, or demoting the sole remaining active `super_admin`; combined with `guardTarget()`'s blanket block on any admin acting on their own account, the guard is reachable and correct whenever a real multi-admin roster exists (e.g., admin A can be blocked from demoting the last-remaining super_admin B, since A ≠ B).

**Why it's listed here:** Production currently has exactly one real admin account (the founder's). Every verification of this guard across Phases 3, 4, and 7 has necessarily used disposable, non-production admin accounts to exercise the multi-admin scenario the guard protects — the guard's logic has been proven correct, but its real-world protective effect has not yet been observed against a genuine second production admin.

**Future trigger:** The team growing to two or more real admin accounts in production.

**Recommended long-term direction:** No code change needed. Worth a one-time manual confirmation the first time a second real admin is invited, simply to close the loop on this being observed in the real environment it was built for.

## 7. `wrangler dev`'s route-pattern simulation makes `environment` always read `"production"` locally

**Current approach:** The Settings diagnostics page's `environment` field reads `"production"` even under local `wrangler dev`, because Wrangler's local route-pattern simulation doesn't distinguish the two the way real deployed routing does.

**Why it was chosen:** This wasn't chosen — it's a `wrangler dev` limitation discovered during Phase 5 verification, judged harmless and self-correcting (it reads correctly the moment code is actually deployed) rather than worth working around with an artificial environment-detection hack that would itself be a source of drift between local and real behavior.

**Future trigger:** None expected; this is cosmetic and self-correcting by nature.

**Recommended long-term direction:** No action. Documented here only so a future developer who notices it locally doesn't mistake it for a real bug.

## 8. No automated backup export beyond D1's built-in Time Travel retention window

**Current approach:** D1's own point-in-time recovery ("Time Travel") is the only backup mechanism this project has. A weekly Cron-Trigger-driven export to R2 was planned in the original `docs/backup-and-recovery.md` but never built.

**Why it's listed here:** This is not a mitigated trade-off — it's a genuine current gap, found and corrected during the Phase 7 audit when the runbook describing it as if built was rewritten to state the truth. It's listed in the register rather than only in the corrected runbook so it has an explicit owner and revisit trigger, matching every other item here.

**Future trigger:** Either D1 Time Travel's retention window is judged insufficient for this project's real recovery needs, or a real incident occurs where data older than the retention window needed to be recovered and couldn't be.

**Recommended long-term direction:** If the trigger occurs, the original plan is still sound and requires no redesign: a Cron Trigger running `wrangler d1 export` (or the equivalent D1 API) on a schedule, writing to R2. This would be the first Cron Trigger introduced anywhere in this project — see item 1's migration trigger, which would likely arrive first and make this one cheap to add alongside it.

## 9. Media Library uploads (R2) have no backup path

**Current approach:** An admin-uploaded file in the Media Library exists only as bytes in the `STORAGE` R2 bucket; the `media_assets` D1 row records metadata (filename, content hash, dimensions) but not the file content itself. If the R2 bucket were lost, every admin-uploaded file would be unrecoverable — only the metadata row would remain.

**Why it's listed here:** This is a real change in risk profile from `docs/backup-and-recovery.md`'s original architecture, which assumed every R2 object was either git-rebuildable (product/resource assets sourced from `assets/products/`, `assets/covers/`) or D1-regeneratable (receipts). Media Library (Version 2.0 Phase 1) introduced true user-uploaded originals with no such fallback, and the backup documentation wasn't updated to reflect it until this Phase 7 audit caught the gap.

**Future trigger:** Media Library upload volume or importance growing to the point where losing the bucket would be a genuinely costly incident (today, at low volume, most uploads could plausibly be re-sourced or re-uploaded manually if lost).

**Recommended long-term direction:** If the trigger occurs, R2 has no native versioning, so the practical option is a periodic copy job (could reuse the same Cron Trigger primitive as item 8, if that's ever built) that mirrors new/changed objects in `media/` to a second bucket or external storage. Not built now — this item exists so the gap is visible rather than silently assumed away.
