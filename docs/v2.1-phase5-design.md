# Version 2.1 Phase 5 — Settings: Design Document

**Status:** Design only. No implementation code has been written. Per the user's own instruction, implementation begins only after this document is reviewed and approved.

---

## 1. Current configuration architecture (audit findings)

Read fresh for this phase: `backend/wrangler.jsonc`, `backend/worker/env.ts`, `assets/config/site.json`, `backend/routes/health.ts`, `backend/services/emailService.ts`, `backend/services/productCatalogService.ts`/`productService.ts` (download-policy fields), `backend/database/schema.sql`'s `payment_transactions`/`email_log` sections, `docs/deployment-checklist.md`, `admin/settings/index.html` (confirmed: still the original empty "Coming soon" shell).

Configuration in this project already lives in **four genuinely distinct places**, each with a different owner and a different reason for existing there. Phase 5 does not blur these boundaries — it adds a fifth, narrowly scoped one.

| Layer | Owner | Examples today | Editable by this admin? |
|---|---|---|---|
| **Cloudflare Secrets** | Whoever has Cloudflare account access, via `wrangler secret put` | `PAYSTACK_SECRET_KEY`, `RESEND_API_KEY` | Never — not even a future version of this admin. This is a hard, non-negotiable boundary already established in the Version 2.1 kickoff decision on Payment Configuration. |
| **`wrangler.jsonc` vars** | Whoever has repo + deploy access, requires a Worker redeploy to change | `SITE_BASE_URL`, `PAYMENT_PROVIDER`, `PAYSTACK_BASE_URL` | No — changing these means changing code and redeploying, not an admin-panel action. |
| **`assets/config/site.json`** | Whoever has repo access, edited via direct file commit + GitHub Pages rebuild | Company info, contact emails, social links, branding paths | No — explicitly out of scope per the user's own instruction this phase (and the Version 2.1 kickoff's Decision 1). |
| **D1 (`admin_users`, `products`, etc.)** | This admin, via existing CRUD modules | Per-product `max_downloads`/`download_expires_days`, admin roles, content | Yes, already — but there is **no site-wide default** for anything today; every "default" that exists is either hardcoded in code (e.g. `emailService.ts`'s `from:` address) or simply absent (a product with no `max_downloads` set is unlimited, full stop, with nothing to fall back to). |

**The gap Phase 5 fills:** a fifth layer — **genuinely runtime, Worker/D1-native operational configuration that should be changeable by a Super Admin without a code deploy, but was never given anywhere to live.** Maintenance mode doesn't exist at all today. Email sender info and the Resend `from:` address are hardcoded in `emailService.ts`. Download defaults are per-product only, with no site-wide fallback. This is exactly the gap the original architecture plan's Site Settings recommendation identified, and nothing about auditing the current system this phase changes that conclusion.

**Payment and email diagnostics need no new storage at all** — `payment_transactions` (webhook/payment history) and `email_log` (send history) already exist and are already populated by every purchase and every email this system sends. Phase 5's payment/email visibility sections are read-only aggregation queries against tables that already exist, not new tracking.

## 2. Proposed settings model

One new table, matching the exact key-value shape the Version 2.1 architecture plan already proposed and the user has not objected to:

```sql
CREATE TABLE site_settings (
  key          TEXT PRIMARY KEY,
  value        TEXT NOT NULL,                  -- JSON-encoded; typed per-key at the service layer, not the DB layer
  updated_by   INTEGER REFERENCES admin_users(id),
  updated_at   TEXT NOT NULL DEFAULT (datetime('now'))
);
```

A single generic table, not one column per setting — a future operational setting becomes a new row via a service-layer default, not a new migration. This is a real, working instance of "feature flags" as a *mechanism* (the table itself is flag-shaped: any key can be added later without touching the schema again) — but this phase ships UI and behavior for only the settings explicitly requested below, not a generic "create your own flag" admin screen. That's a deliberate scope decision, not an oversight: no specific additional flag was requested, and building a generic flag-authoring UI now would be speculative.

**Settings this phase actually stores and uses:**

| Key | Type | Default | Effect |
|---|---|---|---|
| `maintenance_mode` | `{ enabled: bool, message: string }` | `{ enabled: false, message: "" }` | See §5 — gates the Worker-served surface only. |
| `default_max_downloads` | `number \| null` | `null` (unlimited) | Pre-fills a **new** product's download policy when left blank — see §5's scope note. |
| `default_download_expires_days` | `number \| null` | `null` (lifetime) | Same as above. |
| `email_sender_name` | `string` | `"Robayer WealthLab"` | Used in the Resend `from:` header. |
| `email_reply_to` | `string \| null` | `null` (no reply-to header sent) | New capability — `emailService.ts` sends no `reply-to` today. |
| `email_template_enabled` | `Record<templateName, bool>` | every template `true` | A per-template kill switch — see §5. |

Everything else the user asked for (webhook health, last successful payment/webhook, connection status, environment, email diagnostics, system information) is **derived, not stored** — computed live from `payment_transactions`, `email_log`, `env` presence checks, and the D1-native `d1_migrations` table on every `GET /api/admin/settings/status` call. Storing a second copy of data that already exists elsewhere would be a real, avoidable consistency risk for zero benefit.

## 4. API endpoints

All under `/api/admin/settings*`, all requiring `requireAuth` + `requireRole(['super_admin'])` — **Settings is `super_admin`-only for reads and writes alike**, the same posture as Phase 4's Users module and for the same reason: payment/email operational data and the ability to flip maintenance mode are not general content `editor`/`support` need visibility into.

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/admin/settings` | The editable `site_settings` key-value map (with defaults filled in for any key never explicitly set). |
| `PATCH` | `/api/admin/settings` | Partial update — body is a subset of the keys in §2, each validated server-side against its own type/range before writing. CSRF-protected, audit-logged with before/after per changed key. |
| `GET` | `/api/admin/settings/status` | Read-only diagnostics: payment (connection/environment/webhook health/last successful payment/last successful webhook), email (recent send counts and last-success timestamp per template, from `email_log`), system information (environment, current D1 migration, deployed commit + timestamp if available — see §6). |

No export/backup endpoint this phase — the user's current Phase 5 instructions don't ask for one (the earlier architecture-plan draft's "Backup/export where appropriate" isn't in this phase's actual scope list), and `site_settings` is small enough that adding one later, if wanted, is a trivial follow-up.

## 5. Configuration ownership and the two real code touch-points

Everything above is additive — one new table, one new route file, one new service. Two existing files are touched, both narrowly and both explained here in full per the "stop and explain if the architecture changes" instruction (neither is a change to core architecture; both are small, additive checks inside already-existing shared functions):

**(a) `worker/index.ts`'s fetch handler gains a maintenance-mode check**, run once per request, immediately after request-ID/logger setup and before route dispatch. When `maintenance_mode.enabled` is true, every route returns `503` with the configured message **except**:
- `/api/health` (must stay reachable for monitoring — it already carries "no auth, no state" as its explicit design intent).
- Anything under `/api/admin/*` (a Super Admin must always be able to log in and turn maintenance mode back off — the same reasoning the original architecture plan already stated).
- Anything under `/api/webhooks/*` (**a new exemption, reasoned through here**: Paystack's webhook delivery is asynchronous and server-to-server; blocking it during maintenance protects nothing a maintenance window is meant to protect, and *would* create a real, avoidable payment-reconciliation problem if a webhook for a payment made just before maintenance started got rejected instead of processed).

**Important architectural honesty, stated plainly:** because `/admin/*` (the frontend pages) are static files served directly by GitHub Pages — never proxied through this Worker — maintenance mode **cannot** take down the public marketing site, the admin login page, or any static page. It can only gate what this Worker actually serves: `/api/*` (including checkout, purchases, downloads), `/books/*`, `/resources/*`, `/blog/*`. This is a narrower blast radius than the word "maintenance mode" might suggest to someone unfamiliar with this project's split architecture, and it's the correct, honest scope given the instruction not to introduce a GitHub-publishing mechanism — a maintenance banner across the *entire* static site would require exactly that.

**Performance note, considered and accepted:** this adds one D1 read to every single Worker request, including ones that don't otherwise touch the database. Given this project's own repeatedly-stated realistic traffic volume ("a few requests per minute," per `docs/backend-security.md`), this is judged an acceptable simplicity trade-off rather than something worth a KV cache layer — flagged here rather than silently decided.

**(b) `emailService.ts`'s `sendEmail()` gains two small, additive checks at the top**, before anything else happens:
1. If `email_template_enabled[options.template]` is explicitly `false`, the send is skipped entirely — logged and recorded to `email_log` with a new `status: 'skipped'` (a genuine, honest state distinct from `sent`/`failed`, not overloading either), and the function returns without ever calling Resend. Every existing caller (newsletter, consultation, contact, purchase receipt, secure download, password reset, admin invite — 8 templates total) is affected by this check equally, automatically, with zero changes to any of those 8 call sites — this is the entire reason the toggle lives in the one shared function rather than being duplicated at each call site.
2. `callResend()`'s `from:` header and (new) `reply-to` header are built from `site_settings` (with the existing hardcoded values as the fallback if unset), instead of the literal string currently hardcoded.

**Both touch-points are opt-in-safe by default:** with `site_settings` empty (the state immediately after this migration runs), `maintenance_mode.enabled` defaults to `false` and every `email_template_enabled` entry defaults to `true` — meaning the deployed behavior is *identical* to today's until a Super Admin explicitly changes something. No existing flow's real behavior changes as a side effect of this phase shipping.

**One further scope decision on download defaults, stated explicitly:** `default_max_downloads`/`default_download_expires_days` are wired into exactly one place — `productService.createProduct()`, where they're substituted only when the admin leaves the *new* product's own fields blank. `fulfilmentService.ts` (the actual entitlement-granting code, which runs at real payment time) and `productCatalogService.ts` (what the live checkout/download flow reads) are **not touched at all** — an already-created product's stored `max_downloads`/`download_expires_days` values remain exactly what they are, and the proven commerce/fulfilment path is not modified in this phase. This was a deliberate choice over the alternative (resolving the default live, at fulfilment time, so changing the site-wide default retroactively affects existing NULL-configured products) — the create-time approach touches zero financially-sensitive code, at the cost of the default only applying to products created after this phase ships. Flagged here as the trade-off it is.

**Payment secrets remain untouched, per the non-negotiable Version 2.1 kickoff decision:** the Settings status endpoint reads `env.PAYSTACK_SECRET_KEY.length > 0` (a boolean presence check) and, if present, inspects only its fixed-length prefix (`sk_test_` vs `sk_live_`) to report **Test** or **Live** mode — the same pattern real payment dashboards use to show a test-mode banner without ever displaying a key. The actual secret value is never read into a variable that outlives that single comparison, never logged, never included in any API response, and never stored anywhere.

## 6. System Information — what's honestly available

- **Environment:** derived live from the request's hostname (`robayerwealthlab.com` → Production, anything else → Development/preview) — the same logic `routes/health.ts` already uses, reused rather than duplicated.
- **Current database migration:** read live from D1's own `d1_migrations` tracking table (`SELECT name FROM d1_migrations ORDER BY id DESC LIMIT 1`) — real, always-accurate, zero new config.
- **Application version:** `backend/package.json`'s `version` field (currently `0.1.0`, never bumped across any phase so far) — reported honestly as what it is, not a fabricated per-phase number.
- **Worker deployment version / build info / deployment timestamp:** **a real gap, addressed with a small deploy-process change, not a code architecture change.** Cloudflare Workers has no built-in way for a Worker to introspect its own deployed git commit or deploy time at runtime. This phase adds two new non-secret `wrangler.jsonc`-overridable vars — `DEPLOYED_COMMIT` and `DEPLOYED_AT` — set via `wrangler deploy --var DEPLOYED_COMMIT:$(git rev-parse --short HEAD) --var DEPLOYED_AT:<UTC timestamp>` at deploy time, documented as a new step added to this project's deploy checklist (matching `docs/deployment-checklist.md`'s own already-stated, never-yet-implemented intent to "tag deployments with a version identifier matching the git commit hash"). If unset (e.g. a deploy that didn't pass these flags), the status endpoint reports "Not available" honestly rather than a stale or fabricated value — the same "real data or a clear absence" discipline Analytics already established.

## 7. Security considerations

- **`super_admin`-only, reads and writes** — enforced via `requireRole(['super_admin'])` on every `/api/admin/settings*` route, identical shape to every existing role-gated route.
- **CSRF** on `PATCH /api/admin/settings` — the standard `requireCsrf()` call every mutating admin route already makes.
- **Server-side validation per key** — `maintenance_mode.message` length-capped and HTML-free (rendered as plain text in the 503 response, never as raw HTML — closes an obvious reflected-content path); `default_max_downloads`/`default_download_expires_days` validated as positive integers or `null`; `email_reply_to` validated as a syntactically real email address or `null`; `email_template_enabled` keys validated against the fixed 8-template enum (an unrecognized template name in the payload is rejected, not silently stored).
- **Audit logging** — every `PATCH` writes one `audit_logs` row per changed key (`site_settings.updated`, `metadata: { key, before, after, ip, userAgent }`), reusing the exact `auditService.record()` call and before/after metadata convention Phase 3 and Phase 4 already established.
- **No secret ever stored, displayed, or accepted as input** — `site_settings.value` is a JSON blob but the `PATCH` validation allowlists only the 6 keys in §2; there is no code path by which a request body could ever cause a Paystack/Resend credential to be written to D1, even a malicious one, since the allowlist rejects any unrecognized key outright.

## 8. Rollback strategy

- **Migration:** purely additive (one new table) — no destructive change, trivially reversible by a follow-up migration if ever needed, though (as with every prior migration in this project) forward-only is the standing convention.
- **`maintenance_mode` and `email_template_enabled` default to fully-open/fully-enabled** — if something goes wrong with this phase's own logic, the honest worst case is those two checks doing nothing (since the defaults are "don't block," "don't skip"), not an accidental site-wide outage caused by the feature itself.
- **Fastest mitigation if a bug is found in the maintenance-mode gate specifically** (the one check with real blast-radius, since it can 503 real traffic): a single `UPDATE site_settings SET value = '{"enabled":false}' WHERE key = 'maintenance_mode'` run directly against D1 immediately disables it, with no code deploy needed — the D1 row *is* the kill switch.
- **Worker/frontend rollback** otherwise follows the exact same path documented for every prior phase (`wrangler rollback` / prior Version ID; `git revert`).
