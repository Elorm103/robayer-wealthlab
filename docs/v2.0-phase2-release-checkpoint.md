## Production architecture as it exists today

```
GitHub Pages (static: everything except /api/* and /books/*)
        │
        ▼
Cloudflare Workers Route (robayerwealthlab.com/api/*, robayerwealthlab.com/books/*)
        │
        ▼
Worker: robayer-wealthlab-api
   ├── D1: robayer-wealthlab-db      (catalog, auth, commerce, media, all app data)
   ├── R2: robayer-wealthlab-storage (media/product files)
   └── KV: RATE_LIMIT_KV             (rate limiting)
```

- **Same-origin, no CORS**: `/api/*` and `/books/*` are intercepted by the Worker before reaching GitHub Pages; every other path (including `/admin/*`, which is static HTML + JS calling the same-origin `/api/*`) is served by GitHub Pages directly. No cross-origin requests exist anywhere in the app.
- **`/books/*` is fully self-contained** — the Worker renders every `/books/*` request itself (index, detail, 404) from D1, and never proxies out to GitHub Pages for a fallback. This was a deliberate design choice: a Workers `fetch()` subrequest to a URL matching the zone's own Route can get re-routed back through the same Worker instead of reaching the true origin, and this repo has no version-controlled DNS record to rule that out — so the safer design was chosen instead of one whose failure mode couldn't be tested.
- **D1 is the sole source of truth** for products (not a JSON mirror) — `content/products/*.json` is retired as a live data source, kept only as historical/pre-migration reference.
- **Media files** live in R2, referenced by `media_assets` rows; product cover/thumbnail/preview/OG images and downloadable files all reference Media Library asset IDs — never a duplicated upload.

## Current R2 bucket structure

Bucket: `robayer-wealthlab-storage`. Two coexisting key conventions (both intentional, not drift):

1. **Legacy digital-fulfilment convention** (pre-Media-Library): `ebooks/{slug}.pdf` — one object, the migrated ebook (`ebooks/starting-to-invest-with-gh100.pdf`), referenced by `product_files.media_id` → `media_assets.storage_key`.
2. **Media Library convention**: `media/{type}/{folder}/{uuid}.{ext}` — e.g. `media/documents/uncategorized/07e10df0-....pdf`, `media/images/books/....jpg`. All new uploads (Products or otherwise) use this convention going forward; the legacy key is a one-time exception preserved so existing `deliveries`/`download_tokens` rows keep resolving without a file move.

Public read access to a `media/*` object goes through `GET /api/media/file/:key` — which explicitly denies (identical 404) any storage key belonging to a `product_files` row on a product with `price_pesewas > 0`, forcing paid files through the entitlement → download-token flow instead. Free products' files remain publicly fetchable through this route as normal.

## Current admin capabilities

**Roles:** `super_admin`, `editor` (both can create/edit/publish/archive/duplicate/soft-delete/restore products and manage files/gallery/relations), `support` (read-only — list/view only, every mutation 403s, verified live).

**Product lifecycle:** `draft` → `active` (needs a price) → `archived`/`hidden`/`unavailable`/`coming-soon`, via either the per-product edit page's status control (fixed this phase) or the list page's bulk-action bar. No hard-delete exists anywhere — only soft delete/restore, by construction (matches "editors can create/edit but not permanently delete").

**Editor capabilities:** title/subtitle/descriptions (rich text, server + client sanitized), topic/product type (fixed taxonomies), pricing (price, compare-at price, currency, tax behavior), SKU, version, language, estimated reading time, author, cover/thumbnail/preview/OG images (via Media Library picker — never a duplicated upload), featured/bestseller/new-release flags, tags, download policy (max downloads, expiry days), full SEO fields (title, description, canonical URL), downloadable files (multi-file support), gallery images, related/cross-sell/recommended product links.

**List page:** search, filter by status/topic/product type/featured/soft-deleted, sort (newest/oldest/title/price), pagination, bulk actions (publish/unpublish/archive/unarchive/delete/restore).

## Known limitations and future work

1. **Bulk "Unarchive" UX gap** — only surfaces under the soft-delete list filter, not the "Archived" status filter. Per-product unarchive via the edit page works correctly; this is a missing convenience action, not a correctness gap. Recommended as a small standalone fix before or early in Phase 3.
2. **No Twitter-specific description field** — `/books/*` detail pages' `twitter:description` always mirrors the SEO description; the original hand-authored pages had distinct, shorter Twitter copy. No schema field exists for this; judged not worth adding for one cosmetic string.
3. **No gallery or related-product content yet** — both features are fully built (schema, API, admin UI) but unused (0 rows in both tables) since no admin has populated them yet. Not a defect — genuinely optional per-product data.
4. **Two coexisting R2 key conventions** (see R2 section above) — intentional, not a cleanup item; migrating the one legacy key would require also updating the `deliveries`/`download_tokens` rows that reference it, for no functional benefit.
5. **`content/products/*.json` is retired but still present in the repo** — kept as historical reference for exact original SEO copy, not deleted. Safe to remove in a future cleanup once nobody needs to diff against it.
6. **Checkout could not be fully exercised end-to-end through Paystack from this environment** (no network egress to `api.paystack.co` in the sandbox this work was done in) — verified up to "D1 product resolved correctly, Paystack session request sent" in both local and production; the payment→webhook→fulfilment path itself is unchanged by this phase and was verified in earlier commerce-foundation phases.

**Recommended next phase:** Version 2.0 Phase 3 per `docs/v2-development-roadmap.md` (the CMS/rich-content phase), now that this phase has proven the D1-as-source-of-truth + admin-CRUD + server-rendered-public-page pattern it will reuse.

## Rollback procedure

1. **Worker code:** `cd backend && npx wrangler rollback` (reverts to the previously deployed Version ID), or `git checkout <prior-commit> && npx wrangler deploy` (`111730d` is the last commit before Phase 2 began entirely).
2. **Database:** migrations in this project are forward-only by convention (no `.down.sql` files exist for 0001–0009). Rolling back 0008/0009's schema would require a hand-written reverse migration and should not be attempted without first confirming no post-migration writes (new products, purchases against them) would be lost — none of the current data existed in the old schema's shape.
3. **`/books/*` Route specifically:** removing the `robayerwealthlab.com/books/*` entry from `backend/wrangler.jsonc`'s `routes` array and redeploying falls back to GitHub Pages serving the untouched static `books/` files — a fast, code-only, database-independent rollback for just the public-facing surface.
