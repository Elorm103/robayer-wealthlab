## 1. Current state (baseline for this plan)

**7 of 12 admin modules are real and D1-backed**: Dashboard, Products, Media Library, Orders, Consultations, Contacts, Analytics. **5 remain empty shells with zero backend routes**: Resources, Blog, Newsletter (campaigns — the subscriber *list* already exists via Analytics/Dashboard, only campaign *sending* is missing), Settings, Users. This is exactly the gap Version 2.1's 7 phases close.

**Two proven architectural patterns already exist in production and should be reused, not reinvented:**
1. **D1-direct CMS pattern** (Products): D1 is the sole source of truth, full admin CRUD via `services/admin/*.ts` + `routes/admin/*.ts`, server-rendered public pages via a dedicated Worker Route (`routes/books.ts` → `/books/*`). No GitHub-commit publish pipeline exists or is needed — that mechanism was explicitly designed in the original V2.0 roadmap, never built, and superseded by this simpler pattern. **Resources and Blog CMS (Phases 1–2) should copy this pattern exactly.**
2. **Support-workflow pattern** (Consultation/Contact Manager): list/detail/drawer, all-roles-read, role-scoped writes, append-only audit trail. Not directly reused by any Version 2.1 phase, but its role-gating precedent (`support` gets full access to low-blast-radius actions, `editor`/`super_admin` gated on high-blast-radius ones) is the template Phase 4/5/6's role decisions below follow.

**One thing that does *not* yet exist and several Version 2.1 phases need:** a bulk/scheduled email-sending mechanism. Today, `emailService.sendEmail()` (`backend/emails/emailService.ts`) sends exactly one email to one recipient via Resend, synchronously, in the same request that triggers it (webhook fulfilment, contact acknowledgement, Orders' resend actions). There is no queue, no cron trigger, no batch API — `wrangler.jsonc` has zero `triggers`/`crons`/`queues` bindings today. **Phase 3 (Newsletter Campaigns) is the first feature in this project's history that needs to send to more than one recipient per action**, and Phase 4's scheduled/deferred needs (session expiry sweeps, lockout) are comparatively lighter. Section 3 below resolves this once, since it affects two phases.

**One static-file architecture constraint that directly shapes Phase 5:** `assets/config/site.json` is a **static file served by GitHub Pages**, read at runtime by `js/content-inject.js`. It is not served by the Worker and not backed by D1. An admin "Settings" page that edits it would need one of: (a) a GitHub-commit publish mechanism (the exact thing this project deliberately avoided building for Products/Blog), (b) migrating site-wide config into D1 with a new public read endpoint the frontend fetches instead of the static file (a real, non-trivial architecture change touching every page), or (c) scoping Settings to *operational* configuration that already lives in the Worker's own domain (D1/env vars) and explicitly excluding site-wide branding/copy edits from this phase. **Section 8 recommends (c)** and states why.

## 2. Cross-cutting architectural decisions

These apply across multiple phases below; stated once here rather than repeated per phase.

**Rich text and content sanitization.** `backend/utils/richTextSanitizer.ts` (HTMLRewriter-based, zero new dependency, already proven in production on Products' `description` field) is the *only* rich-text sanitizer this project should use. Resources (short descriptions, mostly plain text) and Blog (full article bodies, genuinely rich) both reuse it — no new sanitizer, no npm package, matching the explicit precedent stated in `docs/v2.0-phase3-architecture-plan.md`'s own security review.

**Soft delete, audit logging, role gating — one shape, reused everywhere.** Every new mutating table gets `deleted_at TEXT` (soft delete only, no hard delete anywhere in this codebase, by established convention), every mutation calls `auditService.record()`, and every route's role gate is `requireAuth()` first, `requireRole()` where the action is genuinely higher-stakes, `requireCsrf()` on every mutation, `isRateLimited()` on every route. No new middleware is introduced — Phases 1–6 all extend the existing composable `requireX()` call-at-the-top-of-the-handler convention, not a framework change.

**Bulk/deferred email sending (needed by Phase 3, touched by Phase 4).** No Cloudflare Queue is proposed. At this platform's real, current scale (10 newsletter subscribers as of the V2.0 checkpoint), a **synchronous, in-request batch loop** — iterate subscribers, call `emailService.sendEmail()` per recipient, record one `campaign_sends` row per attempt, all inside a single Worker invocation — comfortably fits inside Workers' CPU-time budget. This avoids adding genuinely new infrastructure (a Queue binding, a Durable Object, a cron trigger) for a problem this platform doesn't have yet, matching the same "a concrete, observable trigger, not a speculative one" discipline `docs/v2.0-phase3-architecture-plan.md` used for Analytics caching. **Stated concrete future trigger:** if the subscriber list grows into the hundreds/thousands, a synchronous loop risks the Workers wall-clock limit — at that point, move to a Cloudflare Queue-driven sender (one message per recipient, consumed by a separate Worker handler). Not built now; the `campaign_sends` table design below is queue-compatible (each row is already an independent unit of work) so this migration, if it's ever needed, requires no schema change.

**Password reset and account-security emails reuse `emailService.ts` exactly** — two new template files (`password-reset.html`, `admin-invite.html`), same `sendEmail()` call shape every other transactional email in this codebase already uses. No new provider, no new send path.

**Session/token generation reuses the existing pattern**, not a new one: `download_tokens`/`unsubscribe_tokens`/`admin_sessions` already establish "256-bit random token, `expires_at`, a nullable single-use `consumed`/`revoked_at` marker" — password-reset tokens and admin-invite tokens (Phases 4/6) are two more instances of this exact shape, not a new mechanism.

## 3. Phase 1 — Resources CMS

**Scope confirmation against real code:** the public `/resources/` page today is hand-authored static HTML with hard-coded `.resource-card` markup (title, format badge, category, description, download/detail link) — `content/resources/README.md` already documents the intended future shape ("one `resources.json` array, one object per resource") but that file was never built; this phase supersedes it with a real D1 table instead of a static JSON array, following the Products precedent (D1-direct beats a static JSON mirror for anything admin-editable).

**Database — new table, migration `0011_resources_module.sql`:**
```sql
CREATE TABLE resources (
  id                  INTEGER PRIMARY KEY AUTOINCREMENT,
  resource_id         TEXT NOT NULL UNIQUE,   -- stable external id, same convention as products.product_id
  slug                TEXT NOT NULL UNIQUE,
  title               TEXT NOT NULL,
  short_description   TEXT,
  description         TEXT,                    -- rich text, richTextSanitizer.ts on write

  category            TEXT NOT NULL CHECK (category IN ('budgeting', 'saving', 'debt', 'investing', 'planning')), -- matches existing data-category values in resources/index.html, confirmed against the live page
  format               TEXT NOT NULL CHECK (format IN ('template', 'checklist', 'tracker', 'worksheet', 'guide')), -- matches existing resource-card__format badges

  status              TEXT NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'published', 'archived')),
  tags                TEXT,                     -- comma-separated, same convention as products.tags

  file_media_id       INTEGER REFERENCES media_assets(id),  -- the actual downloadable file, via Media Library — never a duplicated upload
  cover_media_id      INTEGER REFERENCES media_assets(id),
  thumbnail_media_id  INTEGER REFERENCES media_assets(id),

  seo_title           TEXT,
  seo_description     TEXT,
  seo_canonical_url   TEXT,

  featured            INTEGER NOT NULL DEFAULT 0,
  download_count      INTEGER NOT NULL DEFAULT 0,   -- incremented server-side on real download, not client-trusted

  published_at        TEXT,
  created_by           INTEGER REFERENCES admin_users(id),
  updated_by           INTEGER REFERENCES admin_users(id),
  created_at           TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at           TEXT NOT NULL DEFAULT (datetime('now')),
  deleted_at           TEXT
);
CREATE INDEX idx_resources_status ON resources(status);
CREATE INDEX idx_resources_category ON resources(category);
CREATE INDEX idx_resources_deleted_at ON resources(deleted_at);
```
**Why no `resource_files`/`resource_gallery` join tables like Products has:** a resource is, by the platform's own existing content model, a single downloadable file with one cover image — never multiple files, never a gallery. Building the join-table generality Products needed (multi-file digital products, galleries) here would be premature complexity for a content type that has never needed it. If a future resource genuinely needs multiple files, add the join table then — a concrete trigger, not a speculative one, matching this project's established anti-over-engineering discipline.

**Public delivery — no entitlement/checkout needed.** Every resource is free (no `price_pesewas` field exists on this table at all — deliberately, since Resources has never been a commerce surface and Phase 1 doesn't make it one). Public download goes through the *existing* Media Library public route (`GET /api/media/file/:key`), exactly as free products already do — no new download-token mechanism, no new security surface. The public `/resources/` page becomes server-rendered (new `routes/resources.ts`, mirroring `routes/books.ts`'s `/resources/*` — wait, resources are top-level today (`/resources/`, not nested), so the new Workers Route pattern is `robayerwealthlab.com/resources/*`, added to `wrangler.jsonc`'s existing `routes` array alongside `/api/*` and `/books/*`).

**API (admin):** `GET /api/admin/resources/meta`, `GET /api/admin/resources` (list, filter by status/category/format, search), `GET /api/admin/resources/:id`, `POST /api/admin/resources`, `PATCH /api/admin/resources/:id`, `DELETE /api/admin/resources/:id` (soft delete), `POST /api/admin/resources/:id/restore`, `POST /api/admin/resources/:id/duplicate` — this is Products' exact endpoint shape, field-for-field renamed. `EDITOR_ROLES = ['super_admin', 'editor']` gates every write, `support` reads only — identical to Products' own role table, not Consultation/Contact Manager's all-roles-write pattern (Resources is content management, not a support-triage workflow).

**Admin UI:** `/admin/resources/`, `/admin/resources/new/`, `/admin/resources/edit/` — copy `admin/products/*` structurally (list with search/filter/pagination/bulk actions; editor with title/category/format/rich-text description/Media Library file+cover pickers/SEO fields/status lifecycle/preview link). Zero new CSS primitives needed — every component Products' editor uses already exists.

**Preview:** a `?preview=<short-lived-token>` query param on the public resource URL, allowing a draft to be viewed by an authenticated admin before publishing — the one genuinely new capability Products didn't build (Products never needed a preview state distinct from its own admin detail view, since a draft product simply 404s publicly and the admin views it in the editor). Token: reuse the existing 256-bit random-token generation pattern, single-purpose, admin-session-gated (checked via `requireAuth()` inside the public route handler when the query param is present — not a new auth mechanism).

**Risk: low.** Directly copies a pattern already proven correct through Products' own two-pass adversarial audit. The one new piece (preview tokens) is small and additive.

## 4. Phase 2 — Blog CMS

**Database — new tables, migration `0012_blog_cms.sql`:**
```sql
CREATE TABLE blog_posts (
  id                  INTEGER PRIMARY KEY AUTOINCREMENT,
  post_id             TEXT NOT NULL UNIQUE,
  slug                TEXT NOT NULL UNIQUE,
  title               TEXT NOT NULL,
  excerpt             TEXT,
  body                TEXT,                       -- rich text, richTextSanitizer.ts

  category            TEXT,
  tags                TEXT,

  status              TEXT NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'in_review', 'scheduled', 'published', 'archived')),
  scheduled_for       TEXT,                        -- nullable; when set with status='scheduled', the post publishes automatically once this time passes (mechanism below)

  cover_media_id      INTEGER REFERENCES media_assets(id),
  author_id           INTEGER REFERENCES admin_users(id),  -- real attribution, not a free-text name field — matches how consultation_notes.author_id already resolves to a real admin
  author_display_name TEXT,                        -- optional override (e.g. a guest-author byline) — falls back to admin_users.name if null

  seo_title           TEXT,
  seo_description     TEXT,
  seo_canonical_url   TEXT,

  published_at        TEXT,
  created_by           INTEGER REFERENCES admin_users(id),
  updated_by           INTEGER REFERENCES admin_users(id),
  created_at           TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at           TEXT NOT NULL DEFAULT (datetime('now')),
  deleted_at           TEXT
);
CREATE INDEX idx_blog_posts_status ON blog_posts(status);
CREATE INDEX idx_blog_posts_scheduled_for ON blog_posts(scheduled_for);
CREATE INDEX idx_blog_posts_deleted_at ON blog_posts(deleted_at);

CREATE TABLE blog_post_revisions (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  post_id      INTEGER NOT NULL REFERENCES blog_posts(id),
  title        TEXT NOT NULL,
  body         TEXT,
  author_id    INTEGER REFERENCES admin_users(id),
  created_at   TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_blog_post_revisions_post ON blog_post_revisions(post_id);
```
**Revisions, scoped honestly ("if practical," per the brief):** append-only snapshot on every save (matching `consultation_notes`' "never rewrite history" precedent) — a full copy of `title`+`body` at each save, not a diff engine. No restore-to-revision UI in Phase 2 (viewing history is the practical, low-risk slice; restoring is a real but separable feature — flagged as a natural Phase 2.1 follow-up, not blocking this phase). This keeps revisions genuinely practical (a `SELECT`-and-display feature) rather than building real version-control machinery this platform has never needed.

**Scheduled publishing — the one genuinely new mechanism in this phase.** No cron trigger exists in this project (Section 2). Two viable designs:
1. **Lazy evaluation** (recommended): the public blog route (`routes/blog.ts`) and the list query both treat a `status='scheduled' AND scheduled_for <= datetime('now')` row as published — no background job at all. A post "goes live" the moment someone (anyone — a visitor hitting the blog index, or the Worker's own next request) causes that `WHERE` clause to be evaluated. Simplest possible design, zero new infrastructure, and correct for a low-traffic blog where "live within one page load of the scheduled time" is indistinguishable from "live at the scheduled second."
2. **Real cron trigger** (`wrangler.jsonc`'s `triggers.crons`, a genuinely new binding): a scheduled Worker invocation flips `scheduled` rows to `published` on a fixed interval (e.g. every 5 minutes). More "correct" in the abstract, but adds real new infrastructure (a cron handler, a new code path with no HTTP request behind it, a new class of thing to test and monitor) for a benefit this platform's actual traffic pattern doesn't need.

**Recommendation: option 1 (lazy evaluation).** Matches this project's repeated "don't build infrastructure for a problem you don't have" discipline. If audit/precision (e.g. "the post must go live at exactly 9:00am for a scheduled announcement") becomes a real requirement later, option 2 is a contained, additive change — nothing in option 1's schema needs to change to add it.

**API (admin):** `GET /api/admin/blog/meta`, `GET /api/admin/blog`, `GET /api/admin/blog/:id`, `POST /api/admin/blog`, `PATCH /api/admin/blog/:id`, `DELETE /api/admin/blog/:id` (soft delete), `POST /api/admin/blog/:id/restore`, `GET /api/admin/blog/:id/revisions`. Same `EDITOR_ROLES` write gate as Products/Resources. Status transitions (`draft → in_review → scheduled/published → archived`) validated server-side against a real allowed-transition table (not every status can jump to every other status — e.g. `archived` cannot go directly to `scheduled`) — a genuinely new validation shape this project hasn't needed before (Products/Resources have simpler, less ordered status vocabularies).

**Public delivery:** `routes/blog.ts`, server-rendered, `/blog/*` added to `wrangler.jsonc`'s Workers Routes — identical mechanism to `/books/*`. The current 1 real article + 7 "coming soon" placeholder cards are migrated into this table as real rows (1 `published`, 7 `draft` or simply not created — recommend not creating placeholder rows for unwritten content, since a real CMS shouldn't need fake placeholder database rows to represent "content that doesn't exist yet"; the public blog index's empty/sparse state should be a genuine empty-state UI, matching this project's "no fake data, no fake numbers" discipline applied to content instead of numbers).

**Risk: low-moderate.** The CRUD/CMS shape is a proven pattern (Products/Resources). The two genuinely new pieces — status-transition validation and revisions — are each small, additive, and well-precedented by patterns elsewhere in this codebase (validation: Consultation Manager's status-value checking; revisions: `audit_logs`' append-only philosophy).

## 5. Phase 3 — Newsletter Campaigns

**Database — new tables, migration `0013_newsletter_campaigns.sql`:**
```sql
CREATE TABLE campaigns (
  id                  INTEGER PRIMARY KEY AUTOINCREMENT,
  campaign_id         TEXT NOT NULL UNIQUE,
  subject             TEXT NOT NULL,
  body_html           TEXT NOT NULL,               -- rich text, richTextSanitizer.ts — this is real outbound email HTML, sanitization matters even more here than on-site content
  template_key        TEXT,                        -- optional: which starting template this was built from (see below)

  status              TEXT NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'scheduled', 'sending', 'sent', 'cancelled')),
  scheduled_for       TEXT,

  audience_filter     TEXT,                        -- JSON: {"status":"subscribed"} today; a real segmentation predicate is a documented non-goal for Phase 3 v1 (see below)

  total_recipients    INTEGER,                      -- snapshotted at send time, not recomputed later
  sent_count          INTEGER NOT NULL DEFAULT 0,
  failed_count        INTEGER NOT NULL DEFAULT 0,

  created_by           INTEGER REFERENCES admin_users(id),
  created_at           TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at           TEXT NOT NULL DEFAULT (datetime('now')),
  sent_at              TEXT
);
CREATE INDEX idx_campaigns_status ON campaigns(status);

CREATE TABLE campaign_sends (
  id                  INTEGER PRIMARY KEY AUTOINCREMENT,
  campaign_id         INTEGER NOT NULL REFERENCES campaigns(id),
  subscriber_id       INTEGER NOT NULL REFERENCES newsletter_subscribers(id),
  status              TEXT NOT NULL CHECK (status IN ('pending', 'sent', 'failed')),
  provider_id         TEXT,                          -- Resend's own message id, same convention as email_log.provider_id
  error               TEXT,
  opened_at           TEXT,                          -- nullable; populated only if open tracking is wired (see below)
  clicked_at          TEXT,
  created_at           TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_campaign_sends_campaign ON campaign_sends(campaign_id);
CREATE UNIQUE INDEX idx_campaign_sends_unique ON campaign_sends(campaign_id, subscriber_id); -- one send row per subscriber per campaign, hard constraint against an accidental double-send
```

**Audience selection/segmentation — scoped honestly.** The brief asks for "audience selection / segmentation." The real, honest scope for a v1: **status = `subscribed`, always** (the platform has no other subscriber attribute today — no tags, no signup-source-based lists beyond the existing `source` column, no engagement scoring). `audience_filter` is stored as JSON specifically so a future segmentation dimension (e.g. "subscribed via free-guide only") is additive — no schema change needed later, just a new filter key the query builder understands. **Building real multi-dimensional segmentation now would be speculative** — `newsletter_subscribers` has exactly one meaningful boolean today (subscribed/unsubscribed). Recommend v1 ships "send to all subscribed" only, with the schema shaped to grow into more later.

**Rich email editor + templates.** No new WYSIWYG library is proposed — reuse the same contenteditable + toolbar pattern `richTextSanitizer.ts` already sanitizes for Products/Resources/Blog bodies, with an email-specific CSS reset applied only at render/send time (email clients need inline styles, not linked stylesheets — the existing `emails/templates/*.html` already hand-author this correctly; the campaign composer's saved `body_html` gets wrapped in the *same* base email template shell those files already establish, not a new one). "Templates" = a small, fixed set of starting-point HTML snippets (e.g. "Announcement," "Newsletter digest," "Simple update") stored as static files under `backend/emails/campaign-templates/`, selected once when creating a campaign, then freely edited — not a full drag-and-drop template builder, which is a materially larger feature than this platform's real need justifies today.

**Sending mechanism:** per Section 2/3's cross-cutting decision — a synchronous batch loop inside `POST /api/admin/campaigns/:id/send`. For each `subscribed` row in `newsletter_subscribers` not already in `campaign_sends` for this campaign: call `emailService.sendEmail()`, write the resulting `campaign_sends` row (`sent`/`failed` + `provider_id`/`error`), increment the campaign's counters. Wrapped in the existing per-recipient try/catch pattern `fulfilmentService.ts` already uses for its two emails ("one recipient's failure never blocks the rest") — here it matters even more, since 1 of N failing must never abort N-1 real sends.

**Test emails:** `POST /api/admin/campaigns/:id/test` — sends the exact same `body_html` to a single admin-supplied address (defaulting to the acting admin's own email), writes **no** `campaign_sends` row (a test send is explicitly not counted as a real delivery) and is **not** rate-limited by the campaign-send-once constraint (`campaign_sends`' unique index only applies to real sends).

**Delivery history + open/click tracking, scoped honestly.** Delivery history = the `campaign_sends` table itself, surfaced as a per-campaign detail view (sent/failed counts, a searchable per-recipient status list) — real, buildable today, zero new infrastructure. **Open/click tracking is the one item in this phase genuinely gated on a third-party capability this project hasn't verified**: Resend supports open/click webhooks in some plan tiers, delivered as async webhook events the Worker would need a new `POST /api/webhooks/resend` handler to receive (mirroring the existing Paystack webhook's HMAC-verification pattern). **This is a real external dependency to confirm before Phase 3 implementation begins** (does the current Resend account/plan expose these webhooks?) — the schema above (`opened_at`/`clicked_at` on `campaign_sends`) is ready for it either way; if the webhook isn't available, those two columns simply stay `NULL` forever and the admin UI's "Open rate" stat degrades to "Not available" (the same honest "real data or a clear absence, never a fake number" pattern Analytics already established for Visitors/Sessions).

**API (admin):** `GET /api/admin/campaigns`, `GET /api/admin/campaigns/:id`, `POST /api/admin/campaigns`, `PATCH /api/admin/campaigns/:id` (draft edits only — a `sent` campaign is immutable, matching `audit_logs`' "don't rewrite history" precedent applied to a sent email), `POST /api/admin/campaigns/:id/send`, `POST /api/admin/campaigns/:id/test`, `POST /api/admin/campaigns/:id/cancel` (only valid pre-send), `GET /api/admin/campaigns/:id/deliveries`. **`super_admin`/`editor` only for every write, including test sends** — this is this project's single highest-blast-radius action to date (a bad send reaches every real subscriber at once, exactly as `docs/v2-development-roadmap.md`'s original Phase 5 reasoning already flagged), `support` gets read-only.

**Risk: moderate-high — the highest of any Version 2.1 phase.** Not a technical-complexity risk (the mechanism is a straightforward loop over an existing table using an existing send function) but a **blast-radius** risk: a real send action, to real subscriber inboxes, that cannot be undone once triggered. Mitigations: mandatory test-send-first UX nudge (not a hard block, but the compose UI should make "Send test" the visually primary action and "Send to all subscribers" a distinctly separate, confirmed, consequence-stated action — matching the Orders drawer's existing "state the real consequence in the confirm modal" pattern); the `campaign_sends` unique constraint makes a double-send of the same campaign to the same subscriber structurally impossible even under a retried/duplicate request; `editor`/`super_admin`-only gating; full audit logging (`campaign.sent`, `campaign.test_sent`, actor-attributed).

## 6. Phase 4 — Identity & Security

**Database — migration `0014_identity_security.sql`:**
```sql
ALTER TABLE admin_users ADD COLUMN must_change_password INTEGER NOT NULL DEFAULT 0;
ALTER TABLE admin_users ADD COLUMN failed_login_attempts INTEGER NOT NULL DEFAULT 0;
ALTER TABLE admin_users ADD COLUMN locked_until TEXT;
ALTER TABLE admin_users ADD COLUMN password_updated_at TEXT;

CREATE TABLE password_reset_tokens (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  token        TEXT NOT NULL UNIQUE,               -- 256-bit, same generation pattern as download_tokens
  admin_id     INTEGER NOT NULL REFERENCES admin_users(id),
  expires_at   TEXT NOT NULL,                        -- short-lived: 30 minutes, tighter than a 12h session — this is a bearer-style credential to an account, not a file download
  used_at      TEXT,
  created_at   TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_password_reset_tokens_admin ON password_reset_tokens(admin_id);

CREATE TABLE login_history (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  admin_id     INTEGER NOT NULL REFERENCES admin_users(id),
  outcome      TEXT NOT NULL CHECK (outcome IN ('success', 'failed_password', 'failed_locked', 'failed_inactive')),
  ip_address   TEXT,
  user_agent   TEXT,
  created_at   TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_login_history_admin ON login_history(admin_id);
```
**Why a dedicated `login_history` table rather than reading `audit_logs`:** `audit_logs` already records `admin.login`/`admin.unauthorized_access` events, but it's a shared, general-purpose table across every admin action in the system (order resends, note additions, everything) — a "Login History" UI reading it directly would need to filter a large, unrelated event stream on every page load. A dedicated table, written alongside (not instead of) the existing `audit_logs` write, is a small amount of duplication in exchange for a genuinely simpler, faster, purpose-built query — the same reasoning `docs/v2.0-phase3-architecture-plan.md` already used to justify `consultation_notes`/`contact_notes` as dedicated tables instead of a polymorphic one.

**Authentication features:**
- **Change password** — `POST /api/admin/auth/change-password`, requires current password re-entered (not just an active session — a stolen/left-open session shouldn't be enough to change the password), new password validated against the strength policy below, `password_updated_at` set, **all other active sessions for that admin are revoked** (a password change should invalidate every session except the one that just made the change — a real security property, not just a nice-to-have).
- **Force password change on first login** — `must_change_password` flag, set `true` by default when an admin account is created (Phase 6) or has its password reset by another admin; checked by `requireAuth()` (or a thin wrapper around it) on every request — a `true` flag redirects every admin page except the change-password form itself to that form. This is a genuinely new gate in the auth flow, the one piece of this phase that touches `requireAuth()`'s own logic rather than adding a new route.
- **Forgot password / secure reset** — `POST /api/admin/auth/forgot-password` (accepts an email, always returns the identical generic success response whether or not the email exists — the same no-user-enumeration discipline this project's login endpoint already established — sends a reset email only if it does exist), `POST /api/admin/auth/reset-password` (token + new password; the token is single-use, `used_at` set atomically with the password update in one transaction to prevent a race where the same token resets the password twice). New email template `password-reset.html`, following the exact structure of the existing 6 templates.
- **Password strength validation** — server-side, not just client-side cosmetic hints: minimum length (12 characters, matching the real strength of this project's own existing PBKDF2-100k-iteration hashing being worth defending with a real minimum), rejects the exact current password on change, rejects a small deny-list of the most common weak patterns (`password`, `123456`-shaped, the admin's own email local-part) — deliberately not a complex "must contain 1 uppercase, 1 number, 1 symbol" rule (well-established security guidance, e.g. NIST 800-63B, now recommends length over composition rules; matches this project's own preference for real, tested measures over box-ticking ones).

**Sessions features:**
- **View active sessions** — `GET /api/admin/auth/sessions`, lists the calling admin's own non-revoked, non-expired `admin_sessions` rows (never another admin's, even for `super_admin` — a session list is inherently personal, unlike Orders/Consultations which are legitimately shared operational data).
- **Sign out other sessions** — `POST /api/admin/auth/sessions/:id/revoke`, sets `revoked_at`; the endpoint verifies the target session's `admin_id` matches the caller's own id before revoking (an IDOR check — this is exactly the shape of bug a fresh adversarial pass must probe for).
- **Last login / login history** — `last_login_at` (already exists on `admin_users`, currently only set on login — unchanged) surfaced in the Users module (Phase 6) and a `GET /api/admin/auth/login-history` endpoint reading the new `login_history` table, own-history only (same reasoning as sessions).

**Profile features:**
- **Update profile** (`name`) — trivial, `PATCH /api/admin/auth/profile`.
- **Change email** — genuinely more sensitive than a name change (email is the login identifier). Requires current password re-entry (same reasoning as change-password) and, recommended, a confirmation email to the *new* address before the change takes effect (a `pending_email`/`email_change_token` pair, reusing the password-reset-token shape) — prevents a compromised session from silently redirecting an account's recovery email to an attacker-controlled address. **Flagged as the one sub-feature in this phase worth confirming scope on before build** — a same-request, no-confirmation email change is simpler to build but meaningfully weaker; recommend the confirmation-email version.
- **Avatar** — explicitly optional per the brief. If built: reuses Media Library exactly (an avatar is just another `media_assets` row, `admin_users.avatar_media_id` a new nullable FK) — zero new upload/storage mechanism. Recommend deferring to a fast-follow rather than blocking Phase 4 on it, since it's cosmetic and every other sub-feature in this phase is a real security control.
- **Notification preferences** — the honest scope check: **what notifications does this platform currently send an admin, at all?** Answer: none — there is no existing admin-facing notification system (email-to-admin-on-new-consultation, etc. don't exist today). Building a preferences UI for notification types that don't exist yet would be speculative. Recommend scoping this sub-feature down to a single real toggle that has genuine meaning today ("email me a login-history digest" or similar) or deferring it entirely until a real notification-sending feature exists to have preferences about — flagged as a scope question for confirmation, not silently built as a no-op UI.

**Security features:**
- **Session timeout** — already exists (12h absolute, `admin_sessions.expires_at`) — this phase's real addition is **exposing it in the UI** (a visible "session expires in X" or a graceful re-auth prompt rather than a silent 401) and confirming the existing value is still the right one (unchanged from Phase 0.1, no evidence it needs to change).
- **Account lockout** — `failed_login_attempts` increments on `INVALID_CREDENTIALS`, resets to 0 on success; at a threshold (recommend 5), set `locked_until` (recommend 15 minutes, not permanent — a permanent lockout requiring another admin's manual intervention is a real support burden for a 1–3-person team, and a time-boxed lockout already defeats a brute-force attempt at this project's realistic threat model). `login_history` records `failed_locked` outcomes distinctly from `failed_password` for real visibility into whether lockout is actually triggering.
- **Password policy** — the strength rules above, applied consistently at signup (Phase 6, admin creation), change-password, and reset-password — one shared `validatePasswordStrength()` utility, not three separately-maintained copies of the same rule.
- **Rate-limit review** — an actual audit task, not a new feature: re-verify `isRateLimited()`'s existing login/forgot-password/reset-password limits are tuned correctly now that these flows carry real account-takeover risk if under-limited (forgot-password especially — an unlimited forgot-password endpoint is itself a spam/harassment vector against a real admin's inbox even without any credential risk).
- **Security review of authentication flows** — this phase's own Phase 7-style audit, scoped just to auth: re-read every auth-adjacent file fresh, adversarially test lockout (does 5 real failures actually lock, does the 6th attempt correctly reject even with the right password?), token reuse (does a used password-reset token correctly fail on a second use?), session-revocation-on-password-change (does an old session cookie genuinely stop working immediately?), IDOR on sessions/login-history endpoints.

**Risk: high — the second-highest of any Version 2.1 phase, alongside Newsletter Campaigns.** This phase modifies `requireAuth()` itself (the `must_change_password` gate) and the core login flow (lockout) — the one subsystem every other module in this entire platform depends on being correct. **Recommend building this phase with the same "prove the smallest slice first" discipline Consultation Manager established for Phase 3**: change-password and forgot/reset-password first (additive, don't touch the existing login path's logic), lockout and `must_change_password` second (these genuinely modify `adminAuthService.ts`'s existing login function and `requireAuth()`), sessions/profile/login-history last (lowest-risk, purely additive read/own-scoped-write endpoints).

## 7. Phase 5 — Site Settings

**Scope decision, stated plainly up front:** per Section 1's architecture constraint, this phase **does not** migrate `assets/config/site.json` into an admin-editable surface — doing so would require either resurrecting the GitHub-publish pipeline this project deliberately chose not to build twice now (Products, then implicitly again here), or a materially larger architecture change (moving every page's config-read to a new public API call, touching `content-inject.js` and every page that uses it). **Recommended scope: a new `site_settings` D1 table for configuration that is genuinely operational and already Worker-side, not for the static site's branding/copy/SEO defaults.**

**Database — migration `0015_site_settings.sql`:**
```sql
CREATE TABLE site_settings (
  key          TEXT PRIMARY KEY,
  value        TEXT NOT NULL,                       -- JSON-encoded value; typed per-key at the service-layer, not the DB layer, matching how site.json itself is untyped JSON
  updated_by   INTEGER REFERENCES admin_users(id),
  updated_at   TEXT NOT NULL DEFAULT (datetime('now'))
);
```
A single key-value table, not one column per setting — new operational settings (e.g. a future feature flag) become a new row, not a new migration, matching the flexibility `audience_filter`'s JSON-column design in Phase 3 already established for a similar reason.

**In scope (real, Worker-side, currently either hard-coded or env-var-only):**
- **Maintenance mode** — a `site_settings` key (`maintenance_mode: {enabled: bool, message: string}`) checked at the top of the Worker's request handler (or via a small new middleware, `checkMaintenanceMode()`) — when enabled, every public route (not `/admin/*`, which must stay reachable to turn it back off) returns a 503 with the message. This is the one setting in this phase with genuine, immediate operational value and no static-file dependency at all.
- **Email settings** — which of the 6 existing templates are active/inactive (a kill-switch per template, e.g. disable `newsletter-welcome` temporarily without a code deploy), the reply-to address used across `emailService.ts` (currently likely hard-coded — confirm at build time).
- **Download settings** — `max_downloads`/`download_expires_days` *defaults* (Products already has these as per-product fields; a site-wide default that a new product inherits unless overridden is a real, useful setting).
- **Payment configuration — read-only status display, not credential editing.** `PAYSTACK_BASE_URL`/`PAYMENT_PROVIDER` are already `wrangler.jsonc` environment variables; `PAYSTACK_SECRET_KEY`/`PAYSTACK_WEBHOOK_SECRET` are Worker secrets (`wrangler secret put`, never in any D1 table, never rendered in any admin UI, ever — this is a hard, non-negotiable boundary, not a scope choice). **This phase's "payment configuration" UI is a status page** (which provider is active, is the webhook secret configured — a boolean presence check, never displaying the secret's value — last successful webhook received timestamp) **not a credential-entry form.** Real credential rotation stays a `wrangler secret put` operation performed by whoever has Cloudflare account access, exactly as it is today — moving payment secrets into an admin-UI-editable D1 table would be a genuine security regression (a database row is a meaningfully weaker protection boundary than a Cloudflare Worker secret), and is explicitly rejected here, not merely deferred.
- **Backup/export "where appropriate"** — scoped to what's genuinely appropriate: a `GET /api/admin/settings/export` that produces a JSON snapshot of `site_settings` (this table only, not a full database dump — a full D1 export is an infrastructure/ops task performed via `wrangler d1 export`, already documented in `docs/backup-and-recovery.md`, not a feature this admin UI should reimplement).

**Explicitly out of scope for Phase 5 (stated as non-goals, not oversights):**
- Company info, branding, logo/favicon, contact info, social media links, SEO defaults — all of these live in `assets/config/site.json` today and stay there. Editing them remains a direct file edit + `git push`, exactly as every prior version of this platform has done it. **This is a real, honest scope reduction from the brief's literal list** — flagged clearly here rather than silently narrowed, since it's the one Phase 5 item most likely to need a follow-up conversation about whether the larger site.json-to-D1 migration is worth doing in a future version.

**API:** `GET /api/admin/settings`, `PATCH /api/admin/settings` (body: partial key-value updates), `GET /api/admin/settings/status` (the payment/email operational status view), `GET /api/admin/settings/export`. `super_admin` only for writes (this is the highest-blast-radius *configuration* surface in the admin, even in its scoped-down form — maintenance mode alone can take the entire public site offline) — `editor`/`support` read-only.

**Risk: moderate, contained by the scope reduction above.** The rejected-outright payment-credential-editing idea would have been the single highest-risk sub-feature in all of Version 2.1 if built as literally specified; scoping it to a read-only status view removes that risk entirely rather than mitigating it.

## 8. Phase 6 — User Management

**Database — migration `0016_user_management.sql`:**
```sql
CREATE TABLE admin_invites (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  token        TEXT NOT NULL UNIQUE,
  email        TEXT NOT NULL,
  role         TEXT NOT NULL CHECK (role IN ('super_admin', 'editor', 'support')),
  invited_by   INTEGER REFERENCES admin_users(id),
  expires_at   TEXT NOT NULL,                        -- 7 days, longer than a password-reset token since this is a lower-frequency, planned action
  accepted_at  TEXT,
  created_at   TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_admin_invites_email ON admin_invites(email);
```
No schema change needed to `admin_users` beyond what Phase 4 already added (`must_change_password` is exactly the flag a newly-created/invited admin needs set).

**Create administrators (direct) vs. invite administrators — both, serving different real needs.** Direct creation (`POST /api/admin/users`, `super_admin` only): sets a temporary password, `must_change_password=true`, the new admin can log in immediately — useful when the inviter needs to hand over a credential in person/via a secure channel today. Invite (`POST /api/admin/users/invite`): sends an email (new template `admin-invite.html`) with a signed link to `/admin/accept-invite/?token=...`, where the invitee sets their own first password — no temporary password ever transmitted over email at all, meaningfully more secure, and the recommended default path. Both converge on the same `admin_users` row shape.

**Edit users / disable / reactivate:** `PATCH /api/admin/users/:id` (name, role — `super_admin` only, and a `super_admin` cannot demote or disable *themselves* if they're the only remaining active `super_admin`, a real guardrail against an admin accidentally locking the platform out of its own highest-privilege role), `is_active` toggle (soft-disable, not delete — `admin_users` already has `deleted_at` for a genuine account removal, kept separate from `is_active` since "temporarily disabled" and "permanently removed" are different real states with different real reasons, matching how Products distinguishes `archived` from soft-deleted).

**Assign roles:** part of the same `PATCH /api/admin/users/:id`, `super_admin`-only, validated against the existing 3-value CHECK constraint (no new role is proposed — `super_admin`/`editor`/`support` already cover every real permission boundary this platform has established across Products/Orders/Consultation-Contact/Analytics/Campaigns/Settings).

**Force password reset / force logout:** `POST /api/admin/users/:id/force-password-reset` (sets `must_change_password=true` and revokes all that admin's active sessions — the same "a credential-affecting action ends existing sessions" principle Phase 4's own change-password endpoint establishes, applied here by another admin instead of by the account holder themself), `POST /api/admin/users/:id/force-logout` (revokes sessions only, no password impact — a lighter action for "I think this person's laptop was stolen" versus "I think this person's password is compromised").

**View last login / security status:** surfaces `admin_users.last_login_at`, `failed_login_attempts`, `locked_until` (all added in Phase 4) in the Users list/detail — this phase is purely a *view* onto Phase 4's data for every admin other than yourself (Phase 4's own session/login-history endpoints are strictly own-account-only, by design — Phase 6's Users module is where a `super_admin` gets the *other-admin* visibility Phase 4 deliberately withheld).

**Audit history:** a per-admin filtered view of the existing `audit_logs` table (`WHERE actor_id = ?`) — no new table, reusing the exact mechanism every other module's audit trail already writes to.

**Risk: low.** Every sub-feature here is either a straightforward CRUD extension of the already-existing `admin_users` table or a read-only view onto data Phase 4 already created. The one real guardrail worth calling out explicitly (Section above: cannot self-demote/disable the last `super_admin`) is a small, well-defined server-side check, not a structural risk. **Correctly sequenced last**, per the original roadmap's own reasoning (restated in the V2.0 Final Release Report): zero real urgency with one real admin account in production today, and every sub-feature here depends on Phase 4's schema additions existing first.

## 9. Phase 7 — Final Acceptance Audit

Structurally identical to Version 2.0's own Stage 5 (Section 7 of `docs/v2.0-phase3-implementation.md`), scaled to 6 phases instead of 4: fresh re-read of every file across Phases 1–6, cross-module verification (all 12 admin modules — the 7 already live plus the 5 shipped this version — exercised together in a real browser), the full adversarial battery (auth, authz, CSRF, XSS, SQLi, IDOR, cache, **plus this version's two new categories: account-lockout/session-revocation correctness from Phase 4, and mass-email-send correctness/idempotency from Phase 3**), confirmation of zero leftover test data/debug code/temp accounts in production, CHANGELOG + a `docs/v2.1-release-checkpoint.md` (matching the established format), git tag `v2.1-content-administration-complete` (or the exact name given at release time) — only if no blocking defects remain, per the same discipline used at the close of every prior phase.

**One addition specific to this version, not present in Version 2.0's own Stage 5:** because Phase 3 (Campaigns) and Phase 4 (lockout/forced-password-change) both have real, hard-to-reverse production consequences (an actual email to actual subscribers; an actual admin locked out of their own account), Phase 7's production verification for those two phases specifically should be planned with extra care — a campaign test-send only (never a real send-to-all as part of verification), and lockout/forced-password-change tested exclusively against disposable test admin accounts, never the one real production admin.

## 10. Cross-phase sequencing and dependency graph

```
Phase 1 (Resources)  ──┐
                        ├── independent, can build in either order, share the D1-CMS pattern
Phase 2 (Blog)       ──┘

Phase 3 (Campaigns)  ── depends on nothing new architecturally, but is highest-blast-radius —
                         recommend building after 1–2 so the team is "warmed up," matching
                         Phase 3 (V2.0)'s own "prove lower-stakes patterns first" sequencing.

Phase 4 (Identity)   ── depends on nothing above; touches requireAuth()/login itself —
                         recommend building in parallel-eligible position but verified in
                         total isolation from 1–3/5–6, since a login-flow regression would
                         be the single most severe possible defect in this entire version.

Phase 5 (Settings)   ── depends on nothing above technically, but its Users-adjacent parts
                         (viewing which admin changed a setting) benefit from Phase 4/6
                         existing first for full attribution UX. Not a hard dependency.

Phase 6 (Users)      ── depends on Phase 4's schema (must_change_password, lockout fields,
                         login_history) — must follow Phase 4.

Phase 7 (Audit)      ── depends on all of 1–6.
```

**Recommended build order:** **1 (Resources) → 2 (Blog) → 4 (Identity & Security) → 6 (User Management) → 5 (Settings) → 3 (Campaigns) → 7 (Audit).**

**Why Campaigns moved to second-to-last, not third as the brief's own numbering implies:** it is this version's single highest-blast-radius action (Section 5) — sequencing it after Identity & Security and User Management means account lockout, forced-logout, and role management (real safety nets if a campaign-sending credential were ever compromised) already exist and are tested *before* the mass-send capability goes live, not after. This is the same reasoning Version 2.0's own plan used to sequence Orders' resend actions after the lower-stakes Consultation/Contact modules — apply it here at the phase level instead of the stage level.

**Why Identity & Security moved up, ahead of Settings/Users/Campaigns:** it touches the one piece of infrastructure every other phase (and every existing module) depends on being correct — the login/session system. Building and hardening it early means every subsequent phase's own verification work benefits from the stronger guarantees (lockout, forced-password-change, session revocation) rather than retrofitting them after Campaigns/Settings already shipped.

Each phase should follow the same per-stage discipline this engagement has used consistently since Version 2.0: fresh audit before implementation → implement → verify locally in a real browser → adversarial self-audit → fix genuine defects → deploy → production verification with disposable test data, cleaned up → documentation update → move to the next phase only after explicit approval.
