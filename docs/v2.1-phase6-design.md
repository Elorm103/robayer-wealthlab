# Version 2.1 Phase 6 — Newsletter Campaigns: Design Document

**Status:** Design only. No implementation code has been written. Per the user's own instruction, implementation begins only after this document is reviewed and approved.

---

## 1. Architecture review (current state)

Read fresh for this phase: `backend/services/newsletterService.ts`, `backend/services/emailService.ts`, `backend/services/unsubscribeService.ts`, `backend/utils/richTextSanitizer.ts`, `backend/database/schema.sql`'s `newsletter_subscribers`/`unsubscribe_tokens`/`email_log` sections, `backend/wrangler.jsonc`, `backend/services/admin/auditService.ts`, `backend/routes/admin/blog.ts`, `js/components/admin/admin-blog-editor.js`, `admin/newsletter/index.html` (confirmed: still the original empty "Coming soon" shell — "Manage subscribers and send campaigns. This module arrives in a later phase.").

**Subscribers.** `newsletter_subscribers` (`id`, `email` UNIQUE, `status` CHECK IN `('subscribed', 'unsubscribed')`, `source`, `subscribed_at`, `unsubscribed_at`) already exists and is the correct, only audience source — no new subscriber storage is needed. Production today has **9 subscribed, 1 unsubscribed** — a real number worth stating plainly, since it shapes the sending-architecture trade-off in §4.

**Unsubscribe.** `unsubscribe_tokens` + `unsubscribeService.getOrCreateUnsubscribeToken()` already provide a single-use-safe, per-subscriber token (the same proven shape as `download_tokens`/`password_reset_tokens`). Every newsletter-family email already carries a real unsubscribe link and `List-Unsubscribe`/`List-Unsubscribe-Post` headers (`docs/newsletter-unsubscribe-design.md`) — campaigns must reuse this exactly, not build a second unsubscribe mechanism.

**Sending pipeline.** `emailService.ts`'s `sendEmail()` is the single, already-established send path every existing template (newsletter-welcome, free-guide-delivery, consultation/contact acknowledgements, purchase-receipt, secure-download, password-reset, admin-invite) goes through. Per call, it: checks Phase 5's per-template kill switch first; renders one of eight **compile-time-bundled** template files (`{{placeholder}}` substitution into `emails/layouts/base.html`, every placeholder value HTML-escaped); calls Resend once, retries once on a transient failure; and always records the outcome to `email_log` (`sent`/`failed`/`permanently_failed`/`skipped`) via the shared `entity_type`/`entity_id` pattern — regardless of success, so a send is never silently lost. This is the one pipeline Phase 6 must extend, not duplicate.

**A real, load-bearing constraint: no queue, cron, or durable-object infrastructure exists anywhere in this project.** `wrangler.jsonc` has no `queues`, `triggers`, or `durable_objects` binding at all. `emailService.ts`'s own header comment states this plainly: *"The scheduled Cron Trigger that later retries rows still `failed` in `email_log` is NOT implemented... building the consumer is future work."* Every existing send happens synchronously, one recipient at a time, inside the HTTP request that triggered it (a newsletter signup, a purchase, an admin invite) — because each of those triggers has exactly one recipient. **A campaign has, by definition, many.** This is the one genuinely new architectural problem this phase introduces, and it's addressed head-on in §4 rather than assumed away.

**Content editing.** Blog posts, product descriptions, and resource descriptions already share one established rich-text pattern: a lightweight `contenteditable` editor client-side (`admin-blog-editor.js`'s `sanitizeRichText()`, a defensive allowlist mirror) plus a mandatory, independent server-side pass through `utils/richTextSanitizer.ts`'s `sanitizeRichTextHtml()` (an `HTMLRewriter`-based allowlist — no client input is ever trusted, per this codebase's stated security posture). A campaign body is exactly this same kind of admin-authored HTML content and should reuse this pattern identically — no new editor, no new sanitizer.

**One genuine gap, not a flaw:** `emailService.ts`'s `substitute()` always HTML-escapes every placeholder value (correct for every existing template, which only ever substitutes plain strings like a name or a URL). A campaign body is different — it is *already-sanitized, trusted HTML* (via `sanitizeRichTextHtml()`) that must be inserted **as HTML**, not escaped into visible tag text. This needs one small, narrowly-scoped extension to `emailService.ts` (§3), not a parallel rendering path.

**Admin UI placeholder already exists.** `/admin/newsletter/` is the pre-built empty shell, and the sidebar already links to it — this phase rebuilds that one page, matching the exact "rebuild from empty shell" pattern every prior Phase 5/4/Blog/Products module followed.

**Scope note:** the request lists Campaign Management (draft, test, send-to-all, history, delivery summary) — it does not list subscriber-list management (search, manual add/remove, CSV export/import). This design treats subscriber management as out of scope, matching the explicit bullet list; the only subscriber-facing capability this phase needs is a single `COUNT(*) WHERE status = 'subscribed'` for the pre-send confirmation, which requires no new UI of its own.

## 2. Campaign lifecycle

Following the recommended shape directly:

```
Draft ──(send)──▶ Sending ──(all recipients reach a terminal state)──▶ Sent
  │                  │
  │                  └──(a genuine top-level failure before any
  │                      recipient could be attempted)──▶ Failed
  │
  └──(deleted while still a draft)──▶ (row removed; no lifecycle state needed)
```

- **Draft** — freely editable (subject, body, test-sendable) and freely deletable. Never contributes to `email_log` recipient counts. The *only* status a real send can start from.
- **Sending** — the transient, in-progress state. A recipient roster is snapshotted at the moment of the Draft→Sending transition (§3) and is then fixed for the life of the campaign — later unsubscribes do not retroactively shrink an in-progress or completed campaign's roster; they simply mean that recipient's send attempt is best-effort like any other (see §5).
- **Sent** — terminal, reached once every snapshotted recipient has reached a terminal per-recipient outcome (`sent`, `failed`, or `skipped` — see §5). **`Sent` does not mean 100% delivered** — it means the send process ran to completion. The true outcome is the delivery summary (§7), read live from the per-recipient roster, not implied by the campaign's own status. This mirrors how this codebase already treats a single transactional email's failure (e.g. a receipt email failing never fails the purchase it belongs to) — extended here to "some recipients out of many" instead of "one recipient."
- **Failed** — reserved for the rare case where the campaign could not even begin attempting recipients (e.g. the Draft→Sending transition's own recipient-snapshot write failed). Not used for "some recipients bounced" — that is a normal `Sent` outcome with a non-perfect delivery summary.
- **Cancelled** — see the open question in §8. In the common case ("Sending is cancelled before it begins," the exact scenario named in scope), nothing has happened server-side yet, so there is nothing to cancel — the admin simply never clicks the confirmed-send button, and the campaign remains `Draft`, unaffected. Whether a genuine "cancel a `Sending` campaign that appears stuck" admin action is also wanted is asked explicitly in §8 rather than assumed.

A campaign in `Sending` can be **resumed** (§6) but never **re-sent** — re-sending a `Sent` or already-`Sending` campaign is rejected server-side (§5's duplicate-send prevention), independent of what the UI allows.

## 3. Database schema

**Migration `0016_newsletter_campaigns.sql`** — two new tables, no changes to any existing table.

```sql
CREATE TABLE newsletter_campaigns (
  id                     INTEGER PRIMARY KEY AUTOINCREMENT,
  subject                TEXT NOT NULL,
  body                   TEXT NOT NULL,        -- rich text, sanitizeRichTextHtml() at write time — identical convention to blog_posts.body
  status                 TEXT NOT NULL DEFAULT 'draft'
                           CHECK (status IN ('draft', 'sending', 'sent', 'failed')),
  intended_recipient_count INTEGER,             -- snapshotted at Draft→Sending, NULL while still Draft — the "clear recipient count before send" the confirmation dialog shows is exactly this number, not a live re-query
  created_by             INTEGER NOT NULL REFERENCES admin_users(id),
  created_at             TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at             TEXT NOT NULL DEFAULT (datetime('now')),
  sent_by                INTEGER REFERENCES admin_users(id),   -- set once, at the Draft→Sending transition (the admin who triggered the send, not necessarily who drafted it)
  sending_started_at     TEXT,
  sent_at                TEXT,                  -- set once every recipient reaches a terminal state
  deleted_at              TEXT                   -- soft delete, matching every other CMS module's convention (products/resources/blog) — a sent campaign's history is never hard-deleted
);

CREATE INDEX idx_newsletter_campaigns_status ON newsletter_campaigns(status);

-- One row per (campaign, subscriber-at-send-time) — the recipient
-- roster snapshot that makes duplicate-send prevention and mid-send
-- recovery possible without any queue infrastructure (see §4/§6).
CREATE TABLE newsletter_campaign_recipients (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  campaign_id    INTEGER NOT NULL REFERENCES newsletter_campaigns(id),
  subscriber_id  INTEGER NOT NULL REFERENCES newsletter_subscribers(id),
  status         TEXT NOT NULL DEFAULT 'pending'
                   CHECK (status IN ('pending', 'sending', 'sent', 'failed', 'skipped')),
  email_log_id   INTEGER REFERENCES email_log(id),  -- set once a real send attempt is recorded — the audit-grade detail (provider id, error message, attempt count) lives there, not duplicated here
  attempted_at   TEXT,
  UNIQUE(campaign_id, subscriber_id)   -- the real, database-enforced guarantee that one subscriber can never appear twice in one campaign's roster
);

CREATE INDEX idx_campaign_recipients_campaign_status ON newsletter_campaign_recipients(campaign_id, status);
```

**Why two tables, not one `recipients TEXT` JSON blob on the campaign row:** the per-row `status` with a real index is what lets a resume (§6) cheaply find exactly the `pending` rows without re-deriving anything, and the `UNIQUE(campaign_id, subscriber_id)` constraint is a database-enforced, not merely application-logic-enforced, duplicate-send guarantee — matching this codebase's own stated preference (seen in Phase 4's invite flow) for using a real constraint as the race-serialization point rather than trusting a read-then-write check alone.

**`schema.sql`** will be synced with both tables in the same commit, per standing discipline.

**One small, targeted `emailService.ts` extension** (not a new pipeline): a new bundled template `emails/templates/newsletter-campaign.html`, added to the existing `TEMPLATES` map and `EmailTemplateName` union exactly like the other eight — this alone means campaigns automatically get Phase 5's per-template kill switch, `email_log` recording, sender-name/reply-to settings, and the base layout, for free. The one new piece: `substitute()` gains a second, narrower pass that inserts one specific placeholder (`{{{BODY_CONTENT_RAW}}}`, a triple-brace convention distinct from every existing `{{single}}`-brace, always-escaped placeholder) without HTML-escaping — used *only* for the campaign body, which arrives already sanitized by `sanitizeRichTextHtml()` before it is ever passed to `sendEmail()`. Every other placeholder in every other template keeps today's escape-by-default behavior unchanged.

## 4. Sending workflow

**The core trade-off, stated plainly, per the explicit instruction to stop and explain rather than quietly build complexity:**

A "send to all subscribed users" action needs to call `sendEmail()` once per recipient. At today's real scale (9 subscribers) this is trivial. The question this design has to answer is what happens as that number grows, given this project has no queue/cron infrastructure to fall back on.

**Recommended approach — no new Cloudflare infrastructure, a firm safety cap instead:**

1. `POST /api/admin/newsletter/campaigns/:id/send` performs the Draft→Sending transition (§5's atomic conditional `UPDATE`), snapshots the current `WHERE status = 'subscribed'` roster into `newsletter_campaign_recipients` (all rows `pending`), and returns the confirmed `intended_recipient_count` to the caller *immediately* — the admin's browser is never left waiting on the actual sending.
2. The real send loop — one `sendEmail()` call per `pending` recipient, sequential, exactly reusing the existing retry-once/record-to-`email_log` behavior unchanged — runs inside `ctx.waitUntil()`, so it continues after the HTTP response has already been returned, without requiring any new Cloudflare product. This is a genuine Workers platform feature already available on every request, not new infrastructure to provision or bind.
3. **A hard, enforced cap of 300 recipients per send.** `waitUntil()` extends a Worker's execution past the point the response is sent, but it is not unbounded — Cloudftware's own guidance is to keep `waitUntil` work modest, and this project's Worker has no configured higher CPU/duration limit (`wrangler.jsonc` sets none). 300 sequential `sendEmail()` calls, each a real Resend HTTP round-trip, is a conservative number chosen to comfortably fit inside a `waitUntil` execution window with real margin — not a number derived from a hard platform ceiling, since Cloudflare does not publish one for `waitUntil` specifically. If the subscribed count exceeds 300 at send time, the `send` endpoint rejects the request with a clear error (*"This would require infrastructure this project doesn't currently have — Cloudflare Queues — to send safely. Contact your developer before sending to an audience this large."*) rather than silently truncating the roster or risking a send that's killed mid-way with no way to know which recipients were reached.
4. **If growth beyond 300 subscribers becomes real,** the correct next step is a genuine Cloudflare Queues-backed consumer (a `queues` binding + a `queue()` handler in `worker/index.ts`, batching sends and retrying failed messages independently of any single HTTP request's lifetime) — this is exactly the kind of new infrastructure the instruction asked to be flagged rather than built speculatively now. This design deliberately does not build it, since 300 is well above today's 9 real subscribers and there is no present need.

**Scheduled (future-dated) sending was not in the requested scope** (the bullet list is Draft/Test/Send-to-all/History/Delivery-summary — no "schedule for later"). Per the instruction to stop and explain rather than assume: genuine scheduled sending would need a Cron Trigger (`wrangler.jsonc`'s `triggers.crons`, currently absent) polling for due campaigns — real, but small, new infrastructure. This design does not include it. If scheduled sending is wanted, please say so explicitly and it can be scoped as a small, separate addition; everything else in this design (the recipient-snapshot table, the `Sending` state, the resume mechanism) would compose cleanly with it later without rework.

## 5. Duplicate-send prevention

Three independent layers, not one:

1. **Status-gated, atomically.** `POST .../send` performs `UPDATE newsletter_campaigns SET status = 'sending', sent_by = ?, sending_started_at = datetime('now') WHERE id = ? AND status = 'draft'` and checks `meta.changes === 1`. If zero rows changed, the campaign was already `sending`/`sent`/`failed` and the request is rejected (`409 CONFLICT`) *before* any recipient roster is created. This closes the double-click / two-admins-click-send-simultaneously race without needing a D1 transaction — the same conditional-`UPDATE`-as-serialization-point idiom this codebase already uses (Phase 4's invite-email-uniqueness).
2. **Roster created exactly once**, only inside the same request that won the status flip above (never speculatively, never on a retry of the send endpoint alone).
3. **`UNIQUE(campaign_id, subscriber_id)`** on `newsletter_campaign_recipients` — a database-enforced guarantee, not merely an application check, that a given subscriber can never be inserted into one campaign's roster twice, even if the snapshot logic were ever called more than once by a future bug.

The send loop itself never emails a recipient twice within one run either: each row is flipped `pending → sending` (a per-row conditional `UPDATE ... WHERE id = ? AND status = 'pending'`) immediately before the `sendEmail()` call, then `sending → sent/failed`. A concurrent resume click (§6) racing the same row simply finds it no longer `pending` and skips it — the same pattern as step 1, applied per-row.

## 6. Recovery strategy

Because the roster is durable, real D1 rows (not an in-memory loop variable), an interrupted send is always recoverable without ever risking a duplicate email:

- If the Worker executing the `waitUntil` send loop is ever terminated early (a real, if rare, possibility — a Worker eviction, an unhandled exception mid-loop), the campaign is left in `Sending` with some recipients `sent`/`failed` and the rest still `pending`. Nothing is ambiguous: the roster itself is the ground truth of what happened.
- **Resume** (`POST .../resume`, `super_admin`/`editor` per §9) re-queries `pending` rows for that campaign and continues the exact same loop — already-`sent`/`failed`/`skipped` rows are never touched again. Resuming a campaign that has no `pending` rows left is a no-op that immediately transitions the campaign to `Sent` (closing out a run that actually did finish but, for some reason, never got to flip its own top-level status — e.g. the final status-write itself was interrupted).
- This is deliberately **not** an automatic retry-on-a-schedule (that would need the Cron infrastructure §4 explicitly avoided introducing) — it is an explicit admin action, visible in the campaign's detail view whenever a campaign is sitting in `Sending` with a non-zero `pending` count, matching the "safety over convenience" instruction: nothing resends itself without a human choosing to.

## 7. Failure handling

| Scenario | Behavior |
|---|---|
| Resend is unavailable (every request in the loop fails) | Each recipient's `sendEmail()` call already retries once, then records `failed`/`permanently_failed` to `email_log` exactly as it does today for any other template — no new failure path. The campaign still reaches `Sent` once every row is terminal; the delivery summary (§8) shows 0 successful, N failed, making the outage visible rather than hidden behind a generic "Sent" label. |
| Partial delivery (some recipients succeed, some don't) | Expected, normal outcome — `Sent` with a non-100% delivery summary, not a special state. |
| An individual recipient fails (bad address, Resend rejects it) | That one `newsletter_campaign_recipients` row is marked `failed` (mirroring `email_log`'s `permanently_failed` for a 4xx Resend rejection); the loop continues to the next recipient unaffected — identical resilience to how one transactional email's failure never blocks anything else today. |
| Sending is cancelled before it begins | No server state exists yet at that point (§2) — the confirmed-send request was simply never sent. Nothing to clean up. |
| A recipient unsubscribes *during* an in-progress send | The roster was snapshotted at send-start (§4) and is not re-queried live — that recipient's row is still `pending` and will still be attempted (best-effort, matching "the recipient count shown before sending is the actual count that will be attempted," not a live-shrinking promise). Their `unsubscribe_tokens`/footer link still works normally regardless. |

## 8. Open questions for your review

1. **Should a `Sending` campaign be explicitly cancellable** (not just resumable), for the case where an admin wants to formally stop a stuck/interrupted send rather than resume it? The named scope scenario ("cancelled before it begins") doesn't require this, and the design above works correctly without it — but if you want a "Cancel" action available mid-`Sending`, I'd add it as: mark all remaining `pending` rows `skipped`, campaign status → a terminal state (reusing `Failed`, or a new `Cancelled` status if you'd rather the delivery summary distinguish "we chose to stop" from "something broke").
2. **Who can send a campaign?** Draft/edit/test-email match the existing Blog/Products/Resources CMS convention (`editor` or `super_admin`). Given the blast radius of "email every subscriber" and this phase's explicit "safety over convenience" instruction, I'd recommend restricting the actual **Send** and **Resume** actions to `super_admin` only, while leaving Draft/Edit/Test/History/Delivery-summary visibility open to `editor` too. Please confirm or adjust.
3. **The 300-recipient safety cap** (§4) — reasonable as a starting number, or would you prefer a different value? It's not tied to any hard platform limit, just a conservative margin.
4. **Scheduled (future-dated) sending** — confirmed out of scope per §4, unless you'd like it added as a small follow-up once Cron Trigger infrastructure is introduced deliberately.

## 9. API endpoints (proposed)

All under `/api/admin/newsletter/campaigns`, `super_admin`/`editor` unless noted (pending §8.2):

- `GET /` — campaign history list (id, subject, status, created_by, sent_by, created_at, sent_at, intended_recipient_count, live delivery-summary counts).
- `GET /:id` — full detail, including the live per-recipient status breakdown.
- `POST /` — create Draft (subject, body — sanitized server-side before storage).
- `PATCH /:id` — edit a Draft only (rejected if not `draft`).
- `DELETE /:id` — soft-delete a Draft only.
- `POST /:id/test` — sends the current Draft content to one or more admin-supplied test addresses via the exact same `newsletter-campaign` template/`sendEmail()` path, logged with a distinct `entityType: 'newsletter_campaign_test'` so test sends never inflate a real campaign's delivery-summary counts.
- `POST /:id/send` — the atomic Draft→Sending transition + roster snapshot (§5); `super_admin`-only per §8.2's recommendation.
- `POST /:id/resume` — continue a `Sending` campaign's remaining `pending` rows (§6); same role restriction as `send`.
- `GET /subscribed-count` — the live `COUNT(*) WHERE status = 'subscribed'`, used by the pre-send confirmation dialog.

Every mutating action (`create`, `edit`, `delete`, `test`, `send`, `resume`) writes one `auditService.record()` row (`action: 'newsletter_campaign.<verb>'`), matching the established convention exactly.

## 10. Rollback strategy

- **Migration:** `0016_newsletter_campaigns.sql` only adds two new tables — no existing table is altered, so a rollback is simply "don't run the migration" / drop the two new tables if ever needed; zero risk to `newsletter_subscribers`, `email_log`, or any other existing data.
- **`emailService.ts`'s new raw-placeholder pass** is additive and scoped to one specific placeholder name — every existing template's behavior is provably unchanged (no existing template uses `{{{BODY_CONTENT_RAW}}}`).
- **Deploy-level rollback:** if a real problem surfaces after deploying, the Worker can be rolled back to the prior deployed version via `wrangler rollback` (or a redeploy of the previous commit) exactly as any other phase's Worker changes could be — the new tables sitting unused in D1 in that case cause no harm to anything else.
