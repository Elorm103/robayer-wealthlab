# Version 2.1 Phase 4 — User Management: Implementation Report

**Status:** Complete, deployed to production, verified.
**Design:** See `docs/v2.1-phase4-design.md` (approved before implementation began, per the user's own instruction).

## What was built

- **Migration `0014_user_management.sql`** — `admin_users.created_by` (nullable, NULL for every pre-existing account), and `admin_invites` (single-use, 7-day-expiry, emailed invitation tokens — the same proven shape as `password_reset_tokens`).
- **`services/admin/adminUserService.ts`** (new) — the only code that writes to `admin_invites` and performs lifecycle actions on other admins' accounts: list/detail with full visibility (last login, last activity, active session count, failed attempts, security/account status, created-by, created-at), invite/resend/cancel, edit (name/role), disable/reactivate, soft delete, and four security actions (force password reset, force password change, force logout, unlock).
- **`services/admin/authService.ts`** (extended) — `validateInviteToken()`/`acceptInvite()` (the public half of the invite flow — creating the real `admin_users` row happens here, the one and only place a new admin account is ever created), and an extracted `issuePasswordResetToken()` helper shared between self-service `forgotPassword()` and the new admin-triggered `forcePasswordReset()`.
- **`routes/admin/users.ts`** (new) — every `/api/admin/users/*` endpoint, `super_admin`-only for reads and writes alike; plus the public `GET`/`POST /api/admin/auth/accept-invite`.
- **`admin/users/`, `admin/users/detail/?id=`, `admin/accept-invite/`** (new frontend) — roster + pending-invitations list with an invite modal; a per-admin management page (role edit, sessions, login history, security actions, danger zone); the public password-setting page linked from the invite email.

## Safety rules — what was verified, not just implemented

**Self-targeting is blocked on every mutating endpoint** (`PATCH`, disable, reactivate, delete, force-password-reset, force-password-change, force-logout, unlock) — confirmed via a real API pass: all 8 actions, called by an admin against their own account, returned `SELF_TARGETED` before any write occurred.

**The last active Super Administrator can never be disabled, deleted, or demoted** — the guard (`guardLastSuperAdmin()`) re-derives the active-Super-Admin count fresh on every call rather than trusting a cached value. Verified: demoting a *second* Super Admin down to `editor` while two existed was correctly **allowed** (one remained); the resulting single-remaining-Super-Admin state was then confirmed via direct query.

**A genuine finding, documented honestly:** with self-targeting blocked and this module closed to non-`super_admin` roles entirely, the only account that could ever legitimately trigger the "last Super Admin" rejection *from another admin's perspective* would require a second active Super Admin to exist — at which point the target isn't actually the *last* one. In today's constraint set, `last_super_admin` as a distinct rejection reason is therefore not independently reachable via the API; its practical value is (a) defense-in-depth against a future change to the self-block logic or role model, and (b) a real, if imperfect, mitigation for the documented concurrent-race edge case (two Super Admins demoting each other simultaneously) — the same accepted, pre-existing class of read-then-write race every other status-transition function in this codebase already carries. This does not change any behavior the user approved; it's reported here because it's the kind of thing worth knowing about the system rather than leaving implicit.

**Privilege escalation is structurally impossible, not just checked** — only `super_admin` can reach `/api/admin/users/*` at all (`requireRole(['super_admin'])`), and `super_admin` is already the ceiling of the 3-role hierarchy. Verified: an `editor`/`support` account hitting `GET /api/admin/users` and `POST /api/admin/users/invite` directly (bypassing the UI) both returned `FORBIDDEN`.

**Session safety** — verified precisely, not assumed: a role change revoked the target's active session (1 → 0, confirmed via the detail endpoint); a disable explicitly revoked a session row (`revoked_at` set, confirmed via direct D1 query on the exact session id, not merely inferred from the `is_active` filter). Force-password-reset and force-password-change deliberately do **not** revoke sessions (per the approved design — the reset email doesn't touch the current password until used; the change-flag is read live on every request already).

**Audit logging** — confirmed complete for every action type tested (invite, invite_accepted, updated/role-change, force_password_change, force_password_reset, unlocked): actor id, entity id, timestamp (automatic), user agent, and `before`/`after` values where applicable, all present in `audit_logs.metadata`. IP address was `null` in local `wrangler dev` (no real `CF-Connecting-IP` header exists locally) but populated correctly in production (confirmed via a real request's captured IP during production verification).

**Invite security** — re-inviting the same email revoked the prior unused token (confirmed via direct query: old token's `revoked_at` set, new token active) and the old token was rejected (`INVALID_TOKEN`) on `accept-invite`. Cancelling a pending invite removed it from the pending list. A 7-day `expires_at` was confirmed on issuance.
