# Version 3.0 — Roadmap

**Status: themes, not a plan.** This is a list of directions worth considering for Version 3.0, not a committed scope or a design document — no phase order, no timeline, no implementation detail. Each theme below either builds directly on a trigger already named in `docs/v2.1-technical-debt-register.md` (linked where applicable) or is a genuinely new capability this platform doesn't have yet. When any of these is actually picked up, it should get its own design doc following this project's established discipline (fresh architecture review → design approval → implementation → verification → documentation), the same way every Version 2.1 phase did.

## Infrastructure & reliability

**Scaling newsletter sending with Cloudflare Queues.** The current `ctx.waitUntil()` + durable-roster approach (`campaignService.ts`) works well within its configured recipient cap, but is not the long-term answer for a subscriber list in the thousands. See `docs/v2.1-technical-debt-register.md` item 1 for the exact trigger (subscriber volume approaching the cap) and why the current roster table design requires no rework to become a queue's producer/consumer model.

**R2 backup strategy.** Two real, currently-undocumented-as-mitigated gaps exist today: no backup beyond D1's Time Travel retention window, and no backup path at all for Media Library's R2-stored uploads. Both are named with concrete triggers in the technical debt register (items 8 and 9). A real Version 3.0 answer likely means introducing this project's first Cron Trigger — which would also be the natural moment to build the automated Paystack webhook reconciliation and D1 health-check jobs that were planned back in Version 1.2 but never built (see the corrected `docs/backup-and-recovery.md` and `docs/monitoring-and-alerting.md`).

**Infrastructure automation (CI/CD).** Deploys and migrations are still run by hand via `wrangler`. A real pipeline (GitHub Actions or similar) would remove the human step in tagging each deploy with its commit hash (technical debt register item 3) and make the "staging environment" pattern already sketched in `docs/deployment-checklist.md` worth actually building — right now there's no staging environment at all, and CI is the natural forcing function for finally standing one up.

**Scheduled/deferred campaigns.** Explicitly out of scope for Version 2.1 by design (Decision 4 in `docs/v2.1-phase6-design.md`) — sending a campaign at a future date/time needs real Cron Trigger infrastructure and was deliberately not simulated inside an HTTP request. Once Cron Triggers exist for any of the reasons above, this becomes a much smaller addition than it would be standalone.

## Platform capability

**Analytics and reporting.** The current Analytics module covers orders/revenue/subscribers at a summary level. Deeper reporting — cohort/retention views, campaign-level engagement metrics tied back to `email_log`, export to CSV — is a natural next step once there's enough real production history to make it worth building against.

**Advanced search.** Products, Resources, and Blog each have their own list-page search today (title/slug/tag matching per module). A real cross-content search — spanning all three, ranked, possibly public-facing — is a distinct capability none of them currently offer.

**Subscriber segmentation.** Newsletter Campaigns currently sends to "everyone currently subscribed," full stop — no lists, tags, or targeting. Segmentation (by signup source, engagement history, or admin-defined tags) is a real product capability this platform doesn't have, and is a natural pairing with the Analytics theme above (segment definitions need to be visible in reporting to be useful).

**Multi-language support.** The site and every admin-authored content type (Products, Resources, Blog) are English-only today, with no i18n scaffolding anywhere in the schema or the frontend. This is a substantial, cross-cutting theme — it touches content modeling, the public site, and the admin editors — not a small addition.

**Public API.** If ever desired: a documented, authenticated (likely API-key-based, distinct from the cookie/CSRF admin-session model) read/write surface for programmatic access to Products/Resources/Blog content, separate from the admin UI's own routes. Not currently needed by anything real — worth naming as a theme only because it would meaningfully affect how `routes/`/`services/` are structured if pursued (a public API needs its own versioning and stability guarantees that the admin API, iterated on freely across every Version 2.1 phase, has never had to honor).

## Platform hygiene

**General performance improvements.** The Phase 7 audit found and fixed the one genuine N+1 query in the codebase (`campaignService.listCampaigns()`); no other query-pattern issues were found at Version 2.1's current scale. As real traffic and content volume grow, this is worth a fresh look — particularly around list-page queries and any place read-time joins might start to matter more than they do today.
