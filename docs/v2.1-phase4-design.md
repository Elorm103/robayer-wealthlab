# Version 2.1 Phase 4 — User Management: Design Document

**Status:** Design only. No implementation code has been written. Per the user's own instruction, implementation begins only after this document is reviewed and approved.

This document is the fresh architectural audit the user asked for, plus the planned design. It covers: current authorization model, planned changes, database changes, API endpoints, permission matrix, edge cases, and rollback strategy.

---

## 1. Current authorization model (audit findings)

Read fresh, end to end, for this phase: `backend/middleware/requireAuth.ts`, `backend/middleware/requireRole.ts`, `backend/middleware/csrf.ts`, `backend/services/admin/sessionService.ts`, `backend/services/admin/authService.ts`, `backend/services/admin/loginHistoryService.ts`, `backend/services/admin/auditService.ts`, `backend/database/schema.sql`'s `admin_users`/`admin_sessions`/`audit_logs` sections, and `admin/users/index.html` (confirmed: still the original empty "Coming soon" shell — no backend exists for this module today).

**Session/auth layer (unchanged since Phase 0.1, hardened in Phase 3):**
- `admin_sessions` — one row per login, HttpOnly session cookie + separate readable CSRF cookie (double-submit pattern), 12-hour absolute TTL, `revoked_at` for logout/forced-logout.
- `requireAuth()` — the single chokepoint every `/api/admin/*` route calls first. Validates the session, and (as of Phase 3) rejects with `MUST_CHANGE_PASSWORD` for every path except session-check/change-password/logout when `admin_users.must_change_password` is set. Returns an `AdminAuthContext` (`sessionId`, `adminId`, `role`, `email`, `name`, `csrfSecret`, `mustChangePassword`).
- `requireRole(request, env, logger, auth, allowedRoles)` — a simple allowlist check against `auth.role`, called explicitly by individual route handlers after `requireAuth`. Already fully server-side; the frontend never gates access, only hides UI.
- `requireCsrf()` — double-submit cookie check on every mutating route.
- **Roles today: exactly three, flat, fixed by a DB `CHECK` constraint** — `super_admin`, `editor`, `support`. No granular permission table exists; a role is a single string column, not a set of assignable capabilities. There is currently **no way to change an admin's role except direct D1 access** — every existing admin account (including the real production one) was provisioned by hand.
- `auditService.record()` — the single `audit_logs` writer, already used by every mutation across every module. Reused as-is.

**Password/security layer (built in Phase 3, directly reusable):**
- `passwordHash.ts` (`hashPassword`/`verifyPassword`), `passwordPolicy.ts` (`validatePasswordStrength`) — reused unchanged.
- `admin_users.must_change_password` / `failed_login_attempts` / `locked_until` / `password_updated_at` — all already exist and are already read/written by `authService.login()`.
- `password_reset_tokens` — proven single-use, expiring-token shape (256-bit, `used_at` gate). The invite mechanism below reuses this exact shape rather than inventing a new one.
- `sessionService.listSessions(env, adminId, currentSessionId)`, `revokeSessionById(env, adminId, sessionId)`, `revokeAllSessionsExcept(env, adminId, exceptSessionId)`, `revokeAllSessions(env, adminId)` — **all already take an arbitrary `adminId` parameter, not a hardcoded "self."** Phase 3 only ever *called* them with the caller's own id (self-service `/admin/account/`); nothing in their implementation assumes the caller and the target are the same person. This means "view another admin's sessions" and "force logout another admin" need **zero changes to `sessionService.ts` itself** — only a new route layer that calls the same functions with a *target* id, gated by `super_admin`.
- `loginHistoryService.listLoginHistory(env, adminId, limit)` — same story: already generic, already takes a target `adminId`. Phase 3's route just always passed `auth.auth.adminId` (self). Reusable unchanged.
- `emailService.ts` — template-registry pattern (`EmailTemplateName` union + `TEMPLATES` map), reused unchanged; one new template added (`admin-invite.html`).

**What does not exist yet and is genuinely new in this phase:**
- Any admin-facing CRUD for `admin_users` beyond the account's own self-service page.
- Any invite/account-creation mechanism.
- `admin_users.created_by` (no column exists to answer "who created this admin").
- Any cross-admin "force" action (`force-password-reset`, `force-password-change`, `force-logout`, `unlock`) — Phase 3 built the *primitives* these compose from, but always self-scoped at the route layer.

**Conclusion: the existing authorization model does not need to change.** No new session mechanism, no new token mechanism, no new role taxonomy, no change to `requireAuth`/`requireRole`/`requireCsrf`. Phase 4 is additive: one new route file, one new service, one new small migration, reusing every primitive above as-is. Per the user's own instruction, this is stated explicitly rather than left implicit: **the architectural implications of this phase are "none" — no core auth/session logic changes.**

## 2. Scope decision: "Create" and "Invite" collapse into one real mechanism

The brief lists "Create administrator" and "Invite administrator (preferred where practical)" as two separate lifecycle actions. Considered two designs:

- **(a) Two separate flows** — "Create" immediately inserts an `admin_users` row with a system-generated temporary password; "Invite" sends an email link. This requires the temporary password to reach the new admin somehow — displayed once in the UI (screen-shared/screenshotted risk) or emailed in plaintext (a real credential sent over email, which this codebase's own security posture already treats as unacceptable for anything password-shaped).
- **(b) One real mechanism** — every new admin account is created via an emailed, single-use, expiring link (the exact shape `password_reset_tokens` already proves out), through which the new admin sets their **own** first password. No password ever exists in transit or at rest anywhere except the hash the new admin's own submission produces.

**Decision: (b).** This is a real, honest scope narrowing — flagged clearly here, not silently merged. It's also strictly the more secure option and matches "invite preferred where practical" taken to its logical conclusion: there's no case where directly setting a temporary password is actually better once the invite-link mechanism exists and costs nothing extra to use for both entry points. The admin UI still exposes this as a single "Add administrator" action (email, name, role) — there is no separate "Create" button that behaves differently.

## 3. Database changes — migration `0014_user_management.sql`

```sql
ALTER TABLE admin_users ADD COLUMN created_by INTEGER REFERENCES admin_users(id);
-- NULL for every admin account that predates this migration (including the
-- real production account) — an honest "unknown/pre-existing" rather than
-- a fabricated value. Populated going forward for every admin created
-- through the invite-accept flow below.

CREATE TABLE admin_invites (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  token        TEXT NOT NULL UNIQUE,                 -- 256-bit, same generation pattern as password_reset_tokens
  email        TEXT NOT NULL,
  name         TEXT,
  role         TEXT NOT NULL CHECK (role IN ('super_admin', 'editor', 'support')),
  invited_by   INTEGER NOT NULL REFERENCES admin_users(id),
  expires_at   TEXT NOT NULL,                         -- 7 days — long enough to be practically usable by email, unlike the 30-minute password-reset window (a different threat model: this isn't a bearer credential to an existing account, it's a one-time invitation)
  accepted_at  TEXT,
  admin_id     INTEGER REFERENCES admin_users(id),     -- set once accepted; links the invite to the admin_users row it created
  revoked_at   TEXT,                                   -- set if superseded by a resend, or explicitly cancelled
  created_at   TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_admin_invites_email ON admin_invites(email);
CREATE INDEX idx_admin_invites_token ON admin_invites(token);
```

No other schema changes. `admin_users` itself gains only the one nullable column. `admin_sessions`/`password_reset_tokens`/`login_history`/`audit_logs` are all reused completely unchanged.

## 4. API endpoints

All under `/api/admin/users/*` unless noted, all requiring `requireAuth` + `requireRole(['super_admin'])` — see Section 5 for why this module is `super_admin`-only for reads too, unlike Products/Resources/Blog.

| Method | Path | Action |
|---|---|---|
| GET | `/api/admin/users` | List every admin with full visibility fields (Section 6). |
| GET | `/api/admin/users/:id` | One admin's detail + their sessions + recent login history (reuses `sessionService.listSessions`/`loginHistoryService.listLoginHistory` with the target id). |
| POST | `/api/admin/users/invite` | Body `{ email, name, role }` — creates/replaces an `admin_invites` row, sends the invite email. Serves both "Create" and "Invite." |
| POST | `/api/admin/users/:id/resend-invite` | Re-sends a still-pending (unaccepted, unexpired) invite — revokes the old token, issues a new one. |
| PATCH | `/api/admin/users/:id` | Edit `name`/`role`. Role changes run the last-super-admin and self-demotion checks. |
| POST | `/api/admin/users/:id/disable` | Sets `is_active = 0`. Blocked: self, last active super_admin. |
| POST | `/api/admin/users/:id/reactivate` | Sets `is_active = 1`. |
| DELETE | `/api/admin/users/:id` | Soft delete (`deleted_at`). Blocked: self, last active super_admin. |
| POST | `/api/admin/users/:id/force-password-reset` | Sends a password-reset email to the target (reuses `authService.forgotPassword()`'s exact token/email mechanism, called directly rather than via email lookup). Blocked: self. |
| POST | `/api/admin/users/:id/force-password-change` | Sets `must_change_password = 1`. Takes effect on the target's very next request, since `requireAuth()` already reads this flag live on every call — no session revocation needed for this one specifically. Blocked: self. |
| POST | `/api/admin/users/:id/force-logout` | `sessionService.revokeAllSessions(target)` — already-existing function, unchanged. Blocked: self. |
| POST | `/api/admin/users/:id/unlock` | Clears `failed_login_attempts`/`locked_until`. Blocked: self (not actually reachable in practice — you can't be locked out and also be the one making this request — but checked anyway for consistency). |

Public, unauthenticated (grouped with the existing `forgot-password`/`reset-password` endpoints in `routes/admin/auth.ts`, not under `/users/*`, since a not-yet-an-admin invitee has no session):

| Method | Path | Action |
|---|---|---|
| GET | `/api/admin/auth/accept-invite?token=` | Validates a token (exists, unused, unexpired) — used by the accept-invite page to decide whether to show the form or an error. |
| POST | `/api/admin/auth/accept-invite` | Body `{ token, password }` — validates the token + password strength, creates the real `admin_users` row (`created_by = invited_by`, `must_change_password = 0` since they just set it themselves), marks the invite accepted. |

Every mutating endpoint above: `requireAuth` → `requireRole(['super_admin'])` → `requireCsrf` → the safety checks in Section 5 → the action → `auditService.record()`. Same five-step shape every existing mutating route in this codebase already follows.

## 5. Permission matrix

This module does not introduce new roles or a configurable permission system — it documents what the three existing roles can do *within User Management specifically*, which (unlike every other module so far) is fully closed to non-`super_admin` roles, including read access:

| Action | super_admin | editor | support |
|---|---|---|---|
| View admin list / detail | ✅ | ❌ | ❌ |
| Invite / resend invite | ✅ | ❌ | ❌ |
| Edit name/role | ✅ | ❌ | ❌ |
| Disable / reactivate | ✅ | ❌ | ❌ |
| Soft delete | ✅ | ❌ | ❌ |
| Force password reset / change / logout / unlock | ✅ | ❌ | ❌ |
| Manage **own** account (`/admin/account/`, unchanged from Phase 3) | ✅ | ✅ | ✅ |

**Why closed to `editor`/`support` even for reads**, unlike Products/Resources/Blog (which are open-read, gated-write): the visible fields here — failed login count, lockout status, active session list, another admin's login history — are themselves sensitive security/operational data about *other people's accounts*, not general content. This matches the same reasoning Phase 3 already used for sessions/login-history ("own-account-only, even for `super_admin`") applied in the other direction: here the whole point of the module is `super_admin` looking at *other* accounts, and no other role has a legitimate reason to see it at all.

**Privilege-escalation defense, enforced server-side regardless of the frontend:**
- Role values are validated against the fixed 3-value set before any write; the DB `CHECK` constraint is a second, independent enforcement layer even if application validation were somehow bypassed.
- Every route requires `super_admin` — an `editor`/`support` account hitting any `/api/admin/users/*` endpoint directly (bypassing the UI entirely) gets `FORBIDDEN` from `requireRole()`, identical to every other role-gated route in this codebase.

## 6. Visibility fields (per admin, in the list/detail response)

| Field | Source |
|---|---|
| Last login | `admin_users.last_login_at` (already exists, already set on every successful login) |
| Last activity | `MAX(admin_sessions.last_seen_at)` for that admin — the sliding "still actually using the app" signal, distinct from last *login* |
| Active sessions | `COUNT(*)` from `sessionService`-equivalent query (`admin_sessions WHERE admin_id = ? AND revoked_at IS NULL AND expires_at > now`) |
| Failed login count | `admin_users.failed_login_attempts` |
| Account status | Derived: `deleted` / `disabled` / `active` from `deleted_at`/`is_active` |
| Security status | Derived: `locked` (if `locked_until` in the future) / `must_change_password` / `ok` |
| Role | `admin_users.role` |
| Created by | `admin_users.created_by`, resolved to the creating admin's name/email via a join (falls back to "—" for the pre-existing accounts with `created_by IS NULL`) |
| Created date | `admin_users.created_at` |

## 7. Critical safety rules — implementation

1. **Last active Super Admin protection.** Before disable/delete/demote, a helper `isLastActiveSuperAdmin(env, targetId)` runs: `SELECT COUNT(*) FROM admin_users WHERE role = 'super_admin' AND is_active = 1 AND deleted_at IS NULL` — if the target is currently an active `super_admin` and this count is `1`, the action is rejected with a clear, specific error (not a generic `FORBIDDEN`) before any write happens. Same read-then-write shape every other status-transition function in this codebase already uses (e.g. duplicate-slug checks in `resourceService.ts`) — a theoretical race between two simultaneous requests demoting the last two super_admins is an accepted, pre-existing class of limitation in this codebase, not a new gap introduced here.
2. **Self-lockout prevention.** Every destructive/security action in this module (disable, delete, demote-from-super_admin, force-logout, force-password-reset, force-password-change, unlock) checks `targetId === auth.adminId` first and rejects outright — "you cannot perform this action on your own account." Deliberately simple and blanket rather than case-by-case reasoning about which specific actions would leave you locked out — matches how real systems (GitHub org ownership, AWS IAM) block self-service on your own admin permissions from the same panel that manages others. Note that role *demotion* specifically is blocked for self regardless of whether other super_admins exist — even a `super_admin` who wants to step down needs another `super_admin` to do it, removing any ambiguity about whether a self-demotion attempt was intentional or a mistake.
3. **Privilege escalation via direct API** — covered in Section 5.
4. **Server-side enforcement** — every check above runs in the route/service layer, never trusts a client-supplied flag, and produces the identical rejection regardless of what the UI would have allowed.
5. **Confirmation + audit** — every mutating endpoint writes an `audit_logs` row via the existing `auditService.record()` (actions: `admin_user.invited`, `admin_user.invite_resent`, `admin_user.invite_accepted`, `admin_user.role_changed` — with `metadata: { before, after }` — `admin_user.disabled`, `admin_user.reactivated`, `admin_user.deleted`, `admin_user.force_password_reset`, `admin_user.force_password_change`, `admin_user.force_logout`, `admin_user.unlocked`). Frontend confirmation modals reuse the exact delete-confirmation-modal pattern already used by every other list module (Resources, Blog, Products, Media).

## 8. Edge cases

- Inviting an email that already has an active `admin_users` row → rejected (uniqueness check, same shape as the slug/SKU-taken checks elsewhere).
- Re-inviting an email with a still-pending, unexpired invite → the old invite is revoked and a new one issued (friendlier than an error — matches how GitHub org invites handle "resend").
- Invite token expired or already used → `GET accept-invite` returns a clear "this invitation is no longer valid" state; `POST accept-invite` rejects the same way.
- Two invite-accept requests racing for the same token → the same atomic `used_at IS NULL` gate `password_reset_tokens` already proves out is reused here (`accepted_at IS NULL` as the gate on the accepting `UPDATE`).
- An invite is accepted for an email that became a real admin through some other path in the meantime → the `admin_users.email UNIQUE` constraint (already in the schema) rejects the `INSERT` at the DB layer; surfaced as a clear error, not a 500.
- Demoting/disabling/deleting the last active `super_admin` → rejected server-side (Section 7.1).
- Any of the seven "force"/destructive actions targeted at your own account → rejected server-side (Section 7.2).
- A role value outside the fixed 3-value set submitted directly to the API → rejected by application validation, and by the DB `CHECK` constraint as a second layer.
- A disabled or soft-deleted admin attempting to log in → already correctly rejected by the existing `is_active = 1 AND deleted_at IS NULL` filter in `login()`/`validateSession()` — confirmed via fresh reading, no new code needed here.
- Forcing a password change on an admin who is mid-session → takes effect on their very next request (Section 4's note on `force-password-change`), not merely "next login" in the literal sense — a stricter, more correct interpretation of the requirement than the literal wording, and consistent with how `must_change_password` already behaves for the self-service case in Phase 3.

## 9. Rollback strategy

- **Migration:** purely additive (one nullable column, one new table) — no destructive change, no data loss risk. This codebase's own convention (every migration to date) is forward-only; a rollback, if ever needed, would be a new follow-up migration, not an edit to `0014`.
- **Worker deploy:** reuses the existing rollback path already documented in `docs/deployment-checklist.md` (Cloudflare retains prior Version IDs; `wrangler rollback` or redeploying a prior Version ID).
- **Frontend:** `git revert` of the phase's commit(s), matching every prior phase.
- **Fastest mitigation if a bug is found specifically in the last-super-admin or self-lockout logic** (the highest-risk new code in this phase): since every `/api/admin/users/*` route is net-new and additive, the routes can be removed from `worker/index.ts`'s route table and redeployed in isolation — instantly disabling the entire module without affecting any other admin functionality, while a fix is prepared. No other module depends on anything this phase adds.

## 10. What is explicitly not built (stated, not silently dropped)

- A configurable/granular permission system beyond the fixed 3-role enum — not requested; the brief asks to "assign roles," not to design new ones.
- 2FA/TOTP enrollment — `admin_users.totp_secret` already exists as a nullable column from Phase 0.1 but has never been wired to anything; still out of scope here, unchanged.
- Bulk actions (e.g. disable multiple admins at once) — the brief describes per-admin actions; a bulk bar isn't requested and each of these actions is high-consequence enough that one-at-a-time with explicit confirmation is the safer default, not a shortcut being skipped.
