# Version 2.0 Phase 3 — Architecture & Implementation Plan

**Status: planning only. No code written, no files modified, no migrations created, nothing deployed.** This document is the output of a fresh, ground-truth re-read of the entire system as it exists at tag `v2.0-phase2-products-complete` (commit `fc14e0d`) — every claim below is sourced from actually reading the current code and D1 schema, not from memory or from the original (partially superseded) V2.0 planning docs.

**A note on phase numbering, resolved up front:** the original `docs/v2-development-roadmap.md` numbered phases 0–6, with its own "Phase 3" meaning "the GitHub-publish pipeline + Product Management." That pipeline was never built — Product Management (this project's actual "Phase 2") shipped instead with **D1 as the sole source of truth** and a Worker-rendered public surface (`/books/*`), which sidesteps the GitHub-commit mechanism entirely. The roadmap's own "Phase 1: Read-only visibility" (Dashboard, Orders, Consultation/Contact Manager, Analytics) — explicitly designed to come *before* any mutating module, for exactly the reasons restated in Section 14 — was skipped in the actual build order in favor of Media Library and Products. That gap is real, still open, and is what this document proposes to close as **Phase 3** in the project's actual (not originally-planned) sequence. Section 14 makes the full case for this scope choice.

## 1. Current architecture summary

```
GitHub Pages (static: everything except /api/* and /books/*)
        │
        ▼
Cloudflare Workers Routes (robayerwealthlab.com/api/*, robayerwealthlab.com/books/*)
        │
        ▼
Worker: robayer-wealthlab-api  (single entry point, backend/worker/index.ts)
   ├── D1: robayer-wealthlab-db      — every table, see below
   ├── R2: robayer-wealthlab-storage — media + digital-download files
   └── KV: RATE_LIMIT_KV             — fixed-window rate-limit counters
```

**Routing.** `worker/index.ts` holds one flat array of `{ pattern: URLPattern, method, handler }` entries, matched in order. There is no middleware chain — every route handler calls `requireAuth()` / `requireRole()` / `requireCsrf()` / `isRateLimited()` explicitly at its own top, by convention, not by framework enforcement. `withSecurityHeaders()` wraps every response uniformly (CSP branches on response `Content-Type`: HTML gets a real scoped policy, everything else keeps `default-src 'none'`).

**Authentication.** Session-cookie based (not JWT — chosen for real, instant revocability). `admin_users.role` is a 3-value CHECK constraint: `super_admin`, `editor`, `support`. Password hashing is PBKDF2-SHA256 via Web Crypto, **100,000 iterations** (a hard Cloudflare Workers `SubtleCrypto` ceiling — 600,000, the originally-planned figure, throws `NotSupportedError` in production; this is a real, previously-undocumented constraint worth carrying forward into any future crypto-adjacent design). Login is timing-safe (a fixed dummy hash is compared even when the email doesn't exist). Sessions are 12 hours, absolute, non-sliding (`last_seen_at` updates for observability only). CSRF is double-submit-cookie, but the meaningful control is server-side: `requireCsrf()` compares the `X-CSRF-Token` header against the `csrf_secret` value already resolved server-side for *this session* by `requireAuth()` — not a naive cookie-equals-header check. Every auth failure path (`requireAuth`, `requireRole`, `requireCsrf`) writes a distinct `audit_logs` event (`admin.unauthorized_access` / `admin.forbidden_access` / `admin.csrf_rejected`).

**Admin frontend.** Static HTML under `admin/`, no build step. `js/components/admin/admin-shell.js` is the actual session gate: on the `partials:loaded` event it `await`s `AdminAuth.requireSession()` before rendering anything; a failed session check redirects to `/admin/login/?next=<path>` (with open-redirect protection — only `^/admin/(?!/)` paths are accepted). `AdminAuth.adminFetch()` (`admin-auth.js`) wraps every request: attaches `credentials: 'include'`, reads the CSRF cookie and sets `X-CSRF-Token` on mutating methods, and unwraps the `{success, data}` / `{success:false, error}` envelope into a thrown `Error`. `css/admin.css` (1497 lines) supplies admin-only primitives (sidebar, data table, modal, pagination, skeleton loaders, stat cards); generic primitives (`.btn*`, `.card`, `.field*`, `.badge*`, `.alert*`, `.table`) live in the **site-wide** `css/components.css` and are shared with the public site.

**15 admin pages exist.** Real, fully backed: `admin/index.html` (Dashboard), `admin/login/`, `admin/products/` + `/new/` + `/edit/` (Products Module), `admin/media/` (Media Library). **9 are byte-identical empty shells** — a single `.empty-state` "Coming Soon" card, no page-specific script, no API calls: `orders`, `consultations`, `contacts`, `analytics`, `resources`, `blog`, `newsletter`, `settings`, `users`. The Worker's route table has **zero** `/api/admin/*` handlers registered for any of these 9 domains today.

**Dashboard (`admin/index.html`) already shows real data** — the one exception among the "read-only visibility" set. `GET /api/admin/dashboard/summary` (`services/admin/dashboardService.ts`) returns live D1 aggregates: revenue + order count (`purchase_sessions WHERE status='verified'`), subscriber count, consultation/contact counts (with a `status='new'` sub-count), and the last 10 `audit_logs` rows as a "Recent activity" feed. Every field independently degrades to `null` → "No data yet" on query failure — never a fake number. **One real, small bug found in this audit**: the Products KPI card is hard-coded to `null` with the comment "Product catalog integration arrives in a later phase" — written before migration `0008_products_module.sql` existed. That later phase has since shipped; the card is now stale and trivially fixable (Section 4).

**Products Module** (this project's actual "Phase 2") — D1 is the sole source of truth (`products`, `product_files`, `product_gallery`, `product_relations`), full admin CRUD (13 endpoints, `routes/admin/products.ts`), public read API (`routes/products.ts`), and server-rendered public pages (`routes/books.ts`, via a dedicated `/books/*` Workers Route — the Worker renders full HTML itself, no client-side fetch, no GitHub commit). This is the proven pattern this project should keep reusing for any future admin-managed content.

**Media Library** — `media_assets` D1 table indexing real R2 objects (`robayer-wealthlab-storage`), duplicate detection via SHA-256 content hash, soft delete, public serving via `GET /api/media/file/:key` which explicitly denies (identical 404) any key belonging to a paid product's `product_files` row, forcing paid files through the entitlement flow instead.

**Commerce / digital fulfilment** (unchanged by Phase 2, fully re-verified this pass): `POST /api/checkout/sessions` → `commerceService.createCheckoutSession` (D1-backed product lookup via `productCatalogService.fetchCatalogProduct`, a strict `isPurchasable()` whitelist, locks price/currency/metadata into the Paystack session) → Paystack hosted checkout → `POST /api/webhooks/paystack` (HMAC-SHA512 signature verification, constant-time compare) → `commerceService.handlePaymentWebhook` (two independent idempotency layers: `INSERT OR IGNORE` on `payment_transactions.paystack_reference`, then a status-gated conditional `UPDATE ... WHERE status='pending'` on `purchase_sessions`) → `fulfilmentService.fulfilPurchase` (idempotent `INSERT OR IGNORE` into `deliveries`, policy snapshotted at grant time, two transactional emails via Resend) → buyer's fulfilment page polls `GET /api/purchases/:reference` → `POST /api/purchases/:reference/downloads` mints a single-use 256-bit token (15-minute TTL) → `GET /api/download/:token` atomically consumes it and streams the file straight from R2. No CSRF on this subsystem by design (same-origin deployment plus non-session-authenticated bearer-style credentials — the purchase reference and download token are themselves the only "credentials"). Every table here (`purchase_sessions`, `payment_transactions`, `deliveries`, `download_tokens`, `email_log`) is stable and unchanged since Sprint 2.4/2.5 — Phase 3 reads them, never alters them.

**Public frontend.** Static HTML, no build system. `js/includes.js` injects `partials/header.html`/`footer.html` via `[data-include]`, then fires `partials:loaded`; `js/content-inject.js` listens for that event and populates `[data-content]`/`[data-content-href]` elements from `assets/config/site.json`. Every real page hand-authors its own `<title>`/description/canonical/OG/Twitter tags and JSON-LD (`Organization` sitewide, plus `BreadcrumbList`/`FAQPage`/`Article`/`Book` per page type) — no templating engine, no build step. `sitemap.xml` is hand-maintained (one manual edit per page shipped, confirmed against years of CHANGELOG entries). `robots.txt` is a blanket `Allow: /` with **no `Disallow` for `/admin/`** — the admin tree's only real protection is the auth layer itself, not robots.txt (worth noting, not urgent — see Section 8). The blog has exactly 1 real published article and 7 titled-but-unwritten "Coming soon" placeholder cards.

## 2. Objectives

1. **Close the operational-visibility gap.** Today, the only way to see a customer's order status, a consultation request, or a contact message is a direct D1 query — which is precisely what this engagement has been doing by hand, repeatedly, throughout every prior phase's verification work. That is not sustainable as a real operating model.
2. **Give real support/ops capability**: assign a consultation or contact message to a team member, leave an internal note, change its status, resend a customer's receipt or download link — none of which exist in the admin today.
3. **Give real revenue/growth visibility** without a manual query — a Dashboard KPI card and a dedicated Analytics page, both honest about what's real D1 data versus what genuinely cannot be embedded (Cloudflare Web Analytics has no API — Section 7 states this plainly).
4. **Fix the one small, real regression found in this audit** — the Dashboard's Products KPI card, stale since before Products Module shipped.
5. **Do all of this at the lowest realistic new-risk profile available in the current roadmap** — reuse existing, unchanged tables wherever possible; the one new schema surface (notes + assignment) is deliberately tiny.
6. **Explicitly not in scope for Phase 3**: Blog CMS, Newsletter campaign composer/send, Site Settings, User Management, any GitHub-publish mechanism. Section 14 explains why each is better sequenced later.

## 3. User stories

**Administrator (`super_admin`)**
- "I want to see this month's revenue and order count without writing SQL."
- "A customer says they never got their download email — I want to look up their order by reference and resend it."
- "I want to see every new consultation request the moment it comes in, without checking my personal inbox."

**Editor (`editor`)**
- "I want the same order/consultation/contact visibility as an admin — my day-to-day work already includes support triage, and Products already treats editor and super_admin as equals for content-adjacent work." *(Matches the existing `EDITOR_ROLES = ['super_admin', 'editor']` convention from Products — Orders resend actions and any future mutating action in this phase follow the same line.)*
- "I want to assign a consultation request to myself or a colleague and leave a note about what was discussed."

**Support (`support`)**
- "I want to see and triage consultation and contact requests — that's literally my job — without needing to be promoted to editor just to change a status or leave a note."
- "I want to look up an order to help a customer, including the Paystack gateway response, so I can actually resolve their issue." *(The existing `v2-security-review.md` already made this exact call deliberately: "The `support` role can view Orders, including real customer emails and Paystack gateway responses — reasonable for a support workflow... a deliberate, documented decision." This plan follows that reasoning and extends it consistently: support gets full read + note + status + assignment rights on Consultations/Contacts, and read-only on Orders — with the one real *send* action, resend-receipt/resend-download, reserved for editor/super_admin, since re-sending an email to a real customer is a consequential action, not a triage action.)*

**Customer**
- No new customer-facing surface in this phase. Indirect benefit only: their order, consultation, or contact message is now actually visible and actionable by a real person, rather than sitting unseen in a database table.

**Anonymous visitor**
- Zero visible change. This phase adds no new public routes, no new public API, no new JSON-LD, no new page. The public site is byte-for-byte unaffected — the same "additive, not disruptive" guarantee every prior phase has upheld.

## 4. Database impact

**New tables (2):**

```sql
CREATE TABLE consultation_notes (
  id                      INTEGER PRIMARY KEY AUTOINCREMENT,
  consultation_request_id INTEGER NOT NULL REFERENCES consultation_requests(id),
  author_id               INTEGER REFERENCES admin_users(id),
  note                    TEXT NOT NULL,
  created_at              TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_consultation_notes_request ON consultation_notes(consultation_request_id);

CREATE TABLE contact_notes (
  id                 INTEGER PRIMARY KEY AUTOINCREMENT,
  contact_message_id INTEGER NOT NULL REFERENCES contact_messages(id),
  author_id          INTEGER REFERENCES admin_users(id),
  note               TEXT NOT NULL,
  created_at         TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_contact_notes_message ON contact_notes(contact_message_id);
```
**Why two small tables, not one polymorphic "notes" table:** matches this codebase's own established preference (`consultation_requests`/`contact_messages` are already two distinct tables, not one polymorphic "inquiries" table, for the same reason). Append-only by design — no `UPDATE`/`DELETE` path is ever built for a note, mirroring `audit_logs`' own "don't rewrite history" philosophy. A note is authored by an admin, timestamped, and permanent.

**Modified tables (2, additive only):**
```sql
ALTER TABLE consultation_requests ADD COLUMN assigned_to INTEGER REFERENCES admin_users(id);
ALTER TABLE contact_messages ADD COLUMN assigned_to INTEGER REFERENCES admin_users(id);
CREATE INDEX idx_consultation_requests_assigned_to ON consultation_requests(assigned_to);
CREATE INDEX idx_contact_messages_assigned_to ON contact_messages(assigned_to);
```
**Why a single nullable column, not a join table:** a consultation or contact has at most one assignee at a time (confirmed against the real UX spec's singular "Assignments" language and against how a 1–3-person team actually triages an inbox) — a many-to-many join table would be unjustified complexity for a relationship that is never actually many-to-many in practice.

**Confirmed via direct schema read, not assumed:** `consultation_requests.status` and `contact_messages.status` are both already `CHECK (status IN ('new', 'reviewed', 'responded', 'closed'))` — no status-vocabulary change needed, Phase 3 reads and writes these exact four existing values.

**No new tables for Orders.** The Orders module is a pure read layer over `purchase_sessions`, `payment_transactions`, `deliveries`, `download_tokens`, and `email_log` — every one of these tables is already complete, indexed, and stable. The two "resend" actions (receipt, download link) re-invoke the *existing* `emailService.sendEmail()` with the *existing* templates — zero new send logic, zero new table. **Confirmed the exact `email_log` join key**: `fulfilmentService.ts` writes `entityType: 'purchase_session'` (not `'order'` — the schema comment's own example is slightly stale; the real convention is `'purchase_session'`), so the Orders detail endpoint's email-history query is `WHERE entity_type = 'purchase_session' AND entity_id = ?`.

**No new tables for Analytics.** Every KPI (revenue, orders, subscribers, downloads, consultation/contact volume, top products) is a live D1 aggregate query (`COUNT`/`SUM`/`GROUP BY`) against tables that already exist — matching `v2-analytics-spec.md`'s explicit, still-valid decision: no pre-aggregation table, no scheduled ETL, at this platform's real data volume (dozens to low hundreds of rows per table) a live query is fast enough to run synchronously on every page load. Visitor/traffic/session metrics are **not** buildable — Cloudflare Web Analytics has no programmatic API (confirmed against current Cloudflare documentation in the original spec, unaffected by anything since) — the Analytics page links out to the real Cloudflare dashboard for those rather than faking a number.

**No schema change for the Dashboard fix** — `dashboardService.getDashboardSummary()` gains one more query (`SELECT COUNT(*) FROM products WHERE deleted_at IS NULL`), reusing the exact table Products Module already created.

**Migration file: `0010_operational_visibility.sql`.** Two `CREATE TABLE`, two `ALTER TABLE ADD COLUMN`, four `CREATE INDEX` — no `DROP`, no data migration, no table recreate. This is the same safety shape as `0006`/`0007`/`0008`'s additive portions — the simplest, lowest-risk class of migration this project has shipped, and (unlike `0008`) touches zero existing table's *meaning*, only adds new, empty, nullable surface.

## 5. API design

All routes below live under `/api/admin/*`, follow the router-table pattern (`worker/index.ts`), and are implemented as thin route handlers (`routes/admin/*.ts`) delegating to a service layer (`services/admin/*.ts`), per the established convention. Every route calls `requireAuth()` first; every mutating route additionally calls `requireCsrf()`. Rate limiting reuses `isRateLimited()` with new endpoint keys.

**Consultation Manager**

| Endpoint | Purpose | Role | Notes |
|---|---|---|---|
| `GET /api/admin/consultations` | List, filtered/searched/paginated | any authenticated role | Filters: `status`, `category`, `assignedTo`. Search: name/email/description substring. `page`/`pageSize` (default 20, max 100 — matches Products' own clamping). |
| `GET /api/admin/consultations/:id` | Detail + notes thread | any authenticated role | Joins `consultation_notes` (newest last), resolves `assigned_to`/`author_id` to admin name/email. |
| `PATCH /api/admin/consultations/:id` | Update `status` and/or `assigned_to` | any authenticated role (support included, deliberately — Section 3) | Body: `{ status?, assignedTo? }`. `status` validated against the 4 real CHECK values; `assignedTo` validated as a real, active `admin_users.id` or `null` (unassign). |
| `POST /api/admin/consultations/:id/notes` | Append a note | any authenticated role | Body: `{ note: string }`, 1–2000 chars, required. No edit/delete endpoint exists — append-only by design. |

**Contact Manager** — identical shape, `contact_messages`/`contact_notes` in place of `consultation_requests`/`consultation_notes`. No `category` filter (the table has no `category` column).

**Orders**

| Endpoint | Purpose | Role | Notes |
|---|---|---|---|
| `GET /api/admin/orders` | List | any authenticated role | Filters: `status` (the 6 real `purchase_sessions.status` values), `productSlug`, `dateFrom`/`dateTo`. Search: `purchase_reference`/`customer_email`. |
| `GET /api/admin/orders/:reference` | Detail | any authenticated role | Joins `payment_transactions` (gateway response, for support debugging), `deliveries`, `email_log WHERE entity_type='purchase_session' AND entity_id=?`. Reference validated against the existing `/^RWL-\d{4}-\d{6,}$/` pattern before touching D1 — identical convention to the public `routes/purchases.ts`. A visible flag on any row whose `customer_email` matches the known synthetic test pattern (`checkout+rwl-...@robayerwealthlab.com`) so QA activity is never mistaken for a real sale — carries forward a real, previously-stated finding from this exact engagement's own production-readiness work. |
| `POST /api/admin/orders/:reference/resend-receipt` | Re-send the purchase-receipt email | `super_admin`/`editor` only | No body. Re-invokes `emailService.sendEmail()` with the existing `purchase-receipt` template and the order's real, already-stored data — no new send logic. Session must be `verified` (resending a receipt for a pending/failed order makes no sense and is rejected). |
| `POST /api/admin/orders/:reference/resend-download` | Re-send the secure-download email | `super_admin`/`editor` only | Same shape. Re-sends the `secure-download` template linking to the fulfilment page — **does not** mint a new download token itself (a fresh token is only ever minted when the customer actually clicks Download on that page, preserving the existing "token minted at point of use" security design unchanged). |

**Analytics**

| Endpoint | Purpose | Role | Notes |
|---|---|---|---|
| `GET /api/admin/analytics/summary` | KPI cards with period-over-period comparison | any authenticated role | Query params: `from`, `to` (defaults: last 30 days). Returns revenue, orders, new subscribers, downloads served, consultation/contact volume — each `{ current, previous, deltaPercent }`. Pure D1 aggregates, no caching layer. |
| `GET /api/admin/analytics/timeseries` | Orders-per-day / subscribers-per-day | any authenticated role | Same date range; returns two arrays of `{date, count}` for the dependency-free inline SVG chart (Section 9). |
| `GET /api/admin/analytics/top-products` | Real product ranking | any authenticated role | `GROUP BY product_slug` over `purchase_sessions WHERE status='verified'` in the date range, joined to `products.title` for display. |

**Dashboard (modified, not new)**

| Endpoint | Change |
|---|---|
| `GET /api/admin/dashboard/summary` | Add one query, wire the real `productsCount` field (currently hard-coded `null`) to `SELECT COUNT(*) FROM products WHERE deleted_at IS NULL` — the exact same fault-tolerant "degrade to null on error, never fake a number" pattern the other five cards already use. |

**Validation, consistently applied:** every list endpoint clamps `page`/`pageSize` server-side (never trusts an unbounded client value — same pattern as `handleProductsList`); every status/assignee mutation validates against the real, current set of legal values (never a free-text field); every ID/reference parameter is regex-validated before it reaches a D1 query. Every mutating endpoint requires CSRF; every endpoint requires a valid session. Rate limiting: reads share a generous session-scoped limit (proposed `{endpoint: 'admin-ops-read', limit: 120, windowSeconds: 900}`); writes reuse the stricter shape already proven on Products (`{endpoint: 'admin-ops-write', limit: 60, windowSeconds: 900}`).

## 6. Admin interface

**New pages** — replacing 4 of the 9 empty shells: `admin/orders/`, `admin/consultations/`, `admin/contacts/`, `admin/analytics/`. (`resources`, `blog`, `newsletter`, `settings`, `users` remain shells, deliberately — Section 14.)

**Orders.** Data table (reusing `.admin-table-wrap`/`.toolbar`/`.pagination` from `admin.css`): Reference, Product, Customer email, Amount, Status badge (`.badge--success/warning/error/info`, mapped from the 6 real statuses), Date. Filter row: status, product, date range. Detail is a **slide-out drawer**, not a separate page — per the existing UX spec's own reasoning, quick-review workflows (unlike Products/Blog's deep-editing workflows) keep the list visible behind them. Drawer shows payment details, delivery status, full email history, and the two resend buttons (hidden entirely for `support` role, not just disabled — the frontend hides what the backend would reject, consistent with "the frontend hiding a button is UX, never the security boundary" already stated in this project's own security review, but still worth doing for a clean support-role experience). Every "Resend" click confirms via the existing Modal primitive with the real consequence stated ("Resend the download link to jane@example.com?"), never a bare "Are you sure?".

**Consultation Manager / Contact Manager** (near-identical, built as one shared pattern with two thin instantiations — matching how `admin-product-editor.js`/`admin-products.js` already generalize cleanly). List: Name, Category (consultations only), Status, Assigned to, Received date; filters status/category/assignee; search name/email/content. Detail drawer: full request fields, a Status `<select>` (the 4 real values), an Assignee `<select>` (populated from `admin_users`, active only), an append-only Notes thread (timestamped, author-attributed, no edit/delete UI since none exists server-side either). No `mailto:`-based "Email Reply" action is being deferred out of this plan's scope — it's a one-line addition (`<a href="mailto:...">`) worth including at build time since it requires no new backend at all, but is not itself a section-4/5 dependency.

**Analytics.** Single page, date-range picker (default last 30 days, matching the spec). KPI row reusing `.stat-card`. Two small inline SVG time-series charts (Section 9) — orders/day, subscribers/day. A Top Products table. A visible, honestly-worded link out to the real Cloudflare Web Analytics dashboard for Visitors/Sessions/Traffic Sources — never a blank chart pretending to be data, matching the existing spec's own explicit "show what's real, link to what isn't" rule.

**Dashboard.** No new page — the Products KPI card fix (Section 4/5) is the only change, and it uses the exact same rendering path (`admin-dashboard.js`'s existing "null → 'No data yet'" branch) every other card already goes through.

**Reusable components, all pre-existing, none newly invented:** `.admin-table-wrap`, `.toolbar`/`.search-bar`, `.pagination`, `.modal-overlay`/`.modal`, `.empty-state`, `.skeleton`, `.stat-card`, `.badge*`, `.field*`, `.btn*`, `.card`, `.alert*` — every one of these already exists in `admin.css`/`components.css` from the Dashboard/Media/Products build-out. Phase 3 introduces **zero new CSS primitives** — the drawer pattern for Orders/Consultation/Contact detail is the one genuinely new interaction shape (the UX spec calls for it explicitly and no existing admin page uses one yet), built from `.modal-overlay`'s existing scrim/shadow tokens, anchored to the right edge instead of centered.

**Permissions, summarized:** `super_admin`/`editor` — full read/write on all 4 new modules including Orders resend actions. `support` — full read/write on Consultation/Contact Manager (status, assignee, notes — this *is* the support role's actual job); read-only on Orders (no resend buttons rendered, and the backend rejects the calls regardless per Section 5); read-only on Analytics (nothing to mutate there for any role).

**UX considerations carried forward from the existing spec, unchanged by this plan:** every list has a real, specific empty state (never a blank table); every destructive-adjacent action confirms via modal with the real consequence stated; every save shows a real success/failure state, never a silent no-op; dark mode works automatically everywhere because every new screen is built from the same role-mapped design tokens the rest of the admin and the public site already share — no separate dark-mode admin theme to build or maintain.

## 7. Public website changes

**None.** This is the cleanest possible answer for a phase whose entire premise is internal operational tooling: zero new public routes, zero new public API endpoints, zero new public-facing JSON, zero SEO surface change, zero new page for a visitor to ever land on. `sitemap.xml` is untouched. `robots.txt` is untouched (though see Section 8's flag on `/admin/` not being disallowed — a pre-existing condition this phase doesn't worsen, and arguably should fix in passing since it's a one-line, zero-risk change). The public commerce/checkout/download flow that Section 1 describes is read-only from this phase's perspective — Orders reads `purchase_sessions` but never writes to it, so there is no risk of this phase introducing a regression in the revenue-critical path.

**Indirect visitor-facing benefit, not a "change":** faster, more reliable responses when a customer emails or calls about a consultation/contact/order, since a real person can now find and act on their record in seconds instead of asking an engineer to run a query.

## 8. Security review

**Authentication implications:** none new — every route in this phase reuses `requireAuth()` exactly as Products/Media already do. No new session mechanism, no new cookie, no new token type.

**Authorization:** the one genuinely new authorization decision this phase makes explicit (not implicit) is giving `support` full read/write on Consultation/Contact Manager while keeping Orders read-only and Analytics read-only for that role — reasoned in Section 3, consistent with the existing security review's own precedent of deliberately, visibly deciding what `support` can touch rather than defaulting into it. Every mutation is `requireRole()`-gated server-side; the frontend hiding a button is never treated as the actual boundary.

**CSRF:** every new mutating endpoint (`PATCH` status/assignee, `POST` notes, `POST` resend-*) requires `requireCsrf()` — no exception, unlike the commerce subsystem's deliberate CSRF-free design (which relies on a different trust model entirely, not applicable here since these are real session-authenticated admin mutations).

**XSS:** the one new stored-text surface is `consultation_notes.note` / `contact_notes.note` — free-text, admin-authored, rendered back to other admins (never to a public visitor). Rendered via `textContent`, never `innerHTML`, on the frontend (matching how every other admin-authored text field, e.g. product titles, is already rendered in `admin-products.js`) — this is sufficient because the content is never interpreted as HTML; no sanitizer is needed here the way Products' rich-text `description` field needed `richTextSanitizer.ts`, since notes are plain text by design, not rich text. **If a future iteration ever adds rich-text notes, reuse `backend/utils/richTextSanitizer.ts`'s existing HTMLRewriter-based, zero-new-dependency pattern rather than introducing a new sanitizer or an npm package** — this is a real, proven-in-production pattern this project should keep defaulting to.

**Validation:** every new endpoint follows the existing `validateBody`/whitelist convention — status values checked against the real CHECK-constraint set, assignee IDs checked against real active `admin_users` rows (never trusting a client-supplied ID blindly, mirroring how Products validates `mediaId` references today), references/IDs regex-validated before touching D1.

**Rate limiting:** extends the existing KV-backed `isRateLimited()` — no new mechanism. Read endpoints get a generous session-scoped limit; writes reuse the Products-write shape.

**Audit logging:** every mutation (`consultation.status_changed`, `consultation.assigned`, `consultation.note_added`, `contact.status_changed`, `contact.assigned`, `contact.note_added`, `order.receipt_resent`, `order.download_resent` — and the equivalent `contact.*` set) writes through the single existing `auditService.record()` — the same one function every other admin mutation in this codebase already goes through, feeding the same Dashboard activity feed this phase also (indirectly) makes more useful by giving it more real events to show.

**New attack surface introduced by Phase 3, stated plainly:**
1. **Orders' resend actions are real email sends triggered by an admin action** — a compromised or malicious admin/editor session could spam a real customer's inbox with repeated receipt/download emails. Mitigation: rate-limited (per Section 5), fully audit-logged with actor attribution, and — unlike the commerce subsystem's automated sends — each one requires a deliberate, confirmed (modal) admin click; there's no path to trigger it without a valid, CSRF-verified, role-checked session.
2. **The Assignee field on Consultation/Contact Manager exposes the list of admin users (name/email) to every authenticated role**, including `support` — a deliberate, small information disclosure (any logged-in admin can already see this by nature of picking an assignee from a dropdown). No new risk beyond what already exists once any authenticated session exists at all; not worth gating behind a separate permission for a 1–3-person team.
3. **`consultation_notes`/`contact_notes` are permanent and un-editable by design** — an admin who makes a genuine mistake in a note (e.g. a typo, or notes something incorrect) cannot correct it, only add a follow-up note. This is a deliberate trade-off (matching `audit_logs`' own "don't rewrite history" philosophy) but is worth stating as a real, small UX friction, not an oversight.
4. **`/admin/` is not disallowed in `robots.txt`** — a pre-existing condition, not introduced by this phase, but worth fixing in the same pass since it's a zero-risk, one-line change (`Disallow: /admin/`) and this phase is precisely the moment the admin surface gains its first genuinely sensitive internal data views (Orders' real customer emails/payment data) beyond what Products/Media already exposed to an authenticated session.

No finding above rises to a severity that should block this phase's design — every one has a stated mitigation, matching the discipline the original `v2-security-review.md` already established.

## 9. Performance considerations

**Caching:** none needed, and none proposed. Every new query is either a single indexed lookup (order/consultation/contact detail) or a small aggregate (`COUNT`/`SUM`/`GROUP BY`) over tables in the dozens-to-low-hundreds-of-rows range — the same conclusion `v2-analytics-spec.md` already reached and re-confirmed here against the real, current row counts (checked at the Phase 2 checkpoint: 11 `purchase_sessions`, single-digit `consultation_requests`/`contact_messages`). A caching layer would be solving a scale problem this platform doesn't have.

**D1 queries:** every new list endpoint gets its own targeted index (`idx_consultation_requests_assigned_to`, `idx_contact_messages_assigned_to`, plus the two new note-table indexes) so filtering/joining never falls back to a full table scan, even as data grows. The existing `idx_consultation_requests_status`/`idx_contact_messages_status`/`purchase_sessions` status/slug indexes already cover the other new filters — no redundant index needed there.

**R2 usage:** zero — this phase touches no file storage at all (Orders reads `deliveries`/`download_tokens` metadata only, never the R2 object itself).

**Worker execution:** every new route is a handful of D1 queries per request, well within a Worker's CPU budget; no new external HTTP calls are introduced (unlike Products' Media Library integration or commerce's Paystack calls) — this phase is the lightest-weight module class this project has built.

**Pagination:** every list endpoint follows the exact clamping convention already proven on Products (`page` ≥ 1, `pageSize` 1–100, default 20) — never an unbounded query.

**Scalability:** the single genuine future trigger, already named in the existing risk assessment and re-confirmed still valid here, is: if live D1 aggregation for Analytics measurably slows down as real order/subscriber volume grows, add a scheduled rollup table then — a concrete, observable trigger, not a speculative one. Nothing in this phase should be over-built against a scale this platform doesn't have yet.

## 10. Migration strategy

**Sequencing, matching the "additive, not disruptive" discipline every prior phase has followed:**

1. **Migration `0010_operational_visibility.sql` applied to production D1 first**, before any new route is deployed — the same order Products Module (0008/0009) and Media Library (0007) already used, verified working.
2. **Backend routes + services deployed second**, registered in `worker/index.ts`'s route table. Since every new endpoint is additive (new `URLPattern` entries, no existing pattern touched), this is a zero-risk deploy for every existing route — the same guarantee every prior phase's deploy has upheld and been verified against in production immediately afterward.
3. **Admin frontend pages deployed third** — replacing 4 of the 9 static shell files with real pages. Since these are static files served by GitHub Pages (not the Worker), this step is a normal `git push`, fully independent of the Worker deploy, and can be sequenced either before or after step 2 without breaking anything (a real page calling a not-yet-deployed API would simply show its existing, honest loading-error state, matching how every admin page in this codebase already handles a failed fetch).
4. **Production verification**, following this project's own established pattern: typecheck clean → local `wrangler dev` full-flow verification (create a note, assign a consultation, change a status, resend a receipt against a real but disposable test order) → adversarial pass (role-boundary checks: does `support` correctly get rejected on `POST resend-receipt`? does an unauthenticated request to any new endpoint correctly 401?) → deploy → production smoke test with disposable test data, cleaned up immediately after, exactly as Products Module's own two audit passes did.

**Rollback considerations:**
- **Worker code**: `wrangler rollback` to the previous Version ID, or redeploy the prior tagged commit (`v2.0-phase2-products-complete`) — instant, zero database interaction required, since nothing in this phase's rollback needs the new tables to disappear (they'd simply go unused again).
- **Database**: forward-only by this project's own established convention (no `.down.sql` exists for any of 0001–0009). Rolling back `0010` would mean dropping `consultation_notes`/`contact_notes` and the two `assigned_to` columns — should only be done if genuinely necessary, and only after confirming no real notes/assignments have been created that anyone would miss (a real, if small, judgment call to make at rollback time, not before).
- **Admin frontend**: reverting 4 static HTML files to their prior "Coming Soon" shell content is a trivial, instant `git revert` — no database interaction, no Worker interaction.
- **No public-site rollback consideration exists** for this phase at all, since Section 7 confirms zero public-site change.

## 11. Testing strategy

**Unit-level (logic, not framework-based — matching this project's existing "no test framework installed, verify via typecheck + real requests" pattern):** validate the new status/assignee validation logic against every real CHECK-constraint value plus the invalid-value rejection case; validate the Orders resend endpoints correctly reject a non-`verified` session; validate the Analytics aggregate queries against a known, hand-computed small dataset (e.g. 3 known `purchase_sessions` rows → confirm the revenue sum is exactly right, not approximately right).

**Integration (real `wrangler dev` + real local D1, the pattern every phase since Media Library has used):** full flow per module — create a consultation via the *public* form (already-existing, unchanged code path) → confirm it's visible in the new admin list → assign it → add a note → change its status → confirm the Dashboard activity feed shows all of it. Same shape for Contact Manager. For Orders: use one of the existing disposable-test-purchase patterns already proven in this engagement (a real Paystack test-mode session, or a directly-inserted test `purchase_sessions` row with `status='verified'`) → confirm it appears in the list, the detail drawer shows correct payment/email history, and both resend actions actually re-invoke `sendEmail()` (verifiable via the real `email_log` row it produces).

**Browser verification (not API-only — matching this project's own explicit standing instruction from every prior phase):** walk every new page as each of the three roles in a real browser — confirm `support` genuinely cannot see the Orders resend buttons and genuinely gets a 403 if the request is forced directly; confirm the slide-out drawer opens/closes/traps focus correctly; confirm dark mode renders correctly with zero additional CSS (per Section 6's claim); confirm mobile/responsive behavior at the same breakpoints every prior admin page has been checked against.

**Adversarial testing (matching the two-pass discipline Products Module's final acceptance audit established as this project's own standard):**
- Attempt every mutation as `support` directly against the API (bypassing the UI) — confirm server-side 403, not just a hidden button.
- Attempt a CSRF-token-less mutation — confirm 403.
- Attempt to assign a consultation to a non-existent or soft-deleted `admin_users.id` — confirm rejection, not a silent FK-violation 500.
- Attempt to resend a receipt for a `pending`/`failed`/`expired` order — confirm a clear rejection, not a confusing send-to-nobody.
- Attempt SQL-injection-shaped input in the notes/search fields — confirm parameterized `.bind()` queries hold (this project's unbroken record since V1.0, re-verified rather than assumed).
- Re-run this exact same "second, independent adversarial pass" discipline that found the real status-persistence bug in Products Module — do not skip it because the first pass "felt" thorough.

**Production verification, post-deploy:** create one real, disposable test consultation/contact via the public forms, walk it through assign → note → status-change → resolve in the real production admin, confirm the Dashboard/Analytics numbers reflect it correctly, then clean up the test data exactly as every prior phase's production verification has done — leaving production in the identical state it was in before the test.

## 12. Risks

**Technical risks**
- *Low* — this phase's entire technical shape (thin route → service → D1, role-gated, audit-logged, rate-limited) is a direct copy of the Products Module pattern already proven correct in production, including through Products' own adversarial audit process. The one genuinely new interaction shape (the slide-out drawer) is a UI pattern risk, not a data-integrity risk — worst case is design iteration during build, not a production incident.
- *Mitigation:* build Consultation Manager first (Section 13) as the pattern-proving module, exactly as Media Library was sequenced before Products to prove the upload pipeline at lower stakes first.

**Architectural risks**
- *Very low.* Nothing in this phase introduces a new mechanism at the level of the `/books/*` Workers Route or the GitHub-publish pipeline design (never built, and this phase still doesn't need it). Every new table is additive; every new route follows an existing pattern exactly.
- One real, worth-naming architectural question: **should Orders eventually need a real "refund" action?** `purchase_sessions.status` already schema-supports `cancelled`/`refunded` but no code path reaches either today. This phase deliberately does **not** build refund processing (explicitly named as a non-goal in the original roadmap, still valid) — but the Orders detail drawer built here is the natural future home for that action, so its layout should leave room for it without needing a redesign later.

**UX risks**
- The append-only notes design (Section 8, finding 3) is a real, if small, friction point — an admin cannot fix a typo in a past note. Accepted deliberately for this phase (matching `audit_logs`' own precedent) rather than building an edit/delete capability that would complicate the audit story for a genuinely low-value convenience.
- Giving `support` full read/write on Consultation/Contact but read-only on Orders could feel inconsistent to a support agent who also needs occasional order context. Mitigation: they *can* still read full order detail (including gateway response) — only the two send actions are gated, and that's a deliberate, previously-stated security decision (Section 3), not an oversight to smooth over.

**Mitigations, summarized:** build in the sequence in Section 13 (lowest-stakes module first), reuse every existing pattern rather than inventing new ones, run the same two-pass adversarial audit discipline Products Module established as this project's real standard, and keep the public site completely untouched throughout (Section 7) so there is zero blast radius to the revenue-critical path regardless of what happens during this phase's build.

## 14. Recommendation

**Build Phase 3 in the order given in Section 13: Consultation Manager → Contact Manager → Orders → Analytics/Dashboard fix.**

**Why this scope, not Blog CMS or the GitHub-publish pipeline (the original roadmap's literal "Phase 3"):** the GitHub-publish pipeline was designed to unlock Products/Blog/Resources/Settings publishing — but Products already shipped without it, proving a better pattern (D1-direct + Worker-rendered) exists and works in production. Building the GitHub-publish pipeline now would mean building genuinely the highest-risk new mechanism in the entire original plan (per `v2-risk-assessment.md`'s own explicit ranking) for a problem this project has already solved a different, safer way. There is no longer a good reason to build it, for Blog or anything else — a future Blog CMS should use the exact same D1-direct, Worker-rendered pattern Products proved, not the GitHub-commit pipeline the original spec assumed (the CMS spec's own "Option A" already anticipated this, coincidentally, though it stopped short of full server-rendering — Products' proof that full server-rendering works even better than Option A's client-fetch shell should inform that future design too).

**Why this scope, not Newsletter campaigns, Settings, or User Management:** each of these is explicitly, deliberately sequenced last in the original roadmap for a reason that's still true today — they're the highest-blast-radius mutating actions in the entire admin surface (a bad campaign send reaches every subscriber at once; a bad settings change is instantly, publicly visible; and User Management has zero real urgency with exactly one real admin user in production). Building them now, before the lower-stakes operational-visibility modules exist, would repeat the exact sequencing choice this project already made once (prioritizing Media Library + Products ahead of the roadmap's own "Phase 1" for legitimate reasons) — but doing it a second time, for a second roadmap phase, compounds the same gap rather than closing it.

**Why this scope is the right "Phase 3," concretely:** it is the lowest-risk, highest-certainty-of-real-value work still on the table. It touches zero public-facing surface. It requires the smallest possible new migration this project has shipped since `0006`. It reuses, without modification, every commerce/fulfilment table and every existing service this engagement spent five sprints proving correct. And it closes a gap this exact engagement has personally felt and worked around, repeatedly, by hand, throughout every verification pass performed across Phases 0.1 through 2 — the honest, first-hand evidence that this gap is real and overdue, not a theoretical nice-to-have.

**Sequencing within the phase, restated:** Consultation Manager first because it's genuinely the lowest-stakes place to prove the one new UI pattern (the slide-out drawer) this phase needs; Contact Manager second because it's a near-free re-application of that now-proven pattern; Orders third because it's the first module touching real payment/PII data and a real send action, and benefits from the team being warmed up first; Analytics/Dashboard-fix last because it's genuinely the lowest-urgency piece and is more meaningful to build once Stages 1–3 have produced real data for it to display. A full, fresh acceptance audit closes out the phase — not a formality, but the same discipline that found a real, shipped bug in Products Module's own "final" pass.

**Do not begin implementation from this document alone.** This plan is the input to a future planning/build conversation, not an authorization to start writing code — per this task's own explicit instruction.
