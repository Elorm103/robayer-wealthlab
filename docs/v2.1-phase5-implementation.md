# Version 2.1 Phase 5 — Settings: Implementation Report

**Status:** Complete, deployed to production, verified.
**Design:** See `docs/v2.1-phase5-design.md` (approved before implementation began, per the user's own instruction).

## What was built

- **Migration `0015_site_settings.sql`** — new `site_settings` key-value table (seeded with `settings_schema_version = 1`), plus a recreate-copy-swap widening of `email_log.status`'s CHECK constraint to add `'skipped'` (SQLite has no `ALTER TABLE ... ALTER COLUMN`; all pre-existing rows preserved verbatim).
- **`services/admin/settingsService.ts`** (new) — the only code that writes to `site_settings`. Owns six editable settings (maintenance mode, download defaults, email sender/reply-to, per-template send toggles), each returned tagged with its `SettingsField<T> = { value, source, editable }` — `source` is `site_settings`, `wrangler_var`, `secret`, or `derived`, and `editable` is `false` for anything not authoritatively owned by `site_settings`, regardless of what a client sends. Read-only diagnostics (payment, email, system) are aggregated live from `payment_transactions`, `email_log`, `env`, and D1's own `d1_migrations` table — nothing is duplicated into new storage.
- **`middleware/maintenanceMode.ts`** (new) — checked once per request in `worker/index.ts`, before route dispatch. Gates exactly `/api/*`, `/books/*`, `/resources/*`, `/blog/*` — the only paths this Worker actually serves — with `/api/health`, `/api/admin/*`, and `/api/webhooks/*` exempt. Returns a JSON 503 for `/api/*` and a minimal, self-contained HTML 503 for the other three, both with `Retry-After: 300`.
- **`services/emailService.ts`** (extended) — `sendEmail()` now checks the per-template kill switch before rendering or calling Resend; a disabled template records `email_log.status = 'skipped'` and returns without attempting a send. `callResend()` now uses the configured sender name and optional reply-to.
- **`services/productService.ts`** (extended) — `createProduct()` gained an `options.applyDownloadDefaults` parameter (default `true`); a genuinely blank (`null`/`undefined`) download field on a new product resolves to the configured site default. `duplicateProduct()` explicitly passes `applyDownloadDefaults: false` so a duplicate's real source values are always carried over verbatim, never silently overwritten.
- **`routes/admin/settings.ts`** (new) — `GET`/`PATCH /api/admin/settings`, `GET /api/admin/settings/status`, all `super_admin`-only (reads as well as writes, matching Phase 4's Users-module posture).
- **`worker/env.ts`** — optional `DEPLOYED_COMMIT`/`DEPLOYED_AT` fields, set via `wrangler deploy --var` at deploy time (Workers has no runtime way to introspect its own deployed commit).
- **`admin/settings/index.html`** + **`js/components/admin/admin-settings.js`** (new) — Platform, Email, Payments, and System Information cards; every editable field carries a visible ownership badge; the 8 per-template toggles render dynamically; a settings-schema-version-mismatch banner is wired to the real comparison from the API.

## Verified, not just implemented

**Settings ownership is enforced server-side, not just displayed.** Every field in `GET /api/admin/settings`/`status` responses carries `source`/`editable`; the six `site_settings`-owned fields are the only ones `editable: true`. Confirmed by reading the field-tagging code directly (`settingsService.ts`'s `field()` calls) rather than trusting the UI rendering alone.

**Maintenance mode gates exactly the four path groups, nothing more.** Verified against the real Worker (not the local dev-proxy, which only forwards `/api/*` and would have silently hidden a gap): `/api/health` → 200 (exempt), `/api/newsletter` → 503 JSON with the configured message and `Retry-After: 300`, `/books/`, `/resources/`, `/blog/` → 503 with a minimal, self-contained HTML page, `/api/admin/settings/status` → normal auth flow (401/200, never 503), `/api/webhooks/paystack` → normal routing (404, never 503).

**The per-template kill switch actually gates the send, not just the UI checkbox.** A real newsletter signup with `newsletter-welcome` disabled produced `email_log.status = 'skipped'`; the same signup with the template re-enabled produced `status = 'permanently_failed'` (the real send path was attempted — it only failed because local `wrangler dev` has no valid Resend key) — confirming the disabled path and the real-send path are genuinely distinct code paths, not a cosmetic flag.

**Download defaults apply only to genuinely new products, never retroactively.** A new product created with blank download fields resolved to the configured site default (`7`/`45`) exactly. Duplicating an existing product (real values `5`/`30`) produced a duplicate with those same real values — not the site default — confirming `applyDownloadDefaults: false` is actually wired into `duplicateProduct()`'s call, not just documented as an intent.

**Validation rejects bad input server-side on every editable field**, independent of the UI: an invalid reply-to email, a negative/out-of-range/non-integer download limit, a non-boolean `maintenanceMode.enabled`, an empty sender name, a non-boolean template toggle, and an unrecognized template name were all rejected with `400 VALIDATION_ERROR` and a field-specific message — and none of the rejected attempts wrote anything to `site_settings` (validated fully before any write, not per-field-as-you-go).

**Audit logging is complete, including for simple toggles.** A single `PATCH` touching six keys produced six distinct `audit_logs` rows (`action: 'site_settings.updated'`), each with `{ key, before, after, ip, userAgent }` — verified for a boolean-object setting (`maintenanceMode`), two numeric settings, a string setting, and the per-template toggle map. In production, `ip` is the real `CF-Connecting-IP`; in local `wrangler dev`, it is `null` (no such header exists locally) — a pre-existing, already-documented limitation of this codebase's local dev environment, not new to this phase.

**Access control matches Phase 4's posture exactly.** A `support`-role account (local) and a real `editor`-role account (production) both received `403 FORBIDDEN` on `GET`/`PATCH /api/admin/settings` and `GET /api/admin/settings/status`. A raw `PATCH` with no `X-CSRF-Token` header was rejected with `403` before reaching any validation logic, in both local and production testing.

**Schema-version mismatch detection works.** Directly setting the stored `settings_schema_version` to an unexpected value made `GET /api/admin/settings/status` correctly report `matches: false`, and the admin UI's mismatch banner rendered — confirmed via a real screenshot, not just the API response.

## Findings worth recording

**`wrangler dev`'s route-pattern simulation makes "environment" always read "production" locally.** `getSystemDiagnostics()` derives `environment` from `new URL(request.url).hostname === 'robayerwealthlab.com'`. Because `wrangler.jsonc` configures Workers Routes under `robayerwealthlab.com/*` and no `dev.host` override, `wrangler dev` simulates the request as if it arrived at that hostname even when actually served from `localhost:8787` — this is genuine, existing `wrangler dev` behavior, not a bug introduced this phase, and it self-corrects the moment the Worker actually runs in production (confirmed: production correctly reports `"production"`). The field cannot be observed reading `"development"` in this project's local setup as a result; this is an accepted, documented limitation of the local dev environment rather than a defect in the derivation logic.

**A migration-tooling slip, caught and corrected before it caused drift.** Migration `0015` was applied to production D1 via `wrangler d1 execute --file=`, which runs the SQL correctly but does not register the file in Wrangler's own `d1_migrations` tracking table (unlike migrations 0006–0014, which were applied via `wrangler d1 migrations apply` and registered automatically). The schema changes themselves were correct and verified (`site_settings` seeded, `email_log`'s 25 real production rows preserved through the recreate-copy-swap) — only the tracking row was missing, causing `GET /api/admin/settings/status` to briefly report `currentMigration: "0014_user_management.sql"` in production immediately after deploy. Fixed by inserting the matching `d1_migrations` row directly (`id=15, name='0015_site_settings.sql'`), verified via a fresh status check afterward. No data was at risk at any point — this was purely a tracking-table gap, not a schema divergence.

**Maintenance mode was deliberately not flipped on in production during verification.** Unlike every other Phase 5 behavior, enabling maintenance mode for even a single request would return real 503s to real visitors on `/api/*`, `/books/*`, `/resources/*`, `/blog/*` — a different risk category from testing against a disposable admin account. The gating logic itself is unchanged, already-exercised code (`middleware/maintenanceMode.ts`, verified byte-identical between the fresh read and the deployed version) tested directly against the real Worker runtime locally using the exact same route-matching this Worker uses in production. Given that coverage, flipping the switch live in production was judged unnecessary real-user risk for the verification it would have added.

## Not built (explicitly out of scope, per the approved design)

Editing `assets/config/site.json`, marketing pages, or any GitHub-publishing pipeline — Settings manages Worker/D1 operational configuration only, as scoped. A generic "create your own feature flag" authoring UI — `site_settings`' key-value shape supports this mechanically for a future phase, but no such UI was requested or built now. Retroactive application of download defaults to existing products — deliberately never touched, per the approved design's "future-only" requirement.
