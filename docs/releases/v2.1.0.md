# Robayer WealthLab — Version 2.1.0 Release Notes

**Content & Administration Platform.** This release closes the gap between the admin panel's original 7 live modules (Dashboard, Products, Media Library, Orders, Consultations, Contacts, Analytics) and a complete 11-module platform, and adds the identity/security hardening a multi-admin, multi-module platform needs. Seven phases; Phase 7 (this document's own trigger) was a release-readiness audit, not a feature phase — see `docs/v2.1-release-checkpoint.md` for the full audit report.

This document is a curated summary for operators and future contributors — see `CHANGELOG.md` for the chronological, phase-by-phase detail behind every line item below.

## Major new features

- **Resources CMS** — admin-authored downloadable resources (templates, checklists, trackers, worksheets, guides), server-rendered at `/resources/*`.
- **Blog CMS** — admin-authored articles with rich-text editing, server-rendered at `/blog/*`.
- **Identity & Security** — account lockout after repeated failed logins, forced password change, self-service change/forgot/reset password, session listing and revocation, login history.
- **User Management** — invite-based admin account provisioning (no self-registration), role assignment (`support`/`editor`/`super_admin`), account disable/re-enable/delete, all guarded against locking out the last remaining Super Admin.
- **Settings** — operational configuration (maintenance mode, email sender identity, per-template email kill switches, newsletter campaign recipient cap) editable from the admin panel without a code deploy, plus a System Information diagnostics panel.
- **Newsletter Campaigns** — draft, test-send, and send-to-all-subscribers campaign management with a Draft → Sending → Sent lifecycle, mandatory successful-test-before-send gate, configurable recipient safety cap, and safe resume-after-interruption.

## Breaking changes

None. Every phase in this release was additive — new tables, new routes, new admin pages. No existing endpoint's request/response shape changed, and no existing admin page's behavior changed except where explicitly documented as a bug fix (see "Known issues fixed" below).

## Database migrations introduced

Six new migrations, `0011` through `0016`, applied in order via `wrangler d1 migrations apply`:

| Migration | Adds |
|---|---|
| `0011_resources_module.sql` | `resources` table |
| `0012_blog_cms.sql` | `blog_posts` table |
| `0013_identity_security.sql` | `password_reset_tokens`, `login_history`, plus lockout/must-change-password columns on `admin_users` |
| `0014_user_management.sql` | `admin_invites` table, `created_by` on `admin_users` |
| `0015_site_settings.sql` | `site_settings` table |
| `0016_newsletter_campaigns.sql` | `newsletter_campaigns`, `newsletter_campaign_recipients` tables |

`backend/database/schema.sql` was verified against all 16 cumulative migrations (0001–0016) during this release's Phase 7 audit, and one drift was found and corrected: `payment_transactions.event_type` was missing its `DEFAULT 'charge.success'` clause (present in migration `0003` since Version 1.2, never reflected in `schema.sql`). This is a documentation correction only — the deployed database already had the correct default; no new migration was needed or created.

## Security improvements

- Account lockout (5 failed attempts, 15-minute window) with identical response timing regardless of whether the account exists, is locked, or the password is simply wrong (no user-enumeration signal).
- Forced password change on next login, settable per-admin.
- Self-service and admin-triggered password reset, both invalidating all other active sessions on success.
- Invite-based account creation only — no public admin self-registration exists anywhere.
- A guard preventing the last active Super Admin from being disabled, deleted, or demoted (by themselves or anyone else).
- Every admin mutation across every new module writes its own `audit_logs` row — confirmed complete across all 13 `routes/admin/*.ts` files and 90 handlers during the Phase 7 security audit, which found zero authorization, CSRF, or IDOR issues.
- Newsletter Campaigns' send path required its own dedicated safety design: mandatory successful test-send before "Send" unlocks, a configurable recipient cap enforced before any state change, `super_admin`-only Send/Resume, and a durable per-recipient roster that makes duplicate-send prevention and safe interrupted-send recovery possible without new infrastructure.

## Operational changes

- **`docs/backup-and-recovery.md`, `docs/deployment-checklist.md`, and `docs/monitoring-and-alerting.md` were rewritten during Phase 7** to describe the real, deployed system instead of a pre-launch plan. They previously referenced Cron-Trigger-driven backup exports, Paystack reconciliation jobs, and "KV-backed" admin sessions — none of which were ever built. The rewritten versions state plainly what's real (D1 Time Travel, Worker version rollback, `wrangler tail`) versus what remains manual (backup beyond Time Travel's window, webhook reconciliation, email retry).
- **`docs/v2.1-technical-debt-register.md` was created** — nine deliberate architectural trade-offs and known gaps, each with its current approach, why it was chosen, the concrete signal that would trigger revisiting it, and the recommended direction.
- `backend/README.md` and its `routes/`/`services/` subfolder READMEs, which had stopped narrating after Version 2.0 Phase 1, were corrected to state which modules are live today rather than leaving them listed as "out of scope until its own phase."

## Known limitations

- **No automated backup beyond D1's Time Travel retention window.** No fallback exists if a restore is needed further back than Time Travel retains.
- **Media Library uploads (R2) have no backup path.** An admin-uploaded file exists only in R2; losing the bucket loses the file, with only its metadata surviving in D1.
- **No automated Paystack webhook reconciliation or email-send retry.** Both are manual today — see `docs/v2.1-technical-debt-register.md` items 4 and 8.
- **No staging environment.** `wrangler.jsonc` has no `env.staging` block; verification happens via local `wrangler dev` plus careful, disposable-data-only checks directly against production after deploy.
- **Newsletter campaign sends are capped** at a configurable limit (`site_settings.campaign_recipient_cap`, default 300) — see `docs/v2.1-technical-debt-register.md` item 1 for the Cloudflare Queues migration path if this is ever exceeded.
- **Local `wrangler dev` cannot fully verify newsletter test-sends or the Media Library upload flow** — both require credentials/OS dialogs unavailable in this project's local/CI verification environment; both were verified against real production infrastructure instead where the Phase 6/7 audits judged it safe to do so.

## Upgrade notes

- Apply migrations `0011` through `0016` in order via `wrangler d1 migrations apply --remote` (not `wrangler d1 execute --file=`, which runs the SQL but skips registering it in the `d1_migrations` tracking table — a real gap hit once during Phase 5 and manually corrected).
- No new secrets or environment variables were introduced in this release. The only two real secrets remain `RESEND_API_KEY` and `PAYSTACK_SECRET_KEY`.
- `site_settings` (migration `0015`) ships with sane defaults on first apply — no manual seeding required.

## Deployment requirements

- Cloudflare Worker (`robayer-wealthlab-api`), D1 database, R2 bucket (`STORAGE` binding), KV namespace (`RATE_LIMIT_KV` binding) — all already provisioned; no new bindings were added in this release.
- Deploy with `wrangler deploy --var DEPLOYED_COMMIT:{git commit hash} --var DEPLOYED_AT:{ISO timestamp}` so the Settings page's System Information panel and any future rollback both have an accurate reference point.

## Rollback considerations

- **Worker code:** `wrangler rollback` to the immediately prior tagged deployment — fast, and independent of any database state.
- **Database:** if a migration in this range causes a problem, restore via D1 Time Travel to the timestamp just before it ran (`docs/backup-and-recovery.md`). A Worker rollback does **not** roll back the database, and vice versa — the two must be reasoned about independently.
- **Newsletter Campaigns specifically:** a campaign that reaches `sending` cannot be cancelled, only resumed (a deliberate Phase 6 design decision — see `docs/v2.1-phase6-design.md`). If a rollback is needed mid-send, the durable per-recipient roster means resuming after the rollback will not re-send to already-completed recipients.
