# Version 2.1 Phase 1 — Implementation Log (Resources CMS)

Tracks what was actually built against the plan in `docs/v2.1-architecture-plan.md` Section 3. Written after implementation, matching the established `docs/v2.0-phase3-implementation.md` format.

---

## Scope delivered

Full D1-backed CMS for the free resource library, replacing `resources/index.html`'s hand-authored, static `.resource-card` markup — whose "Download" buttons were `data-placeholder-action` stubs (no resource on that page has ever been genuinely downloadable). Mirrors the Products Module's proven pattern (D1 is the sole source of truth, server-rendered public page via a dedicated Workers Route, no GitHub-publish pipeline) at a scope trimmed to match this content type's real, simpler shape: no pricing, no `resource_files`/`resource_gallery`/`resource_relations` join tables — a resource has never needed more than one file and one cover image.

**Migration `0011_resources_module.sql`** (production, applied before any code deploy): one new table, `resources` — `resource_id`/`slug` (stable identifiers, matching `products`' own convention), `category`/`format` (CHECK-constrained to the 5/5 real values the original static page's filter pills and format badges already used), `status` (`draft`/`published`/`archived`), `file_media_id`/`cover_media_id`/`thumbnail_media_id` (direct FKs to `media_assets`, not join tables), SEO fields, `featured`, `download_count` (server-incremented only, never client-trusted). Four indexes (status, category, deleted_at, created_at). Purely additive.

**Backend**: `services/resourceService.ts` (CRUD, publish lifecycle, soft delete/restore, duplicate — a trimmed, field-renamed copy of `productService.ts`'s exact shape), `routes/admin/resources.ts` (thin HTTP layer, `editor`/`super_admin`-gated writes matching Products' role convention — Resources is content management, not a support-triage workflow like Consultation/Contact Manager), `routes/resources.ts` (public `/resources/*` Workers Route, server-rendered, mirroring `routes/books.ts`'s exact "own the whole path, never proxy to the static origin" pattern).

**Validation, the one genuine content-type-specific rule**: a resource can only transition to `published` with a real `file_media_id` already attached — mirrors `products`' "active needs a price" rule, adapted to what actually makes a free resource real (a downloadable file, not a price).

**Public delivery**: `GET /resources/{slug}/download` looks up the resource, confirms it's genuinely `published` (never serves a draft/archived file, regardless of the file existing), increments `download_count` server-side, and redirects to the file's already-public Media Library URL (`GET /api/media/file/:key`) — no new download-token mechanism, no duplicated byte-serving logic, since every resource is free by construction (no `price` field exists on this table at all).

**Public page** (`routes/resources.ts`): hero, the Financial Calculators section, and the FAQ (with matching `FAQPage` JSON-LD) are carried over verbatim from the real, hand-authored static page — this phase's job was making the resource grid, featured banner, and "Popular resources" list data-driven, not rewriting content that was already correct. "Popular resources" is now genuinely data-driven (top 3 by real `download_count`) rather than a hard-coded list tied to placeholder content.

**Admin frontend**: `/admin/resources/` (list: search/filter by category+format+status+featured/sort/pagination/bulk actions, mirroring `admin/products/`'s exact structure), `/admin/resources/new/` + `/admin/resources/edit/` (editor: title/slug/short description, rich-text description via the same contenteditable+`execCommand`+client-and-server-sanitized pattern Products uses, a single file picker instead of a files list, cover/thumbnail Media Library pickers, SEO fields, tags/featured, live preview, danger zone). Zero new CSS primitives — every component (`.editor-layout`, `.media-ref-picker`, `.richtext-editor`, `.editor-list-row`, `.modal-overlay`) already existed from Products' own build-out.

## Real defect found during local verification, fixed before deploy

**`Response.redirect()` threw on every real download attempt.** The Fetch API's `Response.redirect(url, status)` static method requires a fully-qualified absolute URL — there is no `document.baseURI` in the Workers runtime to resolve a relative one against — but this codebase's established `filePublicUrl` convention (`media_assets.public_url`, used identically by Products' file-serving path) is root-relative (`/api/media/file/...`). Every download attempt returned a 500. Found via a real, disposable local test download (not assumed working from a code read) — the `download_count` increment had already committed by the time the redirect construction threw, confirming the bug was specifically in the redirect step, not the lookup/counter logic. Fixed by constructing the `Response` by hand with a `Location` header instead of the constructor helper — an HTTP `Location` header is not constructor-validated the way `Response.redirect()`'s argument is, and every real browser/fetch client correctly resolves a relative `Location` against the response's own origin per RFC 7231. Re-verified via the identical local download request (302 with the correct `Location`), then re-confirmed end-to-end in production before considering the stage complete.

## Known limitations

1. **Local dev proxy doesn't forward `/books/*` or `/resources/*`** — a pre-existing gap (`scripts/dev-proxy.mjs` only proxies `/api/*` to `wrangler dev`), affecting Products' public route identically. Verifying either route locally requires hitting the Worker directly on its own port (`localhost:8787`) rather than through the static-site proxy (`localhost:5500`). Not fixed this phase — a real but small local-tooling improvement, out of scope for a content-module phase.
2. **No revision history for resources** — unlike Blog (Phase 2's planned `blog_post_revisions`), a resource edit has no undo beyond re-editing by hand. Consistent with Products, which also has no revision history; not a regression specific to this phase.
